## Context

`project-core` 后端规格已归档，并且 `digital-platform-api` 已提供项目列表、项目创建和项目详情基础接口。`technical-architecture` 已确认正式前端采用 Vue + Vite，并通过后端 API 获取正式业务数据。

当前前端 Demo 仍只能作为页面参考。`add-project-core-frontend` 的目标是实现正式项目核心前端页面，接入 `digital-platform-api`，让用户可以查看项目列表、创建项目，并进入项目详情查看基础状态。

## Goals / Non-Goals

**Goals:**

- 配置前端 API base URL，并通过统一 API 客户端访问 `digital-platform-api`。
- 实现项目列表页，调用 `GET /api/projects`。
- 实现新建项目页，调用 `POST /api/projects`。
- 实现项目详情页，调用 `GET /api/projects/:projectId`。
- 在页面中展示项目基础信息、当前阶段和 8 阶段基础状态。
- 保留前端 Demo 作为布局和交互参考，但正式数据必须来自后端 API。

**Non-Goals:**

- 不实现在线表单。
- 不实现文件上传、文件下载或文件管理平台联动。
- 不实现权限、登录、角色控制或菜单权限。
- 不实现业务日志、审计日志或下载日志。
- 不实现管理层看板指标。
- 不实现阶段推进。
- 不实现资料齐套率、资料清单或资料缺失统计。

## Decisions

### 决策一：使用统一 API 客户端封装 base URL

前端通过配置项读取 API base URL，并由统一 API 客户端拼接 `/api/projects`、`/api/projects/:projectId` 等接口。页面不直接散落硬编码服务地址。

不选择在每个页面手写完整 URL，因为后续内网部署时 API 地址可能变化，集中配置更容易维护。

### 决策二：项目列表页只展示项目基础状态

项目列表页读取 `GET /api/projects`，展示项目编号、项目名称、客户、项目经理、项目状态、当前阶段、计划开始时间和计划完成时间，并提供进入详情和新建项目入口。

不在项目列表中展示看板指标、资料齐套率、文件归档状态或资料缺失统计，因为这些能力不属于项目核心前端范围。

### 决策三：新建项目页直接调用项目创建 API

新建项目页收集项目编号、项目名称、客户、项目经理、参与部门、计划时间和备注，提交到 `POST /api/projects`。创建成功后可进入项目详情或返回列表。

不在新建项目时创建文件夹、上传文件、配置资料清单或分配权限，因为这些后续由独立能力处理。

### 决策四：项目详情页展示基础信息和阶段状态

项目详情页读取 `GET /api/projects/:projectId`，展示项目基础信息、当前阶段，以及全部 8 个阶段的顺序、名称、状态和当前阶段标记。

不在详情页展示在线表单、文件区、业务日志、阶段推进按钮、资料清单或资料齐套率，避免前端提前暴露未实现能力。

### 决策五：页面状态覆盖加载、空数据和错误

项目列表、创建和详情页面必须处理加载中、空数据、接口错误和创建校验错误。后端返回的 `VALIDATION_ERROR`、`PROJECT_CODE_EXISTS`、`PROJECT_NOT_FOUND` 应在页面上转成可读提示。

不选择静默失败或只在控制台输出错误，因为第一版需要能支持真实试用。

### 决策六：新建正式前端工程

正式前端工程使用独立目录 `digital-platform-web`，不继续在 `digital-demo` 上开发。`digital-demo` 只作为布局、视觉风格和交互方式参考，不作为正式数据架构或正式数据来源。

### 决策七：使用 Vite 环境变量和开发代理

前端通过 `VITE_API_BASE_URL` 配置后端地址。开发期如有跨域问题，优先使用 Vite proxy 将 `/api` 转发到 `digital-platform-api`。

### 决策八：创建成功后进入项目详情

新建项目成功后默认跳转到该项目详情页，便于立即查看项目基础信息、当前阶段和 8 阶段初始化状态。

## Risks / Trade-offs

- [API base URL 配置错误导致页面不可用] -> 页面应显示接口连接失败提示，并在 README 或环境变量说明中记录配置方式。
- [后端字段变化导致前端展示错位] -> API 客户端集中做响应适配，页面只使用适配后的项目和阶段字段。
- [Demo 静态数据被误用] -> 任务和验收中明确禁止使用 Demo 静态数据作为正式数据来源。
- [范围被扩展到资料或文件能力] -> 页面只保留项目核心入口，不渲染在线表单、文件、日志、权限、看板或阶段推进控件。

## Migration Plan

- 在正式前端工程中加入 API base URL 配置。
- 新增或整理项目列表、新建项目、项目详情三个页面。
- 将页面数据来源从 Demo 静态数据切换为 `digital-platform-api`。
- 验证三条主路径：列表加载、创建项目、详情查看。

## Resolved Questions

- 正式前端另建独立目录 `digital-platform-web`，不继续在 `digital-demo` 上开发。
- API base URL 使用 `VITE_API_BASE_URL`；开发期可用 Vite proxy 转发 `/api`。
- 创建项目成功后默认跳转项目详情页。
