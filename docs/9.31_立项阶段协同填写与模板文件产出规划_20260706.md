# 9.31 立项阶段协同填写与模板文件产出规划

## 背景

当前 `1.1 项目需求表`、`1.2 项目立项审批表`、`1.3 项目立项通知` 已通过在线表单承载填写、提交、评价审批和阶段推进门禁，但提交后的产出仍主要以在线表单内容展示为主，尚未生成与真实模板对应的 Excel / Word / PDF 文件。

与领导确认后，立项阶段产出口径需要进一步明确：负责人仍使用在线表单填写结构化内容；提交后，后续应按模板生成已填充内容的文件；其他有权限人员默认查看对应产出文件。文件生成、模板填充、文件版本、下载预览和文件平台联动拆为后续独立实现，本 change 先固化并实现客户联系人、`1.2` 协同填写、项目编号前置等流程、字段和职责边界。

## 当前差异

| 事项 | 当前系统 | 目标口径 |
| --- | --- | --- |
| 新建项目 | 已有项目名称、客户、客户联系方式、商务负责人、技术负责人 | 新增客户联系人，字段位于客户联系方式前 |
| 1.2 基础模块 | 基础信息仍不完整 | 补齐模板表头区域和对应已有字段，且客户基本信息、项目基本信息都属于基础模块 |
| 1.2 填写责任 | 当前由商务负责人填写整张表 | 改为商务负责人填写基础模块和商务模块，技术负责人填写技术模块 |
| 1.2 项目编号 | 当前在 1.2 总经理最终通过后填写，用于 1.3 | 前置到 1.2 表单填写阶段，1.2 提交前必须已有项目编号 |
| 评价审批启动 | 当前 1.2 表单提交后进入评价审批 | 1.2 商务部分和技术部分都完成后，才进入商务评价、技术评价、总经理审批 |
| 产出查看 | 当前查看已提交产出仍是在线表单 | 后续应默认查看生成后的模板文件，在线表单作为填写入口 |

## 新目标流程

```text
创建项目，填写客户联系人，选择商务负责人和技术负责人
  -> 商务负责人填写 1.1 项目需求表
  -> 商务负责人填写 1.2 基础模块 + 商务模块，并填写项目编号
  -> 技术负责人填写 1.2 技术模块
  -> 1.2 商务部分和技术部分均完成后，启动商务评价、技术评价、总经理审批
  -> 1.2 总经理最终通过
  -> 营销中心负责人确认/提交 1.3 项目立项通知
  -> 立项阶段齐套后推进
```

## 关键口径

### 客户联系人

- 新建项目增加客户联系人字段。
- 字段展示顺序为：客户名称、客户联系人、客户联系方式。
- 新建项目客户联系人必须填写；客户联系人为空时不得创建新项目。
- 旧项目或历史数据缺失客户联系人时允许为空展示，不迁移、不补数据。
- 项目详情、项目列表、项目总览、立项项目输入节点和在线表单自动带出都支持客户联系人。

### 1.2 基础模块

- `客户基本信息` 和 `项目基本信息` 都属于 `1.2 项目立项审批表` 的基础模块，不拆成新的资料项或新的审批节点。
- 基础模块按真实模板表头区域和对应已有字段实现，不扩展模板外客户调查或项目调查字段。
- 基础模块按模板表头层级展示：顶层 `表头信息` 只包含项目名称、项目号/项目编号；`客户基本信息` 只包含客户名称、项目联系人、联系方式；`项目基本信息` 只包含项目负责人、项目负责人联系方式。
- 项目联系人映射创建项目时填写的客户联系人；项目负责人优先沿用商务负责人。
- 技术负责人是流程协同角色，不作为 `1.2` 表头基础模块字段展示。
- 基础模块由商务负责人填写。

### 项目编号前置

- 项目编号在 `1.2 项目立项审批表` 填写阶段填写。
- 项目编号保存到项目主表 `projects.project_code` 或等价稳定项目编号字段。
- `1.2` 提交前必须已有项目编号，项目编号唯一性继续生效。
- `1.2` 退回后项目编号保留，项目结束后项目编号保留用于审计。
- `1.3 项目立项通知` 自动带出 `1.2` 阶段确定的项目编号。

### 1.2 协同填写

- 商务负责人填写基础模块和商务模块。
- 技术负责人填写技术模块。
- 商务负责人不能代替技术负责人提交技术模块，除非后续独立授权规则明确允许。
- 技术负责人不能修改基础模块和商务模块。
- 商务部分提交后，基础模块和商务模块锁定；技术部分提交后，技术模块锁定，需评价退回后才可重新填写。
- `1.2` 不再展示或支持普通资料责任人手工分配，待办只来自商务/技术协同填写合成待办。
- 工作台需要分别生成商务填写待办和技术填写待办。
- 双方都完成后，才允许启动现有商务评价、技术评价、总经理审批。

### 评价审批保持

- 商务评价、技术评价、总经理审批继续复用现有 `1.2` 专用评价审批节点。
- 不新增第二套审批流，不引入 BPM 或通用流程引擎。
- 商务/技术任一退回仍退回 `1.1 项目市场调研`。
- 总经理拒绝仍支持退回项目市场调研或项目结束。

## 文件产出口径

第一阶段仍使用在线表单作为填写入口。后续提交后应生成：

- `智能制造项目管理文件模板/项目需求表-模板.xlsx`
- `智能制造项目管理文件模板/项目立项审批表-模板.xlsx`
- `智能制造项目管理文件模板/关于确定项目名称及编号的通知-模板.docx`

有权限查看者后续默认查看生成后的文件。文件生成、模板填充、文件版本、下载/预览、存储位置、文件生成失败重试、覆盖还是版本化、文件平台是否接入，全部作为后续独立 change 的实现设计点。

本 change 不直接接文件平台，不生成 Excel / Word / PDF，不改变 71 项资料数量，不新增资料项，不删除兼容资料区，不迁移旧项目。

本 change 未新增独立数据库 migration 文件。当前实现通过 schema 初始化补充 `projects.customer_contact_person` 可空字段，`1.2` 协同状态复用在线表单 `form_data_json._collaboration`；若后续生成文件元数据或更复杂协同查询需要结构化字段，必须先明确数据库设计，并通过后续独立数据库设计处理。

## 与当前 OpenSpec 口径的关系

- 本 change 不推翻 v20260629 / 71 项模板。
- 本 change 不新增 `1.1 / 1.2 / 1.3` 以外的立项资料项。
- 本 change 修正上一轮 `1.2 由商务负责人填写整张表` 的口径，改为商务负责人和技术负责人协同填写。
- 本 change 将项目编号从 `1.2 总经理最终通过后` 前置到 `1.2 表单填写阶段`。
- 本 change 将“提交后生成模板文件”作为后续能力方向，但不实现文件生成。
