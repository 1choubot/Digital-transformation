## Context

项目入口和工作区导航已经完成主架构：项目总览是项目入口，`/projects/:id` 是项目工作区，立项阶段已有蓝色节点、节点产出、在线表单和 `1.2` 评价/审批入口。当前需要解决的是前端体验一致性：页面语言、操作层级、旧资料清单兼容区、工作台深链语义和组件边界还未统一。

推荐页面结构：

```text
App Shell
  |-- 主导航
  |-- 项目总览
  |-- 项目工作区
  |   |-- 左侧 8 阶段导航
  |   `-- 右侧阶段工作区
  |       |-- 阶段说明
  |       |-- 蓝色节点
  |       |-- 节点产出
  |       `-- 在线表单/评价审批
  |-- 我的工作台
  `-- 用户管理
```

## Goals / Non-Goals

**Goals:**

- 统一项目总览、项目工作区、我的工作台、新建项目和用户管理的页面语言与基础组件体验。
- 固化项目工作区层级，让阶段、节点、产出和在线表单/评价审批动作按稳定顺序展示。
- 将旧资料清单定义为辅助兼容区，而不是立项阶段主入口。
- 保持工作台深链进入项目工作区后的定位体验一致。
- 以现有接口字段组合 UI，不新增后端接口。
- 保持其他 7 阶段旧资料清单入口可访问。

**Non-Goals:**

- 不新增后端接口、数据库表或 migration。
- 不新增业务权限或改变后端业务状态机。
- 不做第二阶段项目经理、项目模式、立项日期补录。
- 不补齐其他 7 阶段完整蓝色节点映射。
- 不把其他阶段在线表单化。
- 不恢复文件平台联动。
- 不处理 `file-platform-integration-v1` 或 `define-digital-platform-v1`。
- 不改变立项阶段 `1.1 / 1.2 / 1.3` 已确认业务规则。

## Decisions

### 统一入口与页面头部

项目总览继续作为统一项目入口，`/projects` 和主导航进入项目总览。新建项目、项目总览、项目工作区、我的工作台和用户管理应使用一致的页面头部、返回入口、主操作按钮、加载态、空态和错误态语义。

### 项目工作区固定层级

项目工作区继续使用左侧 8 阶段导航。右侧阶段工作区按阶段说明、蓝色节点、节点产出、在线表单/评价审批动作区组织。点击蓝色节点只选择节点并展示产出，不直接打开在线表单。

### 旧资料清单辅助化

旧资料清单保留为辅助兼容区，建议折叠、弱化或放在工作区主操作之后。立项阶段 `1.1 / 1.2 / 1.3` 的主操作继续位于节点产出卡片；其他 7 阶段在蓝色节点未完整配置前仍可通过旧清单处理临时操作。

### 权限和状态来源不变

第一版 UI 统一只使用现有接口返回的 `permissions`、`blockingReasons`、`status`、在线表单权限和工作区节点/产出字段。前端不得因为视觉统一新增本地业务权限判断，也不得临时扩展后端接口。

### 组件边界收敛

后续实现应收敛 App shell、页面头部、状态提示、操作按钮、阶段导航、蓝色节点列表、节点产出区、在线表单动作区和旧资料清单辅助区的组件边界。样式应尽量沉淀到语义化组件或局部 class，避免继续向 `styles.css` 无序追加页面专用规则。

## Risks / Trade-offs

- [Risk] 只做视觉统一可能误伤既有操作入口。Mitigation: 浏览器验收覆盖项目入口、项目工作区、立项阶段主操作、旧清单和工作台深链。
- [Risk] 旧清单弱化可能影响其他 7 阶段临时操作。Mitigation: 保持其他 7 阶段旧清单入口可访问，并明确旧清单辅助区不是删除旧能力。
- [Risk] 工作台深链容易回归。Mitigation: 验证 `taskMode`、`documentId`、`stageId`、`nodeKey` 进入项目工作区后的定位，不自动打开在线表单。
- [Risk] 样式统一如果只堆叠 CSS 会继续扩大维护成本。Mitigation: 实施任务必须同步收敛组件边界和样式职责。
- [Risk] 验收只覆盖项目工作区会遗漏跨页面语义冲突。Mitigation: Implementation tasks must include explicit browser acceptance for project overview, create page, workspace, workbench, user management, legacy checklist, and mobile/desktop layouts.
