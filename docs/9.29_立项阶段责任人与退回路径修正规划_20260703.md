# 9.29 立项阶段责任人与退回路径修正规划

## 背景

领导补充了立项阶段细化流程。现有系统已经启用 v20260629 / 71 项模板，并保留 1.1 / 1.2 / 1.3 在线表单与 1.2 专用评价审批。本规划不推翻 71 项模板，不新增资料项，不接文件平台，而是修正立项阶段内部责任人、项目输入展示和退回路径口径。

## 目标

- 项目仍然先创建，创建后才进入项目详情和立项阶段。
- 创建项目时增加商务负责人、技术负责人选择。
- 商务负责人必须来自营销中心启用用户。
- 技术负责人必须来自研发中心启用用户。
- 项目输入节点展示项目名称、客户名称、客户联系人和电话、商务负责人、技术负责人。
- 1.1 项目需求表和 1.2 项目立项审批表默认由商务负责人填写。
- 1.3 立项通知继续由营销中心负责人填写。
- 商务评价拒绝、技术评价拒绝必须退回项目市场调研。
- 总经理拒绝必须支持退回项目市场调研或项目结束。
- 项目结束后不得继续推进立项通知、方案设计或后续资料操作。
- 总经理拒绝选择“项目结束”时，必须记录结束原因、操作者、操作时间和业务日志。
- 项目结束后，相关待办不得继续作为可处理待办展示。
- 项目总览和项目详情必须展示项目已结束状态，至少展示原因摘要。

## 非目标

- 不切换或重定义 v20260629 / 71 项模板。
- 不新增资料项，不改变 71 项模板数量。
- 不做真实自动生成 Word、PDF 或文件平台文件。
- 不接文件平台，不生成附件，不导出文件。
- 不引入 BPM 或通用流程引擎。
- 不新增第二套资料状态机。
- 本规划阶段不改业务代码，不改数据库 schema，不写 migration。

## 正确业务理解

项目必须先创建，系统才有项目详情、立项阶段和在线表单上下文。当前项目创建方式基本保留，只在创建项目时增加两个项目级责任人：

- 商务负责人：营销中心人员。
- 技术负责人：研发中心人员。

流程图中的“自动生成项目需求表 / 项目立项审批表 / 立项通知”第一版不理解为生成真实文件。第一版口径是：在线表单填写完成后，有权限浏览项目的人可以查看在线表单内容。系统不导出 Word/PDF，不创建文件平台文件，也不生成普通附件。

## 立项阶段流程

```text
创建项目
  -> 填项目名称、客户名称、客户联系人和电话
  -> 选择商务负责人、技术负责人
  -> 进入项目详情 / 立项阶段

项目输入
  -> 展示项目基础信息和商务/技术负责人
  -> 完成后进入项目市场调研

项目市场调研
  -> 商务负责人填写 1.1 项目需求表在线表单
  -> 完成后进入项目立项审批

项目立项审批
  -> 商务负责人填写 1.2 项目立项审批表在线表单
  -> 完成后启动商务评价、技术评价

商务评价审批
  -> 营销中心负责人处理
  -> 通过：节点完成
  -> 拒绝：退回项目市场调研

技术评价审批
  -> 研发中心负责人处理
  -> 通过：节点完成
  -> 拒绝：退回项目市场调研

总经理审批
  -> 商务评价和技术评价都完成后才可处理
  -> 同意：进入立项通知
  -> 拒绝：可退回项目市场调研，或项目结束
  -> 选择项目结束：记录结束原因、操作者、操作时间和业务日志，抑制后续可处理待办

立项通知
  -> 营销中心负责人填写 1.3 立项通知在线表单
  -> 完成后进入方案设计阶段
```

## 项目结束口径

- 总经理拒绝并选择“项目结束”时，必须要求填写或选择结束原因。
- 系统必须记录项目结束原因、操作者、操作时间和业务日志。
- 项目结束后，立项通知、方案设计、阶段推进、后续资料操作必须不可继续处理。
- 项目结束后，相关待办不得继续作为可处理待办展示；历史记录可保留用于审计。
- 项目总览和项目详情必须展示项目已结束状态，并至少展示原因摘要。

## 当前系统差异

| 当前口径 | 目标口径 |
| --- | --- |
| 当前新建项目只有项目名称、客户、联系方式等基础信息。 | 新建项目增加商务负责人、技术负责人。 |
| 当前项目输入节点责任人信息不足。 | 项目输入节点展示项目名称、客户名称、客户联系人和电话、商务负责人、技术负责人。 |
| 当前 1.1 责任口径偏泛化。 | 1.1 项目需求表由商务负责人填写。 |
| 当前 1.2 责任口径偏泛化。 | 1.2 项目立项审批表由商务负责人填写。 |
| 当前商务评价退回能力不足。 | 商务评价拒绝退回项目市场调研。 |
| 当前技术评价退回能力不足。 | 技术评价拒绝退回项目市场调研。 |
| 当前总经理拒绝主要走返工。 | 总经理拒绝支持退回项目市场调研或项目结束。 |
| 当前项目结束后的审计、待办抑制和页面展示口径不足。 | 项目结束必须记录原因、操作者、时间和日志；相关待办不可继续处理；项目总览和详情展示已结束状态与原因摘要。 |
| 当前不生成真实文件。 | 目标仍不生成真实文件，只查看在线表单内容。 |
| 当前 1.1 / 1.2 / 1.3 已是在线表单主流程。 | 继续保持 1.1 / 1.2 / 1.3 在线表单和 1.2 专用评价审批主流程。 |

## 后续实现提示

- 后续实现可改 API/Web 和必要数据库字段或等价绑定，用于保存商务负责人、技术负责人。
- 旧项目兼容策略需要在实现阶段明确，不能在规划阶段默认迁移旧项目。
- 退回项目市场调研应复用现有立项在线表单、精准返工、状态和日志能力。
- 项目结束应是明确终态，阻止继续推进立项通知、方案设计和后续资料操作，并抑制相关可处理待办。
- 项目结束状态应在项目总览和项目详情可见，避免用户继续按正常在办项目处理。
