## 1. 治理结论

本轮不以消灭 Vue scoped 样式为目标，而是统一样式职责：跨页面设计变量、共享布局原语和 Element Plus 全局主题归 `styles.css`；组件专属复杂布局归组件 scoped 样式。禁止同一职责同时存在于全局和局部样式，也禁止把页面强耦合规则继续无序堆入全局文件。

这与 active OpenSpec 的“收敛组件边界、避免全局 `styles.css` 无序堆叠和页面职责强耦合”要求一致。

## 4. 全局样式职责

`styles.css` 继续负责：

- 设计变量、品牌色、语义色、间距、圆角和阴影；
- Element Plus 主题变量及经过批准的全局覆盖；
- App shell、顶部导航、页面布局原语和响应式断点；
- 项目工作区宽度、滚动边界和多节点共享动作区；
- 多页面稳定复用的表单、错误态、只读态和状态展示。

不得在全局文件中重新建立按钮、输入框、选择器、Toast、Spinner、Alert 或 Badge 控件体系，也不得以 `Migrated from ...vue` 为由永久保留仅服务单个组件的规则。

## 5. 自动治理门禁

新增 `scripts/check-style-governance.mjs`，并纳入 `npm.cmd run check`。门禁包括：

- 禁止新增原生可见表单控件和旧通用控件类；
- 原生按钮仅允许应用壳、工作区阶段/节点导航和评分折叠控制，且锁定文件及数量；
- Vue 样式必须 scoped，并限制在12个允许文件；
- Vue 局部样式禁止 `!important` 和硬编码十六进制颜色；
- `:deep()` 仅允许方案评审矩阵和财务审批流使用，并必须受授权根类约束；
- `styles.css` 不得超过7141检查行，`!important` 不得超过16处。

预算不是永久合理性证明。确需增加时，必须在同一变更中说明复用范围、替代方案和视觉回归结果，不能仅调高数字绕过检查。

## 6. 原生按钮例外

当前7处原生按钮承担结构化交互，而不是普通业务操作：

- `MainLayout.vue`：退出、移动菜单和模块导航4处；
- `NodeOnlineFormEditor.vue`：评分区折叠控制1处；
- `ProjectWorkspaceStageNav.vue`：阶段导航1处；
- `ProjectWorkspaceNodeList.vue`：节点导航1处。

保存、提交、审批、退回、删除等业务动作继续使用 `el-button`。可见上传入口继续使用受控 `el-upload` 或 `el-button`。

## 7. 工作区视觉与滚动契约

根据 active OpenSpec：

- 桌面端项目工作区限制在主布局可用高度内，左侧流程树和右侧节点业务卡片分别纵向滚动；
- 节点普通内容不得撑宽父业务卡片；
- 固定最小宽度表格必须在边界明确的局部容器内横向滚动；
- 工作区容器不超过760px时，由 `.app-main` 恢复页面级纵向滚动，节点卡片使用自然高度和 `overflow: visible`；
- 窄屏下宽表格仍可保留局部横向滚动。

这些是功能契约，不得为简化 CSS 擅自改变。
