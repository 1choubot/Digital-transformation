# 数字化管理平台与文件管理平台边界及联动方案

版本：v0.1  
日期：2026-05-16  
适用范围：智能制造项目全过程数字化管理平台、企业文件管理系统

## 1. 文档目的

本文件用于明确“数字化管理平台”和现有“文件管理平台”的职责边界、联动方式、第一版建设范围与后续扩展方向。

当前项目不再按“单纯前端 Demo”推进，而应重新定义为：

> 以数字化管理平台作为项目全过程主入口，以文件管理平台作为底层文件归档、下载和存储系统，逐步实现项目从创建、阶段推进、资料提交到最终归档的数字化闭环。

## 3. 总体定位

### 3.1 数字化管理平台

数字化管理平台定位为“项目全过程业务入口”，负责：

- 项目创建。
- 项目基础信息维护。
- 8 阶段流程管理。
- 阶段资料清单管理。
- 在线表单填写。
- 非模板文件上传入口。
- 阶段资料状态管理。
- 阶段推进。
- 项目风险、暂停、延期等状态展示。
- 管理层看板。
- 业务操作日志。

### 3.2 文件管理平台

文件管理平台定位为“公司文件底座”，负责：

- 项目文件夹树生成。
- 文件真实存储。
- 文件下载。
- 文件查看权限控制。
- 文件搜索。
- 文件操作日志。
- 历史文件和非流程文件管理。
- 原有文件管理功能继续保留。

### 3.3 两个平台关系

两者不是互相替代关系，而是上下层关系：

| 系统 | 定位 | 用户感知 |
|---|---|---|
| 数字化管理平台 | 项目业务主入口 | 用户从项目、阶段、任务角度提交资料 |
| 文件管理平台 | 文件归档和下载系统 | 用户从文件夹、文件角度查看和下载资料 |

第一版采用“弱绑定”方案：

- 数字化管理平台调用文件管理平台接口完成项目目录创建、文件归档、文件状态查询和下载。
- 文件管理平台不理解完整项目流程，只负责文件落库、权限判断、下载和日志。
- 第一版不做复杂权限同步，不做钉钉、金蝶深度集成。

## 4. 已确认的关键原则

根据当前沟通，以下原则作为第一版默认口径：

1. **数字化管理平台是项目创建入口**  
   新项目优先在数字化管理平台创建。

2. **所有相关人员优先从数字化管理平台提交项目资料**  
   标准资料通过在线表单填写，非标准资料也可以从数字化管理平台上传。

3. **文件管理平台作为底层文件归档、下载和查看系统**  
   文件管理平台现有功能不调整，继续保留。

4. **第一版不做复杂权限同步**  
   数字化管理平台控制业务操作权限；文件进入文件管理平台后，下载和查看仍遵守文件管理平台权限。

5. **数字化平台归档文件必须记录实际操作人**  
   文件管理平台日志不能只记录“平台上传”，必须能追溯实际提交人、项目、阶段和资料项。

6. **第一版先做项目流程和资料归档，不接钉钉、金蝶**  
   钉钉、金蝶后续作为外部系统挂接或二期集成考虑。

## 5. 职责边界

| 能力 | 数字化管理平台 | 文件管理平台 | 说明 |
|---|---|---|---|
| 项目创建 | 主责 | 被调用生成目录 | 项目主数据以数字化平台为准 |
| 项目编号/名称/客户 | 主责 | 保存目录或文件元数据 | 文件平台不作为项目主数据源 |
| 8 阶段流程 | 主责 | 不负责 | 文件平台只按目录承载阶段资料 |
| 阶段资料清单 | 主责 | 提供对应文件夹 | 数字化平台决定哪些资料应提交 |
| 在线表单 | 主责 | 保存生成文件 | 表单数据归数字化平台，生成文件归文件平台 |
| 非模板文件上传 | 提供入口 | 真实存储 | 上传时传入项目、阶段、资料项 |
| 文件下载/查看 | 可跳转或调用 | 主责 | “查看”实质是下载，需遵守文件平台权限 |
| 文件权限 | 不重复实现复杂文件夹权限 | 主责 | 第一版不做复杂同步 |
| 业务权限 | 主责 | 不负责 | 谁能填表、上传、确认、推进阶段 |
| 文件日志 | 传递操作人信息 | 主责 | 记录真实提交人和来源系统 |
| 业务日志 | 主责 | 不负责 | 记录表单提交、阶段推进等动作 |
| 管理看板 | 主责 | 提供文件状态 | 看板以项目、阶段、资料齐套率为核心 |

## 6. 人员和权限方案

### 6.1 第一版建议

第一版数字化管理平台可以建立自己的人员和业务角色体系，但必须预留与文件管理平台用户的映射字段。

数字化平台人员至少包含：

- 姓名。
- 所属部门。
- 平台角色。
- 是否启用。
- 可选：文件管理平台用户 ID。

推荐字段示例：

```text
id
username
display_name
department_name
platform_role
file_platform_user_id
is_active
```

### 6.2 权限分层

权限必须分成两层：

| 权限类型 | 归属 | 说明 |
|---|---|---|
| 业务权限 | 数字化管理平台 | 控制谁能建项目、填表、上传、确认、推进阶段 |
| 文件权限 | 文件管理平台 | 控制谁能在文件平台查看、下载文件 |

第一版不要把两类权限混成一套。

### 6.3 推荐的平台角色

第一版可先采用简单角色：

| 角色 | 主要权限 |
|---|---|
| 管理层 | 查看全部项目、查看看板、下载有权限文件 |
| 项目经理 | 创建/维护项目、分配资料责任人、推进阶段、查看项目资料 |
| 部门负责人 | 查看本部门相关项目、确认本部门资料 |
| 普通成员 | 查看个人待办、填写表单、上传资料 |
| 系统管理员 | 维护数字化平台基础配置 |

后续可根据实际组织架构细化为营销、商务、研发、制造、验收、文控等业务角色。

## 7. 文件归档流程

### 7.1 标准表单类资料

适用资料：

- 项目需求表。
- 项目立项审批表。
- 方案评审记录表。
- 成本估算表。
- 设计评审记录表。
- 培训记录表。
- 验收记录表。
- 结题报告。

推荐流程：

1. 用户进入数字化管理平台。
2. 打开项目对应阶段。
3. 选择需要填写的资料表单。
4. 在线填写并保存草稿。
5. 提交表单。
6. 数字化管理平台生成文件。
7. 数字化管理平台调用文件管理平台接口，将文件归档到对应文件夹。
8. 文件管理平台记录文件日志。
9. 数字化管理平台记录业务日志。
10. 阶段资料状态更新为“已提交”或“待确认”。

### 7.2 非模板文件类资料

适用资料：

- 图纸。
- 代码。
- PPT。
- 技术协议附件。
- 合同扫描件。
- 现场照片。
- 调试文件。
- 客户确认文件。

推荐流程：

1. 用户进入数字化管理平台。
2. 选择项目、阶段、资料项。
3. 上传文件。
4. 数字化管理平台调用文件管理平台上传接口。
5. 文件管理平台将文件存入对应文件夹。
6. 文件管理平台记录真实提交人。
7. 数字化管理平台更新资料项状态。

## 8. 文件查看和下载规则

领导已明确“查看文件实际就是下载文件”，因此文件查看应按下载逻辑处理。

建议规则：

- 数字化平台可以展示资料状态、文件名、提交人、提交时间。
- 点击查看文件时，本质是发起下载。
- 下载动作应调用文件管理平台能力。
- 下载权限以文件管理平台判断为准。
- 下载日志由文件管理平台记录。
- 数字化平台可同步记录一条业务侧“查看/下载资料”日志。

这样可以兼顾：

- 管理层对资料状态的可见性。
- 文件管理平台原有保密和下载控制。
- 文件流转可追溯。

## 9. 日志方案

日志建议分为两类。

### 9.1 数字化管理平台业务日志

记录业务动作：

- 创建项目。
- 修改项目基础信息。
- 指派资料责任人。
- 保存表单草稿。
- 提交表单。
- 上传阶段资料。
- 确认资料。
- 退回资料。
- 推进阶段。
- 暂停项目。
- 恢复项目。
- 下载资料。

### 9.2 文件管理平台文件日志

记录文件动作：

- 创建项目目录。
- 上传文件。
- 下载文件。
- 重命名文件。
- 删除文件。
- 数字化平台归档文件。

数字化平台调用文件平台归档时，应传递以下元数据：

```text
source_system = digital_project_platform
source_action = submit_form / upload_file / archive_generated_file
source_user_id
source_user_name
source_department
project_code
project_name
stage_key
stage_name
document_key
document_name
target_folder_id
```

## 10. 文件管理平台建议新增接口

文件管理平台现有功能不做大改，只建议新增或封装一组供数字化平台调用的接口。

### 10.1 项目目录接口

| 能力 | 说明 |
|---|---|
| 创建项目文件夹树 | 数字化平台创建项目后，调用文件平台生成目录 |
| 查询项目文件夹树 | 数字化平台展示文件归档位置 |
| 查询项目阶段目录映射 | 根据项目编号和阶段找到目标目录 |
| 查询资料项目录映射 | 根据项目编号、阶段、资料项找到目标文件夹 |

### 10.2 文件归档接口

| 能力 | 说明 |
|---|---|
| 上传归档文件 | 数字化平台上传或生成文件后归档 |
| 查询资料项文件列表 | 判断某资料项是否已有文件 |
| 查询阶段资料完成情况 | 辅助数字化平台展示资料齐套率 |
| 获取下载链接 | 从数字化平台发起下载 |

### 10.3 日志与来源接口

| 能力 | 说明 |
|---|---|
| 写入来源元数据 | 标记文件来自数字化平台 |
| 记录真实提交人 | 避免日志只显示“平台上传” |
| 查询文件操作日志 | 数字化平台需要展示文件追溯时使用 |

## 11. 第一版范围建议

第一版建议只做主线闭环，不做复杂扩展。

### 11.1 必做

- 数字化平台项目创建。
- 项目基础信息维护。
- 标准 8 阶段流程。
- 阶段资料清单。
- 资料责任人。
- 在线表单原型。
- 非模板文件上传入口。
- 文件归档到文件管理平台。
- 资料状态展示。
- 阶段推进。
- 项目总览看板。
- 业务操作日志。
- 文件归档日志带真实提交人。

### 11.2 暂不做

- 钉钉深度集成。
- 金蝶深度集成。
- 复杂审批流引擎。
- 复杂跨系统权限同步。
- 文件在线预览。
- 文件版本复杂管理。
- 多套部门模板。
- 移动端。
- 自动生成完整 Office 文件的复杂格式还原。

## 13. 当前结论

当前推荐方案为：

> 数字化管理平台作为主业务系统，负责项目创建、阶段流程、资料提交和管理看板；文件管理平台保持现有能力不变，作为底层文件仓库、下载入口和文件日志系统。第一版采用弱绑定方式，只打通项目目录创建、文件归档、文件状态查询和下载日志，不做复杂权限同步。

前期纯前端 Demo 不作为最终系统架构，但仍具有参考价值：

- 已验证项目总览、项目详情、阶段工作台等页面形态。
- 已整理出 8 阶段模型和部分阶段资料清单。
- 可作为后续真实系统的页面原型参考。
