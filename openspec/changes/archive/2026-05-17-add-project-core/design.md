## Context

`define-digital-platform-v1` 已经定义了数字化管理平台第一版的完整边界，但完整第一版同时包含在线表单、资料归档、文件管理平台联动、权限、日志和看板，范围较大。

`add-project-core` 是第一版落地前的最小项目核心切片，只负责让平台具备项目主数据、项目创建、标准 8 阶段初始化、项目列表和项目详情基础状态。该切片为后续资料、文件、权限和看板能力提供项目与阶段基础，但本身不实现这些扩展能力。

## Goals / Non-Goals

**Goals:**

- 建立数字化管理平台自己的项目主数据记录。
- 支持创建项目，并在创建成功后初始化标准 8 阶段。
- 支持项目列表展示项目基础信息、当前阶段和基础状态。
- 支持项目详情展示基础信息、8 阶段进度和当前阶段。
- 保持实现边界足够小，为后续在线表单、文件联动和看板打基础。

**Non-Goals:**

- 不实现在线表单、模板字段录入、草稿、提交或归档文件生成。
- 不实现文件上传、文件管理平台目录创建、文件归档、下载或文件状态查询。
- 不实现复杂权限、角色矩阵、跨系统用户映射或权限同步。
- 不实现业务日志、文件日志或审计追溯。
- 不实现管理层看板、资料齐套率、阶段资料清单、个人待办或阶段推进。

## Decisions

### 决策一：项目主数据先独立落在数字化平台

项目编号、项目名称、客户、项目经理、参与部门、项目状态、计划时间和备注先由数字化平台保存，并作为后续项目资料、阶段、文件映射和看板的业务锚点。

不在本切片中绑定文件管理平台目录，因为文件平台联动需要接口、失败重试、目录映射和文件日志设计，会扩大本变更范围。

### 决策二：创建项目时只初始化阶段，不初始化资料和文件夹

项目创建成功后，系统只生成标准 8 阶段记录。阶段来自已确认的 8 阶段口径：立项阶段、方案设计阶段、合同签订阶段、详细设计阶段、生产制作阶段、预验收阶段、终验收阶段、结题阶段。

不在本切片中生成阶段资料清单，因为资料项、在线表单、文件上传和责任人分配属于后续能力。

### 决策三：项目列表和详情只读取项目与阶段基础数据

项目列表展示项目基础字段、项目状态和当前阶段；项目详情展示基础信息和 8 阶段状态。页面不展示资料齐套率、文件归档状态、待办、日志或看板指标。

不做资料和文件聚合，是为了避免在项目核心切片中引入尚未实现的资料状态和文件状态依赖。

### 决策四：项目状态先作为基础枚举存储

项目状态先支持正常、风险、暂停、延期、完成等基础值的存储和展示，但本切片不实现自动风险判断、自动延期判断、看板统计或状态变更日志。

不在本切片中实现自动状态规则，因为这些规则依赖计划节点、资料缺失、阶段推进和业务日志。

## Risks / Trade-offs

- [后续文件联动需要补充目录映射] -> 本切片只保留项目和阶段稳定标识，后续文件联动变更再增加映射字段和同步状态。
- [后续资料清单需要接入阶段记录] -> 阶段初始化必须使用稳定的阶段标识和顺序，避免后续资料能力重建阶段数据。
- [项目状态只有展示没有自动规则] -> 在页面和接口说明中明确本切片不做自动判断，后续看板或阶段推进变更再扩展。
- [权限边界较弱] -> 本切片只定义项目核心行为，不把角色矩阵写入规格；后续权限变更统一处理。

## Migration Plan

- 新系统初始上线时创建项目与阶段基础表或等价存储结构。
- 若已有测试项目数据，迁移时按项目主数据补齐标准 8 阶段记录。
- 回滚时可以保留已创建项目数据；未接入文件和资料能力，不涉及跨系统回滚。

## Open Questions

- 项目编号是否必须人工录入，还是由数字化平台按规则生成？
- 项目经理和参与部门第一版是否先用文本字段，还是必须关联用户/部门基础表？
- 项目状态是否允许项目创建后直接手工调整，还是暂由管理员维护？
