## Context

项目阶段推进当前同时存在两类容易混淆的机制：

- 旧阶段关口审批：围绕 stage gate approval 的 submit / approve / return / resubmit / history 机制。该机制已不在当前主流程中承担阶段推进前置审批，但代码、API 和仓储仍保留。
- 手动推进阶段：`POST /api/projects/:projectId/stages/advance`。该接口基于当前阶段齐套结果推进到下一阶段，是当前真实仍在使用的阶段切换机制。

`stabilize-solution-design-stage-gate-v1` 已将方案设计专用 workflow 的完成结果接入通用阶段齐套门禁，解决了阶段 2 齐套断链。但完成后仍需要用户点击手动推进，这会把阶段切换变成额外待办。

## Goals

- 实现阶段齐套满足后自动推进到下一阶段。
- 复用现有阶段齐套和阶段推进门禁，不建立第二套判断。
- 明确旧阶段关口审批为 legacy，不作为推进前置。
- 调整工作台和前端口径，避免将正常阶段推进呈现为额外人工待办。
- 明确自动推进的日志、幂等和失败边界。
- 为已齐套但错过自动推进写触发的历史项目或异常项目保留 manual fallback API / 运维兜底能力，不作为主前端入口。

## Non-Goals

- 不新增或修改 migration。
- 不删除旧阶段关口审批表、API 或仓库。
- 不实现合同签订阶段业务。
- 不接文件平台。
- 不改变 8 大阶段或 71 项资料数量。
- 不重构整个项目生命周期底座。

## Current State

### 旧阶段关口审批

旧阶段关口审批是一套独立审批机制，包含提交、审批、退回、重提、历史查询等能力。当前主流程中阶段推进不应依赖该审批机制；工作台不应返回 `stage_gate_approval` 待办，项目详情不应展示 `ProjectStageApprovalPanel` 作为主流程入口。

### 手动阶段推进

当前真实仍在使用的是手动阶段推进接口。它读取当前阶段齐套结果，满足门禁时由用户点击推进按钮进入下一阶段。这个接口可以继续作为兜底能力存在，但不应继续成为主流程中每个阶段结束后的必要用户动作。

## Design Principles

### 1. 自动推进复用现有齐套判断

自动推进只允许由配置的写操作触发点或手动兜底推进接口调用。项目详情、工作台、阶段导航、资料清单等 GET/read-only 接口不得产生阶段变更，也不得写入阶段推进日志。

自动推进必须调用现有阶段齐套/门禁计算链路，包括：

- 普通资料 `completionMode`。
- 资料适用性 `is_applicable`。
- revision / rework 状态。
- 立项在线表单与评审派生状态。
- 方案设计 C04-C19 专用 workflow 派生状态。

自动推进不得重新实现第二套“是否齐套”判断，也不得绕过现有阻塞原因。

### 2. 自动推进必须幂等

同一触发动作可能被重试、重复提交或并发触发。自动推进应具备幂等性：

- 如果项目已不在触发时的阶段，不重复推进。
- 如果阶段已完成，不重复写 `stage.advanced` 成功日志。
- 如果项目已 `ended` 或 `completed`，不推进。
- 如果当前阶段未齐套，不推进。
- 如果下一阶段已是 current，不重复初始化或重复推进。
- 单个触发动作最多推进触发时的当前阶段一次，不在同一触发动作中无控制连跳多个阶段。

### 3. 第 8 阶段完成后自动完成项目

对于第 1-7 阶段，齐套满足后自动进入下一阶段。第 8 阶段齐套满足后应自动将项目置为完成状态，并写项目完成日志。该行为不改变 8 大阶段数量，也不新增资料项。

### 4. 日志保持审计语义

自动推进仍写现有 `stage.advanced` 操作日志，但 summary / details metadata 必须标明：

- 推进方式为系统自动推进。
- 触发动作类型，例如普通资料提交、确认、在线表单提交、适用性变化、方案设计报价接受或投标通过。
- 触发动作的 projectId、stageOrder、documentCode / nodeKey（如适用）。
- 推进前后阶段。
- 使用的齐套摘要。
- actorUserId 使用触发业务动作的当前登录用户。

失败路径不得写成功日志。

### 5. 事务边界固定为同事务回滚

由用户业务动作触发自动推进时，事务边界固定如下：

- 业务动作完成后若齐套门禁未满足，不视为错误；业务动作正常成功返回，项目阶段保持不变。
- 业务动作完成后若齐套门禁满足，但自动推进发生系统错误，则业务动作和自动推进必须在同一事务内回滚，并返回受控错误。
- 系统不得留下“业务动作成功但阶段未变化”的半状态。
- 后续如需异步可靠重试或后台修复任务，必须另行设计，不在本 change 的第一版自动推进事务口径内实现。

## Trigger Points

本 change 已接入以下写操作触发点。读接口仍不得触发自动推进：

- 立项阶段最终完成后重新判断并自动推进。
- 方案设计报价客户接受后重新判断并自动推进。
- 方案设计投标总经理审批通过后重新判断并自动推进。
- 普通资料提交后重新判断。
- 普通资料确认后重新判断。
- 在线表单提交并生成文件成功后重新判断。
- 资料适用性变更后重新判断。
- revision 重提完成后重新判断。

触发点应集中调用一个自动推进服务或 resolver，避免在各仓储中复制推进逻辑。

已枚举并接入的触发点对应后端函数和业务动作包括：

- 立项最终节点通过，即立项评审最终审批通过并使立项阶段齐套的后端动作。
- 方案设计报价客户接受，即报价分支客户接受处理动作。
- 方案设计投标通过，即投标分支总经理审批通过动作。
- 普通资料提交成功。
- 普通资料确认成功。
- 在线表单提交且生成文件成功。
- 资料适用性变更成功。
- revision 重提完成成功。

未列入配置的写操作不得触发自动推进判断。

## Manual Fallback

`POST /api/projects/:projectId/stages/advance` 可以继续保留为运维或异常兜底能力。保留时应满足：

- 与自动推进使用同一齐套/门禁计算。
- 不要求旧阶段关口审批。
- 不作为正常工作台待办。
- 不作为项目详情页主前端入口。
- 用于处理自动推进上线前已经齐套、或业务写触发点错过后无法通过读接口补推进的项目；不得恢复正常 `stage_advance` 待办口径。
- 若阶段已由自动推进完成，应返回幂等或受控错误，不重复推进。

## Frontend Behavior

项目详情页应在业务动作成功后刷新项目详情、阶段资料和工作台数据，展示自动推进后的当前阶段。前端不应要求用户在阶段完成后再点击一次手动推进按钮。

工作台不应为正常项目阶段显示 `stage_advance` 待办。项目详情页不应提供正常手动推进入口；对于已齐套但错过写触发的历史项目或异常项目，manual fallback API 仅作为运维或受控后台调用能力保留。

## Risks / Trade-offs

- [Risk] 业务动作成功但自动推进失败会造成“资料已完成但阶段未变化” -> [Mitigation] 第一版固定同事务回滚并返回受控错误；异步可靠重试另行设计。
- [Risk] 前端动作成功后仍停留旧阶段 -> [Mitigation] 所有可能触发自动推进的动作成功后刷新项目详情和工作台。
- [Risk] 工作台 `stage_advance` 待办口径变化影响旧用户习惯 -> [Mitigation] 将正常推进从待办中移除，错过触发的项目通过 manual fallback API / 运维流程处理。
- [Risk] 旧阶段关口审批 API 仍存在造成误用 -> [Mitigation] 文档和前端明确标记 legacy；后端阶段推进不依赖该审批。
- [Risk] 并发触发重复推进或重复日志 -> [Mitigation] 自动推进服务以项目当前阶段和事务锁为幂等边界。

## Open Questions

- 是否需要进一步增加管理员批量补偿工具，用于批量处理历史已齐套但未推进的项目。
