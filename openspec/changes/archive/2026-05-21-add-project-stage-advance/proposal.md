## Why

当前平台已经具备 8 阶段项目流程、阶段资料清单、资料手工状态流转、资料适用性和按“适用必填资料”计算的阶段齐套摘要，但项目阶段仍停留在只读展示。需要增加阶段手工推进的最小闭环，让已登录用户在当前阶段适用必填资料齐套后，按标准顺序推进项目阶段。

## What Changes

- 后端新增 `POST /api/projects/:projectId/stages/advance` 阶段推进接口，只推进当前阶段，不允许前端指定任意目标阶段。
- 阶段推进必须要求登录态，只做 `requireAuth`，不做项目经理角色校验、复杂权限、角色权限或轻角色校验。
- 服务端校验项目存在、项目有且只有一个当前阶段、当前阶段状态为 `current`，并且只能按 1 -> 2 -> 3 -> ... -> 8 顺序推进。
- 推进前只检查当前阶段的适用必填资料齐套情况；当前阶段资料清单必须已初始化，如果当前阶段没有任何 `project_stage_documents` 资料项记录，必须拒绝推进；只有当前阶段资料项记录存在、但适用必填资料数为 0 时，`requiredTotal = 0` 才视为齐套并允许推进。
- 如果当前阶段存在缺失适用必填资料，后端必须拒绝推进，返回可读错误和缺失资料列表，列表至少包含 `id`、`documentCode`、`documentName`、`status`。
- 推进成功后在事务中将当前阶段改为 `completed`、清除当前标记并记录 `completed_at`；如存在下一阶段，则将下一阶段改为 `current`、设置当前标记并记录 `started_at`。
- 如果当前阶段为第 8 阶段 `closeout`，推进成功后项目不再有当前阶段，并将项目 `status` 改为 `completed`。
- 前端项目详情页展示当前阶段的“手工推进阶段”入口，未齐套或推进失败时展示缺失适用必填资料，推进成功后刷新项目详情、8 阶段状态和阶段资料清单。
- 文案明确阶段推进和齐套门禁基于当前手工状态与人工适用性判断，不代表文件已上传或已归档。
- README 更新说明阶段推进接口、齐套门禁口径、项目完成口径和明确不做范围。
- 本变更不实现阶段回退、跳阶段、批量推进、审批流、业务日志、阶段推进操作人完整日志、管理层看板、文件上传/下载、文件管理平台联动、在线表单、责任人分配、个人待办、通知或权限体系变更。

## Capabilities

### New Capabilities

- 无。

### Modified Capabilities

- `project-core`: 增加项目阶段手工推进接口、顺序推进规则、齐套门禁和第 8 阶段完成项目口径。
- `project-core-frontend`: 扩展项目详情页，展示当前阶段手工推进入口、缺失适用必填资料提示和推进成功后的刷新行为。
- `stage-document-checklist`: 明确阶段资料齐套摘要可作为项目阶段推进门禁的只读输入，同时资料清单、状态操作和适用性操作本身仍不执行阶段推进。

## Impact

- 后端影响 `digital-platform-api` 的项目阶段查询/更新、阶段推进路由、阶段资料齐套摘要读取或等价计算、事务处理和错误响应。
- 前端影响 `digital-platform-web` 的项目详情页、项目 API 客户端、阶段资料清单刷新和推进错误提示。
- 可能需要确认项目阶段表已有或补充 `started_at`、`completed_at` 字段；如字段已存在则复用，不引入阶段推进日志表。
- 不引入文件平台 API、在线表单、审批流、业务日志、通知、看板或复杂权限依赖。
