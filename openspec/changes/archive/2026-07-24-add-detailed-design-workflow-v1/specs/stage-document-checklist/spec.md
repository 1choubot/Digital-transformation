## ADDED Requirements

### Requirement: Detailed Design 7.20 Workflow Execution Baseline
The stage document checklist capability MUST treat detailed design 7.20 workflow outputs as the stage 4 execution baseline and MUST treat v20260629 detailed design documents only as historical or compatibility references unless explicitly reused.

The change MUST NOT introduce a new v20260720 full ordinary document template, MUST NOT change the current ordinary document total or template initialization mechanism, and MUST keep v20260629's 71 items as the compatibility, archive, and derived-display baseline. Detailed design workflow gate and workbench behavior are governed by the 7.20 workflow execution baseline, while ordinary document quantity remains unchanged unless a separate future change explicitly updates templates, migrations, initialization, history migration, and compatibility display.

#### Scenario: New baseline is not constrained by 71 items
- **WHEN** implementing the detailed design workflow change
- **THEN** the system MUST NOT use "keep 71 items" as a hard acceptance constraint for detailed design stage 4
- **AND** any change to detailed design document count, code mapping, or compatibility rows MUST be documented and migrated explicitly
- **AND** the 8 outer project stages MUST remain unchanged

#### Scenario: C25 remains detailed design project kickoff book
- **WHEN** C25 / `4.1` is shown or derived after this change
- **THEN** it MUST mean detailed design stage `项目启动书`
- **AND** it MUST belong to `project_kickoff_meeting` / `项目启动会`
- **AND** it MUST NOT be associated with contract project kickoff notice or contract workflow generated files

#### Scenario: Detailed design old document mapping is explicit
- **WHEN** migrating or initializing detailed design stage documents for the 7.20 baseline
- **THEN** C25 MUST be reused as `项目启动书`
- **AND** C26 MUST be reused as `详细设计工作计划`
- **AND** C27, C28, C29, C30, C32, C33, C34, and C35 MAY be reused only through explicit slot mapping for the 8 detailed design files
- **AND** C31 / `4.7` `控制逻辑流程图` MUST NOT be required by the 7.20 detailed design workflow unless a later change reintroduces it
- **AND** C34 / `4.10` MUST use the 7.20 display name `软件 UI 界面设计 PPT` if reused
- **AND** C36 and C37 MUST be generated by detailed design review online forms
- **AND** C38 and C39 MUST be current upload files for product plan drawing and parts list
- **AND** C40 MUST be retained as drawing review record history if reused
- **AND** C41 MUST be renamed or displayed as `客户会签图纸扫描件` if reused

#### Scenario: C31 is compatibility only and never blocks workflow
- **WHEN** C31 / `4.7` `控制逻辑流程图` appears in ordinary checklist views
- **THEN** it MUST be displayed only as not applicable, compatibility archive, or non-workflow main-process reference
- **AND** it MUST NOT block detailed design workflow completion or stage advance
- **AND** it MUST NOT be treated as a required main-process upload for the 7.20 workflow

#### Scenario: C40 is conditional history with no-issue non-blocking behavior
- **WHEN** the workflow has no drawing review issue and no C40 history exists
- **THEN** C40 / `4.16` MUST NOT block stage advance
- **AND** the system MUST NOT require a drawing review record upload
- **WHEN** a drawing review issue exists
- **THEN** the return path MUST require C40 upload before return
- **AND** uploaded C40 history MUST be retained
- **AND** the final pass after rework MUST NOT require C40 to be uploaded again
- **AND** if C40 history exists, the corresponding record MUST remain downloadable for the research and development center manager

#### Scenario: Deprecated detailed design documents remain compatibility records
- **WHEN** a legacy or mock project still contains v20260629 detailed design rows not used by the 7.20 workflow
- **THEN** those rows MAY remain available for archive, audit, or compatibility display
- **AND** they MUST NOT appear as detailed design workflow main nodes
- **AND** they MUST NOT satisfy detailed design workflow node completion unless explicitly mapped by the workflow

#### Scenario: Ordinary checklist is not the execution source
- **WHEN** the ordinary stage document checklist renders detailed design-related items
- **THEN** it MUST consume workflow-derived state only
- **AND** it MUST NOT become the source used to determine node completion, permissions, workbench todos, stage gates, or automatic advance
- **AND** it MUST NOT reverse-drive workflow state from checklist status back into detailed design workflow tables

### Requirement: Detailed Design Derived Completion
The ordinary stage document checklist MUST receive detailed design completion state from the dedicated workflow only through explicit derived completion mapping.

#### Scenario: Workflow-derived completion for reused documents
- **WHEN** a reused detailed design document such as C25, C26, C36, C37, C38, C39, C40, or C41 is displayed in the ordinary checklist
- **THEN** the checklist MUST show completion based on detailed design workflow node submission/approval state and current revision mapping, not upload presence alone
- **AND** C38/C39/C41 MUST still require their workflow nodes to be approved, while their file readiness is based on whether a current file exists rather than `file revision >= node revision`
- **AND** the checklist MUST show derived blocking reasons when the workflow state is incomplete
- **AND** ordinary document card status MUST NOT override workflow state for these main process outputs

#### Scenario: No-upload detailed design file derived display
- **WHEN** C27-C30 or C32-C35 has a workflow no-upload mark for the current detailed design revision
- **THEN** the ordinary checklist MAY show the row as workflow-derived completed or not-applicable according to the existing dedicated-workflow display convention only after `detailed_design` node submission succeeds
- **AND** C31 MUST remain compatibility-only
- **AND** C25, C26, C38, C39, and C41 MUST NOT be satisfiable by no-upload marks

#### Scenario: Ordinary checklist only exposes compatibility and traceability
- **WHEN** the ordinary stage document checklist renders detailed design-related items
- **THEN** it MUST show only compatibility, archive, derived completion, or generated/upload traceability states
- **AND** it MUST NOT expose detailed design workflow write actions as ordinary document card actions
- **AND** the user MUST NOT be able to bypass project kickoff book upload, role assignment, design review, drawing review, countersign, or stage gate from the checklist

#### Scenario: No old blueprint and workflow double entry
- **WHEN** the detailed design stage is rendered in workspace or checklist-oriented views
- **THEN** the main process entry MUST be the detailed design workflow
- **AND** ordinary checklist cards MAY be used for archive or traceability only
- **AND** users MUST NOT see both an old C25-C41 blueprint node and a new workflow node as two equal ways to perform the same action
