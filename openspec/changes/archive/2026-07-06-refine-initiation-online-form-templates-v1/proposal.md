## Why

v20260629 / 71 项新项目运行基线和立项阶段责任人、退回路径已经完成验收。下一步需要把 `1.1 项目需求表`、`1.2 项目立项审批表`、`1.3 项目立项通知` 的在线表单内容从当前简化版对齐到真实业务模板，避免继续基于临时字段推进。

## What Changes

- 规划 `1.1 项目需求表` 在线表单对齐 `项目需求表-模板.xlsx` 的市场调研结构化字段。
- 规划 `1.2 项目立项审批表` 在线表单对齐 `项目立项审批表-模板.xlsx` 的基础信息、商务 7 项评分、技术 4 项评分和三方意见结构。
- 规划 `1.3 项目立项通知` 在线表单对齐 `关于确定项目名称及编号的通知-模板.docx` 的固定正文、项目信息表格和落款。
- 规划项目编号门禁顺序调整为：`1.2 总经理最终通过 -> 允许填写/生成项目编号 -> 1.3 自动带出项目编号并提交 -> 立项阶段齐套`。
- 明确不迁移旧简化表单数据；旧项目或旧数据返工/重新填写时按新模板处理。

责任人口径：

- `1.1 项目需求表` 由商务负责人填写。
- `1.2 项目立项审批表` 由商务负责人填写整张表，包括技术模块评分。
- `1.3 项目立项通知` 由营销中心负责人填写/确认。
- 技术负责人第一版只作为项目技术协同/后续承接人，不新增表单填写任务。

不做：

- 不生成 Excel / Word / PDF。
- 不接文件平台。
- 不新增资料项。
- 不改变 v20260629 / 71 项模板数量。
- 不新增技术负责人填写任务。
- 不改变当前 `1.2` 商务评价、技术评价、总经理审批流。
- 不改变项目结束、退回、返工主流程。
- 不在本规划 change 修改业务代码、数据库 schema 或 migration。

## Capabilities

### New Capabilities

- 无。

### Modified Capabilities

- `project-core`: 规划真实模板对齐后的 1.1 / 1.2 / 1.3 在线表单 schema 和项目编号门禁顺序。
- `project-core-frontend`: 规划前端渲染分组字段、评分项、表格结构、通知预览和项目编号前置校验。
- `stage-document-checklist`: 明确资料项仍是 1.1 / 1.2 / 1.3，不新增资料项，不改变 71 项数量，不生成真实文件。
- `technical-architecture`: 固化在线表单 schema 扩展、现有表单数据存储复用、不接文件平台和不生成文件的架构边界。
