## Why

当前阶段资料清单已经具备资料状态、适用性、齐套摘要、阶段推进和业务日志能力，但资料项仍缺少明确的责任人字段操作闭环。项目执行时无法在资料项维度说明“谁负责准备或跟进该资料”，会影响后续我的待办、在线表单、文件上传和项目协同能力建设。

本变更用于补齐资料项责任人手工分配的最小闭环，在不引入复杂权限和个人待办的前提下，为资料项建立可展示、可变更、可追溯的责任人关联。

## What Changes

- 在项目级阶段资料项上支持手工分配一个数字化平台用户作为责任人。
- 支持清空资料项责任人。
- 新增 `GET /api/users/responsibility-candidates` 启用用户候选列表能力，用于责任人选择；该能力只要求登录态，不要求平台管理员，也不得被用户管理维护路由的 `requirePlatformAdmin` 包住。
- 阶段资料清单查询返回责任人字段、责任人安全用户信息和最近一次责任人变更追溯字段。
- 分配或清空责任人时记录最近一次变更人和变更时间。
- 分配或清空责任人成功后写入项目业务操作日志 `document.responsible_changed`。
- 项目详情页展示资料项当前责任人，并提供责任人选择和清空操作。
- README 补充 `GET /api/users/responsibility-candidates` 责任人候选用户接口、资料项责任人分配接口、前端展示能力和边界说明。
- 明确第一版只做手工分配，不做责任人权限校验、个人待办、通知、文件平台联动、在线表单或复杂 RBAC。

## Capabilities

### New Capabilities

- 无。本变更基于现有用户、阶段资料清单、项目业务日志和项目详情页能力扩展，不新增独立 capability。

### Modified Capabilities

- `stage-document-checklist`: 增加资料项责任人分配、清空、责任人字段返回、责任人变更追溯和资料清单边界要求。
- `business-operation-log`: 增加 `document.responsible_changed` 项目业务操作日志动作及结构化详情要求。
- `project-core-frontend`: 增加项目详情页资料责任人展示、选择、清空和刷新行为要求。
- `basic-users-auth`: 增加用于业务分配的启用用户候选列表安全返回口径。
- `user-management`: 明确责任人候选列表不是用户管理维护接口，且不使用平台管理员边界。

## Impact

本变更影响后端阶段资料清单查询、资料项责任人分配接口、`GET /api/users/responsibility-candidates` 启用用户候选接口、项目业务日志写入、前端项目详情页资料清单展示和 README 文档。数据库层面需要使用或补充项目级资料项责任人字段和最近一次责任人变更追溯字段。权限边界保持为责任人分配和候选用户查询只要求 `requireAuth`，不使用 `isPlatformAdmin`，不引入项目级权限、资料权限、文件权限或复杂角色权限。
