## Context

`project-core` 和 `project-core-frontend` 已完成项目创建、列表和详情基础能力，但当前数字化管理平台还没有自己的用户登录能力。后续资料提交、文件归档和业务日志都需要稳定识别真实操作人，因此需要先建立第一版基础用户与登录能力。

技术架构已确认：数字化平台后端是独立 Node.js + Express 服务，数据保存在数字化平台自己的 MySQL 中；数字化平台不能直接读取、共用或迁移文件管理平台数据库。文件管理平台用户只允许在本阶段通过数字化平台用户表中的可选 `filePlatformUserId` 字段预留映射关系。

## Goals / Non-Goals

**Goals:**

- 在数字化平台 MySQL 中新增基础用户表设计。
- 用户信息包含账号、姓名、部门、角色、启用状态和可选 `filePlatformUserId`。
- 支持基础账号密码登录，并能识别当前登录用户。
- 支持主动退出登录，结束当前登录态。
- 实现前端登录页。
- 前端保存登录态，并在项目列表、新建项目、项目详情等项目页面展示当前登录用户。
- 项目创建必须基于当前登录态识别创建人，并记录到项目主数据。
- 项目列表和项目详情返回创建人基础信息或创建人字段，用于追溯创建人。
- 为后续资料提交、文件归档和日志记录提供真实操作人基础。

**Non-Goals:**

- 不实现复杂菜单权限。
- 不实现文件下载权限。
- 不实现与文件管理平台用户自动同步。
- 不实现部门权限继承。
- 不实现审批流。
- 不实现单点登录。
- 不实现细粒度按钮权限。
- 不直接读取、共用或迁移文件管理平台数据库。
- 不实现文件平台用户同步、权限同步或下载权限判断。
- 不把创建人记录扩展为复杂项目权限控制。
- 不实现用户管理前端页面；第一版用户可通过初始化数据或种子脚本准备。

## Decisions

### 决策一：数字化平台维护独立用户表

新增数字化平台自己的 `users` 表，保存账号、姓名、部门、角色、启用状态、可选 `filePlatformUserId` 和登录所需凭据字段。

不选择直接复用文件管理平台用户表，因为技术架构要求数字化平台独立服务端和独立 MySQL 数据边界，后续跨平台联动必须通过 HTTP API 或显式映射完成。

### 决策二：`filePlatformUserId` 只作为可选映射字段

`filePlatformUserId` 允许为空，只用于后续建立数字化平台用户与文件管理平台用户的关联。第一版登录不依赖该字段，项目页面也不根据该字段判断文件下载或文件平台权限。

不选择在本变更中做文件平台用户同步或权限同步，因为这会把基础登录能力扩展到跨平台权限能力，超出当前范围。

### 决策三：账号唯一，禁用状态控制登录

用户账号在数字化平台内必须唯一。启用用户可以登录；禁用用户不得建立新的登录态，但历史业务数据和未来日志中的操作人引用需要保留。

不选择物理删除用户作为第一版主要控制方式，因为后续日志和归档追溯需要稳定用户标识。

### 决策四：角色先作为基础用户属性保存

角色字段用于标识第一版用户类型，例如系统管理员、管理层、项目经理、部门负责人、普通成员。当前变更只保存和展示角色，不用角色实现菜单权限、按钮权限、文件下载权限或部门继承权限。

不选择在本变更中引入完整 RBAC，因为当前目标是先得到真实操作人，复杂权限应作为后续独立能力设计。

### 决策五：使用基础登录接口和当前用户接口

后端提供最小认证接口：

- `POST /api/auth/login`：使用账号和密码登录，成功后返回登录态和当前用户信息。
- `GET /api/auth/me`：根据登录态返回当前用户信息。
- `POST /api/auth/logout`：主动结束当前登录态。

登录响应不能返回密码哈希或其他凭据内部字段。错误场景需要区分可读提示，例如账号或密码错误、用户已禁用、登录态无效或已过期。

### 决策六：第一版登录态由前端保存并随请求携带

前端保存登录态，启动时通过 `GET /api/auth/me` 恢复当前用户。项目页面展示当前登录用户，后续 API 请求可以携带登录态，为后续操作人识别打基础。

不选择单点登录或企业身份源集成，因为当前阶段明确不做 SSO。

### 决策七：首批用户通过初始化数据准备

本变更不做用户管理页面。实现阶段可通过迁移、种子脚本或环境变量初始化至少一个启用用户，用于第一版登录验证。

不选择在本变更中做完整用户增删改查页面，因为用户管理 UI 会扩大范围，并且不是当前用户明确要求的第一批能力。

### 决策八：项目创建记录创建人，但不引入权限控制

`projects` 表新增可空 `createdByUserId` 字段。创建新项目时，后端必须从当前有效登录态识别当前用户，并把该用户 ID 写入 `createdByUserId`。如果请求未登录或登录态无效，后端必须拒绝创建并返回需要登录的错误。

不选择让前端直接提交创建人 ID，因为创建人必须来自后端认证结果，避免伪造。该字段只用于认证和操作人记录，不用于复杂菜单权限、项目访问权限、文件下载权限或按钮权限判断。

### 决策九：兼容历史项目创建人为空

已有测试项目和历史项目允许 `createdByUserId` 为空。项目列表和详情返回创建人基础信息或创建人字段时，必须能处理创建人为空的项目，例如返回 `null` 或空创建人字段。

不选择强制回填历史数据，因为当前已有测试数据不一定能可靠追溯真实创建人，强行回填会制造错误审计信息。

### 决策十：项目查询返回创建人追溯字段

项目列表和项目详情在返回项目基础信息时，应包含创建人基础信息或创建人字段，至少支持追溯是谁创建的项目。创建人基础信息应来自数字化平台用户表，不能访问文件管理平台数据库。

不选择在本变更中记录完整业务操作日志，因为日志能力属于后续独立范围；当前只补齐项目创建人追溯字段。

## Risks / Trade-offs

- [本地保存登录态有浏览器端泄露风险] -> 第一版仅用于内网基础登录，避免保存敏感用户字段；后续如接入更严格安全要求，可升级为更强会话策略。
- [没有用户管理页面导致维护不便] -> 第一版通过初始化数据或种子脚本解决首批用户，用户管理能力后续单独拆 change。
- [角色字段被误认为已经有权限控制] -> 页面和接口只展示或返回角色，不基于角色实现复杂权限、按钮权限或文件权限。
- [文件平台用户映射被误用为同步机制] -> `filePlatformUserId` 只保存可选映射值，本变更不读取文件平台数据库，不同步用户，不判断文件权限。
- [历史项目没有创建人导致追溯不完整] -> `createdByUserId` 保持可空，列表和详情用空创建人字段表达未知，不伪造历史操作人。
- [前端携带创建人字段被误信任] -> 后端只信任当前登录态识别出的用户，忽略或拒绝前端提交的创建人字段。

## Migration Plan

- 新增数字化平台用户相关数据库迁移。
- 新增至少一个可登录启用用户的初始化方式。
- 新增基础登录和当前用户接口。
- 新增退出登录接口。
- 项目表新增可空 `createdByUserId` 字段，并在项目创建时由后端写入当前登录用户 ID。
- 项目列表和项目详情返回创建人基础信息或创建人字段。
- 前端新增登录页和登录态保存。
- 前端创建项目请求携带登录态，未登录时展示需要登录的错误。
- 项目页面接入当前用户展示。
- 验证登录成功、退出登录、登录失败、禁用用户、登录态恢复、项目页面显示当前登录用户、项目创建人记录、历史项目创建人为空兼容，以及未发生文件平台数据库访问或同步行为。
