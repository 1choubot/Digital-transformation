## Context

数字化平台已经完成项目主流程和阶段资料核心能力。当前用户模型已有 `users.department`、`users.role` 和 `users.is_platform_admin`，但尚未区分“组织角色”和“岗位/职务文本”。项目经理当前也是项目主数据文本字段，尚不能表达项目内身份、职责范围或后续权限边界。

本 change 先做规格设计，目标是明确组织和项目治理模型，为后续权限约束、审批节点、报表和工作流能力打基础。

## Goals / Non-Goals

**Goals:**

- 固定四个业务部门：运营中心、营销中心、制造中心、研发中心。
- 固定五类组织角色：总经理、系统管理员、总经理助理、中心负责人、员工。
- 将独立全局角色和部门内角色分开建模。
- 将项目经理建模为项目内用户关联，而不是全局角色。
- 明确项目模式只影响业务说明、展示和默认分配建议，不改变流程和状态机。
- 沿用资料项责任人表达技术负责人和项目参与人派生规则。
- 规划第一版最小权限边界，不引入复杂 RBAC 或审批流引擎。

**Non-Goals:**

- 不做日报周报。
- 不做文件管理平台联动。
- 不做自动通知。
- 不做自动阶段流转。
- 不做复杂 RBAC。
- 不做第三方账号管理。
- 不做项目成员表、技术负责人表或项目参与人表。
- 不做阶段审批流引擎。
- 不做旧模拟数据兼容迁移。

## Decisions

### 用户组织模型

新增 `organizationRole` 业务字段，建议数据库字段为 `organization_role`。枚举值：

- `general_manager`: 总经理
- `system_admin`: 系统管理员
- `general_manager_assistant`: 总经理助理
- `center_manager`: 中心负责人
- `employee`: 员工

保留现有 `role` 字段作为岗位/职务展示文本，例如“软件工程师”“机械工程师”“营销专员”等，避免把岗位文本和组织角色混在一个字段里。

部门字段继续使用 `department`，但后续应使用稳定枚举值：

- `operations_center`: 运营中心
- `marketing_center`: 营销中心
- `manufacturing_center`: 制造中心
- `rd_center`: 研发中心

部门约束：

- `general_manager`、`system_admin`、`general_manager_assistant` 的 `department` 必须为空或 `null`。
- `center_manager`、`employee` 的 `department` 必须是四个业务部门之一。

### 与现有字段关系

- 推荐新增 `organization_role` 字段，不复用现有 `role` 字段。
- `role` 保留为岗位/职务展示文本。
- `isPlatformAdmin` 继续作为系统管理能力开关。
- `organizationRole = system_admin` 必须与系统管理权限开关保持一致：系统管理员用户必须具备 `isPlatformAdmin = true`，拥有用户、组织和基础配置管理能力。
- 第一版系统管理账号保护口径为：至少保留一个启用用户同时满足 `isEnabled = true`、`organizationRole = system_admin`、`isPlatformAdmin = true`。
- 用户管理新增、编辑、禁用、删除或权限降级操作不得导致同时满足上述三个条件的账号数量变成 0。
- 长期权限口径不得写成“`system_admin` 或 `isPlatformAdmin` 任一存在即可”。如果后续为了清理旧模拟数据需要临时恢复逻辑，只能作为初始化或修复脚本行为，不能作为运行时长期权限判断。

### 全局角色边界

总经理：

- 查看所有项目、所有流程、所有资料进度。
- 可参与关键审批，具体审批规则后续按流程节点细化。

系统管理员：

- 管账号、组织、基础配置。
- 默认不参与项目审批。
- 默认不作为项目经理、资料负责人或资料审批人。
- 不因系统管理身份自动获得业务项目全局查看、资料确认/退回、阶段推进、责任人分配或适用性变更权限。

总经理助理：

- 查看所有项目、所有流程、所有资料进度。
- 可生成汇总类报表。
- 只拥有思维导图中的查看、生成、汇总类权限：生成所有部门工作日报、生成项目工作日报、生成项目一览表、查看所有业务流程。
- 不审批、不确认资料、不退回资料、不推进阶段、不分配或清空资料责任人，不代替总经理审批。
- 后端必须拒绝总经理助理直接调用资料确认、资料退回、阶段推进、资料责任人分配/清空和代替总经理审批接口；不能只依赖前端隐藏按钮。错误码第一版可使用稳定的 `FORBIDDEN_OPERATION`，如代码库已有统一权限错误码则沿用既有错误码。

中心负责人：

- 本 change 暂不实现 `/api/users` 下中心负责人管理本中心员工的维护入口；用户新增、编辑、启用/禁用和重置密码仍由 `system_admin + isPlatformAdmin` 执行。中心负责人本中心员工管理作为后续单独能力时，必须限制在本中心员工范围内。
- 查看本中心相关项目和资料。
- 分配本中心员工或项目允许范围内用户为本中心相关资料责任人。
- 审批本中心相关资料或流程节点；不得跨中心审批、退回、分配责任人、标记不适用或推进阶段。

员工：

- 查看自己相关项目。
- 查看和提交自己负责的资料。
- 查看自己的资料任务。

### 项目可见范围

项目列表、项目详情、项目总览看板、项目详情支撑的阶段资料清单、项目业务日志读取和阶段资料附件上传/列表/下载/删除 MUST 基于当前用户过滤可见项目：

- `general_manager`、`general_manager_assistant` 可查看全部项目。
- `system_admin` 不因系统管理身份自动获得业务项目查看权限；第一版按无业务项目权限处理。
- `projectManagerUserId = 当前用户 id` 的项目可见。
- 当前用户在该项目中至少负责一项资料的项目可见。
- `center_manager` 只能查看本中心相关项目；第一版以 `participating_departments` 包含其部门，或项目中存在本中心责任人资料作为相关依据。
- `employee` 只能查看自己负责资料或自己作为项目经理的项目。

无权限访问具体项目详情或详情支撑只读接口时，后端返回稳定权限错误码 `FORBIDDEN_OPERATION`，不伪装成项目不存在；项目确实不存在时仍返回项目不存在错误。

附件接口无权失败时，后端不得读取上传文件、不得写入附件文件、不得新增附件记录、不得软删除附件、不得写业务日志。

### 项目模式

新增 `projectMode`，建议数据库字段为 `project_mode`。第一版枚举：

- `self_developed`: 自研模式，公司员工自己产出资料。
- `outsourced`: 供应链/外包模式，第三方产出，系统内仍由公司员工负责检查、整理成公司模板并提交。

项目模式不代表两套流程：

- 自研和外包使用同一套 8 阶段。
- 自研和外包使用同一套 54 项资料。
- 不改变阶段推进规则。
- 不改变资料状态机。
- 不改变适用/不适用规则。
- 不改变附件规则。
- 第一版只影响业务说明、展示和默认分配建议。

标准开发项目模式暂不实现；如业务后续确认，可作为单独 change 增加枚举和默认分配建议。

### 参与部门

新增或沿用项目参与部门字段 `participatingDepartments`，数据库字段为 `participating_departments`。该字段用于辅助中心负责人判断本中心相关项目，保存值 MUST 是四个业务部门稳定枚举数组：

- `operations_center`
- `marketing_center`
- `manufacturing_center`
- `rd_center`

规则：

- 前端显示中文部门名，但请求体必须提交枚举数组。
- 后端不得接受中文部门名、未知值、非数组非空值或任意自由文本。
- 空值或空数组表示未配置参与部门。
- 重复部门值去重后保存。
- 非法参与部门返回稳定错误码 `INVALID_PARTICIPATING_DEPARTMENT`，HTTP 400。
- 中心负责人项目可见范围依赖该枚举数组，或项目中存在本中心责任人资料；不得因自由文本参与部门获得或丢失预期外权限。

### 项目经理

新增项目经理用户关联字段：

- `projectManagerUserId`
- 响应中返回 `projectManagerUser`

规则：

- 新的项目经理权威字段是 `projectManagerUserId`；响应返回 `projectManagerUser`。
- 旧 `projectManager` 文本不得再作为权限判断依据。
- 项目创建/编辑接口后续应以 `projectManagerUserId` 为准。
- 如果响应仍保留 `projectManager` 文本，只能作为展示兼容字段，且必须从 `projectManagerUser.name` 派生，不能由前端任意提交。
- 项目经理必须是启用用户。
- 项目经理必须是 `center_manager` 或 `employee`。
- 项目经理必须隶属于四个业务部门之一。
- `general_manager`、`system_admin`、`general_manager_assistant` 不能被指定为项目经理。
- 非法项目经理 ID、项目经理不存在或被禁用、项目经理组织角色不允许，后续实现必须返回稳定错误码：`INVALID_PROJECT_MANAGER_USER_ID`、`PROJECT_MANAGER_USER_NOT_FOUND_OR_DISABLED`、`PROJECT_MANAGER_USER_ROLE_NOT_ALLOWED`。
- 项目经理是项目内身份，不是全局角色。
- 项目经理负责项目推进、任务分配、催办、查看项目全量进度和齐套后阶段推进。
- 项目经理可以同时是某些资料的责任人。
- 项目经理默认不是资料审批人。
- 如果项目经理同时也是某中心负责人，他能审批是因为“中心负责人”身份，不是因为“项目经理”身份。
- 项目经理可以推进自己负责的项目，且仍必须满足齐套门禁。
- 非该项目项目经理不得仅凭普通员工身份推进该项目。
- 总经理助理不得推进任何项目。
- 项目经理可以分配或调整自己负责项目的资料责任人。
- 非该项目项目经理、非中心负责人、非系统允许角色，不得分配或调整该项目资料责任人。
- 项目经理不能仅因项目经理身份确认或退回资料。

### 资料责任人与项目参与人

- 继续沿用阶段资料项的 `responsibleUserId`。
- 不新增技术负责人表。
- 不新增项目参与人表。
- 技术负责人 = 某些技术资料的资料责任人。
- 项目参与人 = 在该项目中至少负责一项资料的人。
- 项目参与人从资料责任人派生。
- 项目经理可以同时被分配为某些资料的责任人。
- 资料责任人负责提交、整理资料，不代表拥有审批权。
- 责任人候选用户第一版只返回启用用户、`organizationRole` 为 `center_manager` 或 `employee`、`department` 为四个业务部门之一的用户。
- 责任人候选用户不得返回总经理、系统管理员、总经理助理、禁用用户、密码字段或非展示必需的平台管理员内部字段。

### 第一版权限边界

本 change 不实现完整权限引擎，但后续实现最小边界时应遵循：

- 系统管理员只默认拥有系统配置、账号、组织和基础配置管理权限，不默认参与项目审批。
- 总经理可查看全局项目、流程和资料进度，可参与关键审批；具体审批节点后续细化。
- 总经理助理可查看全局并生成汇总类报表，但不审批、不退回、不推进阶段、不分配资料责任人。
- 中心负责人可查看本中心相关项目和资料，分配本中心员工或项目允许范围内用户为本中心相关资料责任人，审批本中心相关资料或流程节点；本 change 不开放中心负责人用户管理维护接口。
- 员工可查看自己相关项目、查看和提交自己负责的资料、查看自己的资料任务。
- 项目经理可查看自己负责项目的全量进度，分配或调整自己负责项目的资料责任人，催办资料，在齐套后推进自己负责项目的阶段；资料责任人分配范围第一版可限制在项目参与部门内或责任人候选用户集合内。
- 项目经理不因项目经理身份自动获得资料审批权。
- 后端必须在资料确认、资料退回、阶段推进、责任人分配/清空、标记不适用和恢复适用等接口执行权限拒绝，前端隐藏按钮只是用户体验，不是安全边界。

### 审批和阶段推进边界

- 本 change 不实现完整审批流引擎。
- 当前资料确认/退回能力继续存在。
- 后续实现权限约束时，资料确认/退回应由总经理或本中心相关资料的中心负责人等审批身份执行。
- 阶段推进仍以当前阶段适用必填资料全部 `confirmed` 为门禁。
- 阶段推进还必须满足操作者权限边界：项目经理只能推进自己负责项目；中心负责人只能推进本中心相关项目；总经理助理和系统管理员不得推进任何项目；普通员工不得仅凭员工身份推进非自己负责的项目。
- 标记不适用和恢复适用会影响齐套门禁，必须与资料审批相同按总经理或本中心相关中心负责人边界拒绝无关用户，防止任意登录用户绕过必填资料齐套门禁。
- 不做“上传文件后自动进入下一阶段”。
- 不做自动通知。

## Migration Strategy

后续实现可能需要新增迁移为用户和项目增加字段。当前仍处于开发阶段，不做旧模拟数据兼容迁移。后续实现可通过初始化脚本或默认值建立开发库基础数据，但不得引入旧模型和新模型长期共存。

## Risks / Trade-offs

- 组织角色和 `isPlatformAdmin` 同时存在可能造成理解混乱：通过设计明确 `organizationRole` 是组织身份，`isPlatformAdmin` 是系统管理能力开关。
- 项目经理从文本升级到用户关联会影响现有项目表和前端表单：接受该变化，后续实现不做旧模拟数据兼容迁移。
- 不建立项目参与人表会限制复杂项目成员权限：第一版接受，通过资料责任人派生项目参与人，后续如确需项目成员管理再单独设计。
- 总经理助理具备全局查看和汇总能力但不能审批：后续前端必须避免展示审批、退回、阶段推进和责任人分配入口。
