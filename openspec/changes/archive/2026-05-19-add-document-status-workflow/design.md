## 背景

数字化管理平台当前已经有项目级阶段资料项数据，表名为 `project_stage_documents`，并且项目详情页已经能只读展示阶段资料清单。该表已经保存四种基础状态：`not_submitted`、`submitted`、`confirmed`、`returned`，也已经有部分提交和确认追溯字段，但还没有资料项状态变更接口，也没有前端操作入口。

本变更是在文件上传、在线表单、文件平台联动、资料齐套率和阶段推进之前，先补一个最小可用的资料项状态流转闭环。该闭环明确定位为“手工标记状态”，用于试运行和流程验证，不代表已经真实上传文件或提交在线表单。

## 目标 / 非目标

**目标：**

- 新增需要登录态的后端资料项状态操作，用于手工标记提交、确认和退回。
- 在后端统一执行资料项状态机校验，保证非法流转被一致拒绝。
- 保存提交、确认、退回动作的最小追溯字段。
- 在现有阶段资料清单查询结果中返回状态追溯字段。
- 在项目详情页阶段资料清单中增加手工状态操作按钮和退回原因输入。
- 页面文案明确说明手工标记状态不等同于文件上传或在线表单提交。

**非目标：**

- 不实现文件上传、文件下载或文件平台联动。
- 不实现在线表单编辑、草稿保存、归档文件生成或表单提交记录。
- 不实现资料齐套率、缺失资料统计或阶段推进。
- 不实现业务日志、个人待办、责任人分配或通知流程。
- 不实现复杂权限、角色权限或轻角色校验；本变更只通过 `requireAuth` 要求登录。

## 设计决策

### 决策一：定义为手工状态操作，而不是真实资料提交

前端和后端接口会使用 `submit`、`confirm`、`return` 这类动作名称，但页面文案必须明确这些动作只是手工标记状态。这样可以避免用户把 `submitted` 误解为已经上传了文件，或者已经产生了在线表单记录。

备选方案是等文件上传和在线表单能力完成后再做状态流转。这个方案会推迟基础状态机和确认/退回闭环的验证，也会把多个较大的能力耦合在一起。

### 决策二：在项目资源下使用显式动作接口

后端建议新增以下接口：

- `POST /api/projects/:projectId/stage-documents/:documentId/submit`
- `POST /api/projects/:projectId/stage-documents/:documentId/confirm`
- `POST /api/projects/:projectId/stage-documents/:documentId/return`

每个接口都必须先校验资料项属于 URL 中的项目，再执行状态流转。退回接口必须要求请求体提供非空 `returnReason`。

备选方案是使用一个通用 `PATCH` 接口，由前端传目标状态。显式动作接口可以减少前端随意传入目标状态的空间，也让每个动作的校验路径更清楚。

### 决策三：状态机集中在后端领域逻辑中校验

允许的状态流转为：

- `not_submitted` -> `submitted`
- `returned` -> `submitted`
- `submitted` -> `confirmed`
- `submitted` -> `returned`

仓储层更新资料项状态时，应在事务或带当前状态条件的受保护更新路径中完成。非法流转必须返回可读的领域错误，并且不能部分更新追溯字段。

备选方案是只在前端禁用非法按钮。前端禁用有助于交互体验，但不能作为状态机的可信来源。

### 决策四：最小追溯字段保存在 `project_stage_documents`

实现时应复用现有字段：

- `submitted_by_user_id`、`submitted_at`
- `confirmed_by_user_id`、`confirmed_at`
- `returned_at`

迁移需要补充：

- `returned_by_user_id BIGINT UNSIGNED NULL`
- `return_reason VARCHAR(1000) NULL`

当已退回资料项被重新手工标记提交时，系统必须清空 `returned_by_user_id`、`returned_at` 和 `return_reason`。确认已提交资料项不代表文件已存储，也不代表表单已归档成功。

备选方案是新增独立状态历史表。历史表更完整，但本变更明确不做业务日志，只需要保存最近一次动作的最小追溯字段。

### 决策五：登录校验保持最小化

所有状态操作接口都必须使用 `requireAuth`，并使用当前登录用户 ID 写入追溯字段。本变更不引入角色校验。这样可以与当前基础登录能力保持一致，避免在项目成员范围和责任人分配规则尚未建立前提前定义权限体系。

备选方案是按角色限制提交、确认和退回。该能力应放到后续权限变更中处理，因为当前平台还没有实现责任人分配或项目成员范围。

## 风险 / 取舍

- [手工标记容易被误解为真实提交] -> 页面文案必须说明这些动作不会上传文件，也不会提交在线表单。
- [没有角色校验意味着任意已登录用户都可以变更状态] -> 在范围中明确只做 `requireAuth`，角色权限后续单独实现。
- [最近一次追溯字段不能保留完整历史] -> 对最小闭环可接受，完整业务日志仍然不在本变更范围内。
- [并发点击可能产生状态竞争] -> 后端状态更新必须校验当前状态，拒绝基于旧状态的流转请求。
- [已有数据库可能缺少新字段] -> 增加独立迁移，并同步更新数据表初始化逻辑，保证新旧环境一致。

## 迁移计划

1. 新增数据库迁移，为 `project_stage_documents` 增加 `returned_by_user_id` 和 `return_reason`。
2. 更新数据表初始化逻辑，确保全新环境也会创建这两个字段。
3. 保持已有数据有效：现有 `not_submitted`、`submitted`、`confirmed`、`returned` 状态不变。已有 `returned` 行在执行新的退回动作前，允许退回追溯字段为空。
4. 如需回滚，可在确认业务实现不再依赖这两个字段后移除新增字段。

## 待确认问题

- 本最小变更没有待确认问题。角色权限、业务日志和真实文件/表单提交均有意留到后续变更处理。
