## Context

方案设计 workflow 已包含 C05 项目方案分析表、C15 内部方案评审记录表和 C16 客户方案评审记录表的在线表单与生成文件能力。但当前字段集合和模板填充存在断层：

- C05 在线表单字段与 `项目方案分析表-模板.xlsx` 不一致，无法完整填入模板中的环境要求、场地情况、工件描述、作业工艺、目标说明和图片区域。
- C15/C16 生成文件中的评审类型和项目目标描述存在乱码或编码口径错误。
- C15/C16 前端已有记录人字段，但生成文件记录人为空，导致表单、保存数据和下载文件不一致。

## Goals

- 让 C05、C15、C16 的前端字段、后端保存字段和 Excel 模板填充字段保持一致。
- C05 字段命名尽量对齐既有项目需求表字段，但实际展示、保存和生成以 `项目方案分析表-模板.xlsx` 的字段与区域为准。
- C15/C16 生成文件中评审类型、项目目标描述和记录人可读、准确、可测试。
- 生成文件必须写入真实 Excel 单元格，不只修前端展示。

## Non-Goals

- 不改方案设计节点状态机、审批/退回/revision 规则或自动推进。
- 不实现合同签订阶段业务。
- 不接文件平台，不生成 PDF。
- 不改 C17、C18、C19。
- 不新增无模板依据的业务字段。

## C05 项目方案分析表设计

### 字段来源

C05 字段 schema 和后端 payload 应只保留 `项目方案分析表-模板.xlsx` 已确认字段。字段命名可复用或对齐立项阶段 `项目需求表` 已有命名，但不得保留模板无依据的旧字段。确认字段包括：

- 项目基础信息：项目编号、项目名称、客户名称，只读自动带入。
- 环境要求：工作温度、储存温度、工作湿度、储存湿度、噪音、IP、防腐、海拔、防爆等。
- 场地情况：场地说明、电源、气源、液压源、吊装设备等。
- 工件描述：工件描述文本。
- 作业工艺：作业工艺说明。
- 目标：项目目标描述。

### 图片处理

C05 支持复用现有在线表单图片存储和 Excel 图片渲染能力，不新增表。允许的图片字段为：

- `siteConditionImages`：场地情况图片。
- `workpieceImages`：工件描述图片。
- `operationProcessImages`：作业工艺图片。
- `projectTargetImages`：目标图片。

图片上传和删除权限按方案设计技术负责人、当前项目仍可写、`solution_analysis` 节点可编辑状态判断。C05 DTO 必须返回 C05 资料行 `stageDocumentId` 和已有图片列表，前端据此上传、下载、删除。生成 `项目方案分析表.xlsx` 时必须将图片嵌入模板对应区域，测试需断言生成包包含 media、drawing 和 anchor。

C05 图片资料锚点必须同时支持目标资料编码 `C05` 和 v20260629 legacy 编码 `2.2`。通用在线表单图片 API 必须按资料类型限制字段集：立项项目需求表只允许原有场地、工件、作业工艺三类图片；C05 项目方案分析表允许场地、工件、作业工艺、目标四类图片。C05 图片查看权限应覆盖方案设计 workflow 角色用户，上传/删除权限仍只开放给方案设计技术负责人和可编辑 analysis 节点。

当 C05 已提交并生成文件后，图片上传或删除会改变生成文件输入。系统必须避免图片列表与下载文件形成两套真相。本 change 采用“图片变更后使当前 C05 生成文件失效”的口径：图片变更事务成功后清空当前 analysis form 的 generated file 存储字段并将生成状态置为未生成，保留表单 submitted 状态；用户重新提交 C05 表单后再按最新图片生成 Excel。前端在图片变更成功后必须同步禁用 solution_analysis 节点提交动作，避免旧 workflow 节点权限让用户点击后才被后端拦截。

### 模板填充

生成文件必须按 `项目方案分析表-模板.xlsx` 的真实单元格或区域写入字段。实现不得只新增前端字段，也不得只把字段保存到 JSON 后不写入 Excel。

当前模板已确认的 C05 文本写入位置为：

- B2：项目编号。
- E2：项目名称。
- B3/D3：工作温度、储存温度。
- B4/D4：工作湿度、储存湿度。
- B5/D5：噪音、IP 防护等级。
- B6/D6：防腐等级、海拔高度。
- B7：防爆要求。
- B8：场地说明。
- B9：电源、气源、液压源。
- B10：吊装设备。
- B12：工件描述。
- B17：作业工艺。
- B27：项目目标说明。

当前模板已确认的 C05 图片区域为：

- 场地情况：拆分 B8:E8、B9:E9、B10:E10，文本保留在 B8:C8、B9:C9、B10:C10，图片嵌入 D8:E10。
- 工件描述：拆分 B12:E15，文本保留在 B12:E13，图片嵌入 B14:E15。
- 作业工艺：拆分 B17:E25，文本保留在 B17:E21，图片嵌入 B22:E25。
- 目标：拆分 B27:E36，文本保留在 B27:E31，图片嵌入 B32:E36。

新保存/提交 payload 应按 C05 白名单过滤，不再沉积 `customerRequirements`、`technicalRisks`、`solutionScope`。C15/C16 评审表中的项目需求分析和项目风险评估字段仍保留，不受 C05 字段删除影响。

## C15/C16 方案评审记录表设计

### 评审类型

C15/C16 复用 `方案评审记录表-模板.xlsx`，但必须通过节点上下文和 revision 明确评审类型：

- C15 内部方案评审记录表生成 `内部，第（n）次`。
- C16 客户方案评审记录表生成 `甲方，第（n）次`。

评审类型写入必须保留中文、标点和数字可读，不允许乱码、符号残缺或错误编码。已知模板 B3 继承 Wingdings 2 样式，写入时必须对文本字体做单元格级覆盖或等价处理，不能全局破坏边框和合并单元格。

### 项目目标描述

项目目标描述应支持多行/数组字段。前端可使用多行结构化输入，后端保存为数组或等价结构，生成 Excel 时按模板预留行逐行写入。

生成内容必须与在线表单提交值一致，不得出现乱码、截断或把数组序列化成不可读 JSON 文本。已知模板 B12/B13 继承 Wingdings 2 样式，写入项目目标描述时必须覆盖为正常中文字体。

### 记录人

记录人处理规则：

- 前端只读默认当前用户。
- 保存/提交 payload 必须包含 `recorder`。
- 后端生成文件优先使用 `form.recorder`。
- 当 `form.recorder` 为空时，回退 `submitted_by_display_name`。
- 若仍为空，回退提交账号或当前用户可读名。
- 生成文件必须写入 A42 合并单元格，格式为 `记录人：xxx`。
- 不得再使用 `A42=记录人：` 与 `B42=xxx` 的两段式映射。

## Testing Strategy

- C05 必须新增单元格级断言，确认环境要求、场地情况、工件描述、作业工艺和目标说明写入模板，并断言旧三字段不会进入新提交保存结果。
- C05 必须断言图片生成文件包含 `xl/media/*`、`xl/drawings/*` 和 drawing anchor，并断言图片区域 merge adjustment 不覆盖文本区域。
- C05 必须断言图片上传/删除后当前生成文件失效，且 legacy `2.2` 资料锚点仍可用于 DTO、上传和删除。
- 通用在线表单图片 API 必须断言立项项目需求表拒绝 C05 专属 `projectTargetImages` 字段。
- C15/C16 必须分别断言评审类型、项目目标描述和记录人写入正确，且 B3、B12:B14 不使用 Wingdings 2 文本样式。
- C15/C16 必须独立生成并断言不串数据。
- 测试不得只检查生成文件存在、文件名正确或生成状态为成功。

## Risks

- 真实模板单元格位置可能与现有字段命名不一致。实现应以模板 mapping manifest 或等价注册表为准，不得靠字段名猜测位置。
- 若旧草稿中仍残留客户需求、技术风险、方案范围，新提交时必须通过白名单过滤，避免继续污染 C05 表单 JSON。
- 图片区域拆分需要尊重模板合并单元格，图片嵌入不得破坏文本区域、边框和合并单元格的主要可读性。
- C15/C16 复用同一模板时，若评审类型或 nodeKey 判断分散在多个位置，容易再次串数据。实现应集中映射评审上下文。
