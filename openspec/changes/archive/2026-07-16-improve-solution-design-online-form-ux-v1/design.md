## Context

方案设计阶段已有专用 workflow，C05/C15/C16 在线表单提交后会生成 Excel 文件，但用户仍需要在节点操作区再执行一次“提交节点”才能进入审批；C05 同时依赖产品功能框图，因此表单提交和节点提交存在不同门禁。C07-C14 已支持“无需上传”豁免，但第一版要求填写原因或备注，增加了低价值输入负担。C15/C16 方案评审记录表目前把“项目实施计划”作为普通文本域，不能按需求、目标、风险、建议逐项追踪计划。报价结果处理当前偏下拉式，业务动作不够直接。

本 change 是方案设计阶段体验收口，建立在当前 C05/C15/C16/C18 在线表单、C07-C14 豁免和报价结果处理能力之上，不改变立项阶段后端、8 阶段、71 项资料数量或合同签订阶段。

## Goals / Non-Goals

**Goals:**

- 让 C05/C15/C16 在线表单提交生成成功后自动尝试推进当前节点，减少重复点击。
- C05 缺产品功能框图等其他必需资料时，表单提交成功但节点不推进，并返回可读缺项提示。
- C15/C16 表单提交成功且无其他门禁时直接进入待研发中心负责人审批。
- 取消 C07-C14 “无需上传”的备注必填/输入负担，保留操作日志和取消豁免能力。
- 将 C15/C16 实施计划从普通文本升级为结构化条目，并继续渲染到现有模板单元格。
- 将报价结果处理改为三个按钮，并复用既有后端状态和 payload 语义。
- C18 报价单在线表单实时预览报价明细行金额、总金额和人民币大写金额，但后端仍是唯一权威计算方。

**Non-Goals:**

- 不改立项阶段后端。
- 不改变 8 大阶段。
- 不改变 71 项资料数量。
- 不改合同签订阶段业务。
- 不改旧关口审批。
- 不改 C17 成本估算四段流程。
- 不改变报价单在线表单字段和 Word 模板生成规则，除非复核发现提示或回归测试缺口。
- 不处理无关 untracked。
- 不 push；实现完成并验证后再归档、提交，归档前先 review。

## Decisions

### Decision 1: 在线表单提交后自动尝试节点提交

后端在 C05、C15、C16 表单提交和生成文件成功后，复用现有节点提交编排自动尝试提交对应节点。自动提交不得直接手写局部状态，必须调用或抽取与手动 `submitSolutionDesignWorkflowNode` 等价的状态更新、operation log、待办收敛和 blocking reason 更新逻辑。

- C05：若 C05 表单生成成功但 C06 产品功能框图未上传，接口返回表单提交成功和生成成功，同时返回节点未推进及缺失项提示。
- C05：若 C05 表单和 C06 产品功能框图均满足门禁，接口自动提交 `solution_analysis` 节点，使节点进入待研发中心负责人审批。
- C15/C16：表单生成成功后若当前 review 节点可提交，接口自动提交 review 节点并进入待研发中心负责人审批。
- 生成文件失败时不自动提交节点，并继续返回明确业务错误或失败状态，不得让节点进入待审批。

前端 C05、C15、C16 在线表单页面不再同时暴露“提交表单”和手动“提交节点”入口；这些页面通过 `SolutionNodeActions` 隐藏提交节点按钮，但保留审批通过和审批退回动作。后端 `submitSolutionDesignWorkflowNode` 和 `canSubmitNode` 权限保持不变，用作自动提交编排和异常兜底。

Rationale: 表单提交是用户的真实完成动作；自动提交节点能减少重复操作，但必须复用现有门禁，避免绕过 C05 产品功能图等必需资料。

### Decision 2: 自动推进结果作为 DTO 中的明确结果返回

表单提交接口应返回可供前端展示的 `autoSubmit` 或等价结构，至少包含：

- `attempted`
- `submitted`
- `nodeKey`
- `nodeStatus`
- `blockingReasons` 或 `missingRequirements`
- `message`

前端据此展示“表单已提交并生成文件”“节点已提交待审批”或“表单已提交，仍缺产品功能框图”等提示。前端不得自行推导节点是否已提交。

Rationale: 后端是节点门禁唯一来源，前端只消费结果，避免 UI 和后端状态不一致。

### Decision 3: C07-C14 豁免允许空备注

后端 normalize 允许 `exemption_reason` 为空字符串或 null。标记“无需上传”仍只允许技术负责人对 C07-C14 操作，仍记录状态、操作人和操作时间，operation log 继续记录项目、节点、槽位和操作者；备注字段可以为空。

取消豁免、重新上传自动取消豁免和门禁恢复逻辑保持不变。

Rationale: 当前业务希望“无需上传”是轻量标记，不要求技术负责人额外编写说明。

### Decision 4: C15/C16 实施计划由四类来源字段生成结构化条目

C15/C16 表单 payload 增加 `implementationPlanItems`，每项包含：

- `sourceType`: `requirement`、`target`、`risk`、`suggestion`
- `sourceLabel`: `需求`、`目标`、`风险`、`建议`
- `sourceIndex`: 从 1 开始
- `sourceText`: 来源条目文本
- `planText`: 用户填写的实施计划内容

四个来源字段按非空来源条目生成并保留顺序；后端对数组 payload 按数组项处理，对 legacy 字符串 payload 继续按非空行兼容拆分：

- 项目需求分析 -> 需求1、需求2...
- 项目目标描述 -> 目标1、目标2...
- 项目风险评估 -> 风险1、风险2...
- 项目方案建议 -> 建议1、建议2...

前端 C15/C16 使用显式加行控件维护四个来源字段，而不是用单个 textarea 的换行拆分来源。每个来源条目本身仍是 textarea；条目内回车只作为当前条目内容换行，不代表新增需求、目标、风险或建议。新增条目必须点击“添加需求/添加目标/添加风险/添加建议”，删除条目必须使用对应删除动作。

保存草稿时允许计划项未填完整，但提交时必须校验每个自动生成项都有非空 `planText`。后端保存时根据来源字段重新 normalize 结构化条目，不能信任前端提交不存在来源的计划项。

第一版计划项保留规则固定为：

- 来源字段按非空来源条目生成当前有效计划项。
- 结构化 `implementationPlanItems` 存在时，以结构化项为准；用户清空的 `planText` MUST 保持为空并在提交时触发必填校验，不得被旧 `actionItems` 静默回填。
- 旧 `actionItems` 仅在 payload 完全没有 `implementationPlanItems` 时用于兼容迁移。
- 同一 `sourceType` 的来源条目数量未变化时，前后端 MUST 优先按 `sourceType + sourceIndex` 保留同位置 `planText`，即使编辑后出现重复 `sourceText` 也不得抢占其他条目的计划内容。
- 同一 `sourceType` 的来源条目数量发生增删时，前后端 SHOULD 优先按同一 `sourceType + sourceText` 保留 `planText`，避免删除或插入上方来源条目后计划内容串到错误来源。
- 来源条目删除后，对应计划项删除。
- 新增来源条目后，新增空 `planText` 计划项，提交时必须补齐。

生成 Excel 时，项目需求分析来源条目写入 C15/C16 模板的 B9-B11：第 1 条写入 B9，第 2 条写入 B10，第 3 条及后续内容沿用现有 `repeatRows` 规则合并写入 B11，避免超过 3 条时丢失内容。结构化计划拼接为当前模板“项目实施计划”单元格文本，固定顺序为需求、目标、风险、建议；每项一行，格式为 `需求1：实施计划内容`、`目标1：实施计划内容`、`风险1：实施计划内容`、`建议1：实施计划内容`，使用 Excel 单元格换行连接。

Rationale: 实施计划应逐项回应需求、目标、风险和建议，同时保持现有 Excel 模板单元格不变。

### Decision 5: 报价结果三个按钮复用既有 payload

前端将报价结果处理入口改为三个按钮：

- 客户接受报价 -> `{ result: accepted }`
- 结束项目 -> `{ result: rejected, action: end_project, returnReason }`
- 审批不通过 -> `{ result: rejected, action: return_to_rd_cost, returnReason }`

如后端要求原因，按钮点击后弹窗填写；不再使用下拉菜单选择结果和动作。后端接口和状态机不需要新增第三套状态，只需继续接受现有 payload。

Rationale: 三个按钮与业务语义直接对应，降低错误选择概率，并保留现有后端流程稳定性。

### Decision 6: C18 只做口径复核

C18 报价单在线表单已经由提交生成 Word 文件、处理报价结果和 C18 派生完成组成。本 change 只复核 C18 是否符合“表单提交成功后清晰提示生成状态、失败不推进、后续结果按钮明确”的口径；若后端流程已符合，则只补前端提示或回归测试，不改变报价单字段、后端金额权威计算或模板映射。

前端报价明细表实时预览金额：

- 数量最多支持 4 位小数，单价格式按 2 位小数处理。
- 单行金额按 `数量 * 单价` 计算后四舍五入到 2 位。
- 总金额按每行已四舍五入金额汇总，再四舍五入到 2 位。
- 人民币大写金额按实时总金额预览计算，并与后端大写金额规则保持一致。
- 数量或单价为空、非法时，该行金额显示为空且不计入总金额。
- 金额列只读，用户不得手填行金额；新增或删除报价明细行后总金额同步刷新。

提交报价单时，前端金额和人民币大写预览不得作为权威字段提交给后端；现有 payload 继续只提交报价明细基础字段，后端保存、提交和生成 Word 文件时仍重新 normalize 数量/单价、重算行金额、总金额和大写金额。若历史或异常 payload 携带 `amount`、`totalAmount`、`totalAmountUppercase` 等字段，后端仍不得信任这些字段作为生成依据。

Rationale: 避免在 UX change 中混入报价单生成规则重构。

## Risks / Trade-offs

- [Risk] 自动提交节点可能改变用户对“保存表单”和“提交节点”的心理模型。 -> Mitigation: 仅在用户点击“提交表单”并确认后触发，保存草稿不自动推进。
- [Risk] C05 缺产品功能图时用户误以为节点已提交。 -> Mitigation: 后端返回 `autoSubmit.submitted=false` 和缺项提示，前端必须展示。
- [Risk] 结构化实施计划和旧 `implementationPlan` 文本兼容复杂。 -> Mitigation: 第一版保存结构化字段，提交/生成时以后端 normalize 的 `implementationPlanItems` 为准；必要时从旧文本只读展示但不作为新结构来源。
- [Risk] 空备注豁免降低审计信息量。 -> Mitigation: operation log 仍记录项目、节点、槽位、操作者和时间。
- [Risk] 报价结果按钮若文案不清会误触结束项目。 -> Mitigation: 结束项目和审批不通过按钮使用确认弹窗，要求填写原因时再提交。

## Migration Plan

1. 不新增数据库 migration；优先复用现有表单 JSON payload 和上传槽豁免字段。
2. 后端先扩展 C05/C15/C16 表单 submit DTO 和 normalize，再接入自动节点提交。
3. 前端再调整确认弹窗、结构化实施计划控件、无需上传按钮和报价结果按钮。
4. 测试覆盖 C05/C15/C16/C18 关键路径、生成失败、缺项、空备注豁免、报价三按钮。

Rollback strategy: 可回退前端按钮/控件和后端自动提交逻辑；已保存的 `implementationPlanItems` 保留在表单 JSON 中，不影响旧文本字段读取。若需要回退，生成文件可重新从旧文本字段渲染。

## Open Questions

- None. 第一版按本设计执行：不新增 migration，实施计划结构化字段保存在表单 JSON payload 内。
