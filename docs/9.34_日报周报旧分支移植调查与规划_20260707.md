## 7. 安全移植策略

### 7.1 总体策略

- 不直接 merge 整个旧分支。
- 按后端模块、前端模块、migration、测试和文档分批移植。
- 所有全局入口文件必须以当前主线为基底手工接入，不允许用旧分支版本覆盖。
- 移植过程不得回退当前立项阶段流程、模板文件生成、项目编号 1.3 确定口径或 71 项资料模板。

### 7.2 后端策略

- 先移植报表 domain/repository/service/storage，再接入 routes。
- `app.js`、`errorHandler.js`、`env.js`、`routes/me.js`、`routes/users.js`、`userRepository.js` 必须对照当前主线手工合并。
- 保留当前用户、认证、权限和组织角色模型；同事分支依赖的职位、中心、角色字段要逐项适配。
- 报表附件应使用数字化平台内部存储，不接文件平台。

### 7.3 前端策略

- 移植日报/周报/中心日报页面和 API。
- `router.js`、`App.vue`、导航和 `styles.css` 必须按当前主线结构接入，不能带回旧 UI 改动覆盖项目详情、立项工作区或生成文件入口。
- `api/http.js` 是唯一直接重叠文件，应单独 review 拦截器、错误处理和 token 行为。

### 7.4 数据库策略

- 不复制旧分支 migration 编号。
- 在当前主线 migration 序列后重排日报/周报表结构。
- 先建立空 schema 迁移，再单独考虑样本数据导入脚本。
- 不导入完整 `digital_platform_with_data.sql`，不覆盖当前数据。

### 7.5 验证策略

- `digital-platform-api npm run check`
- `digital-platform-web npm run build`
- `openspec validate --all --strict`
- 日报/周报后端单测和关键流程 smoke
- 日报填写、日报补录、周报从日报预填、周报审批、中心日报汇总手测
- 立项阶段现有 smoke 回归，确保 1.1/1.2/1.3 模板文件生成和编号流程不回退

## 8. 本轮最小实现口径

本轮从旧分支手工移植最小可用切片：

- 后端接入日报填写/保存/提交、日报补录、日报图片附件、周报创建/编辑/提交、周报从日报确定性预填、周报审批、中心日报查询和可关闭的中心日报计划配置。
- 数据库新增当前主线后续编号 migration `017_create_daily_weekly_reporting_core.sql`，只建空 schema，不导入样本数据，不覆盖现有数据。当前 `package.json` 没有通用 migrate 脚本，交接或部署到新环境时必须明确手动执行该 migration，避免日报/周报页面因缺表失败。
- 权限使用当前 `department`、`organization_role`、`is_platform_admin` 模型，不新增 `users.job_title`。
- 前端按当前 `App.vue`、`router.js`、`api/http.js` 接入报表页面，不带回旧分支应用壳。
- 暂缓 AI 预填、复杂评分/最终评分、报表导出、导出美化和样本 SQL 导入。
- 旧分支 DB 集成测试依赖旧 migration 编号、`users.job_title`、导出/AI服务和额外依赖，本轮不直接搬运；新增轻量 Node 单元测试覆盖工作日规则、权限矩阵和报表 payload 校验。
