## ADDED Requirements

### Requirement: 立项关键路径后端回归测试
项目核心 MUST 通过自动化测试覆盖立项阶段关键路径，并将这些测试作为现有业务口径的回归保护；测试 MUST NOT 改变现有提交、审批、退回、重提、项目编号、阶段推进或工作台待办规则。

#### Scenario: 1.1 项目需求表提交回归
- **WHEN** 后端执行 `1.1 项目需求表` 在线表单提交
- **THEN** 自动化测试 MUST 断言提交成功后的表单状态、资料完成状态、责任/权限结果和后续 `1.2` 可处理状态符合现有业务规则
- **AND** 测试 MUST NOT 引入普通资料提交路径绕过立项在线表单规则

#### Scenario: 1.2 多节点审批回归
- **WHEN** 后端执行 `1.2 项目立项审批表` 的营销评价、研发评价和总经理最终审批通过
- **THEN** 自动化测试 MUST 断言 `1.2` 派生完成由在线表单提交、营销评价完成、研发评价完成、总经理最终审批通过和未清返工状态共同决定
- **AND** 测试 MUST 断言总经理最终通过后允许后续 `1.3 项目立项通知` 进入可提交链路

#### Scenario: 1.2 审批不通过触发 1.1 精准返工回归
- **WHEN** 总经理最终审批不通过 `1.2 项目立项审批表`
- **THEN** 自动化测试 MUST 断言系统触发 `1.1 项目需求表` 精准返工状态
- **AND** 自动化测试 MUST 断言未清返工会阻断 `1.3 项目立项通知` 和立项阶段自动推进

#### Scenario: 1.1 返工重提后 1.2 重新审批回归
- **WHEN** `1.1 项目需求表` 因 `1.2` 审批不通过进入精准返工后完成返工重提
- **THEN** 自动化测试 MUST 断言 `1.1` 未清返工状态被清除
- **AND** 自动化测试 MUST 断言 `1.2 项目立项审批表` 可重新进入营销评价、研发评价和总经理最终审批链路

#### Scenario: 1.3 项目立项通知与项目编号回归
- **WHEN** 后端提交 `1.3 项目立项通知`
- **THEN** 自动化测试 MUST 断言提交前必须满足 `1.2` 已完成且 `1.1` 不存在未清精准返工
- **AND** 自动化测试 MUST 断言 `1.3` 提交写入项目编号并执行非空项目编号唯一性校验
- **AND** 自动化测试 MUST 断言重复项目编号被拒绝且不会错误推进阶段

#### Scenario: 立项完成自动推进回归
- **WHEN** `1.1`、`1.2`、`1.3` 按现有规则满足立项阶段齐套
- **THEN** 自动化测试 MUST 断言项目当前阶段自动推进到方案设计阶段
- **AND** 自动化测试 MUST 断言阶段状态、当前阶段标记和工作台关键待办口径不因推进发生乱增或漏项

### Requirement: 方案设计派生齐套自动推进集成回归
项目核心 MUST 通过自动化测试覆盖第 2 阶段方案设计 C04-C19 派生齐套后自动推进到第 3 阶段的集成链路，防止阶段资料派生完成与自动推进触发之间断链。

#### Scenario: 方案设计完成自动推进到合同阶段
- **WHEN** 方案设计 workflow 按现有报价或投标路径完成，并且 C04-C19 派生完成满足第 2 阶段齐套
- **THEN** 自动化测试 MUST 断言项目当前阶段自动推进到第 3 阶段合同签订
- **AND** 自动化测试 MUST 断言该测试不创建合同签订阶段业务数据、不改变 8 大阶段和 71 项资料数量
- **AND** 自动化测试 MUST 断言该链路继续使用现有自动推进语义而不是恢复普通手工阶段推进主流程
