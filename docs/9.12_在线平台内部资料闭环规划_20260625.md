# 9.12 在线平台内部资料闭环规划 20260625

## 当前目标

领导最新口径为：暂时不用联动文件管理平台，先保存在在线平台上。因此本规划先暂停文件管理平台联动实现，让在线平台内部把以下闭环做正确：

- 项目创建允许项目编号后置。
- 阶段资料按 20260625 `completionMode` 完成规则处理。
- 资料提交、资料审核、审核待办和阶段推进按资料自身规则流转。
- 文件/附件先保存在在线平台现有附件系统。
- 不调用文件管理平台，不创建 file platform folder mapping，不产生归档状态。
- 当前实现依据收敛到 20260625 流程图、docs/9.11 和本 docs/9.12，不再把 20260624/v20260624 作为当前主流程或 active 模板依据。

本规划不实现代码、不改数据库、不执行 migration；后续实现需在本规划 review 通过后单独执行。

## 当前代码现状差距

只读检查当前代码后，确认主要差距如下：

- `digital-platform-api/src/domain/projects.js` 的 `normalizeCreateProjectInput` 仍将 `projectCode` 作为创建必填字段。
- `digital-platform-api/migrations/001_project_core.sql` 中 `projects.project_code` 仍为 `VARCHAR(64) NOT NULL`，并带 `UNIQUE KEY uk_projects_project_code`。
- `digital-platform-web/src/pages/ProjectCreatePage.vue` 新建项目表单仍校验项目编号必填。
- `digital-platform-api/src/domain/stageDocumentTemplates.js` 和 `stageDocumentTemplateItemsV20260624.js` 仍以 `isRequired`、`submitMode`、`targetFolderPath`、`targetFolderId` 等字段为主，没有 `completionMode`。
- `digital-platform-api/src/domain/stageDocumentStatus.js` 当前状态流转为 `submit -> submitted`，再 `confirm -> confirmed`，没有按 `completionMode` 分流。
- `digital-platform-api/src/repositories/stageDocuments/shared.js` 阶段齐套仍只认 `status = confirmed`。
- `digital-platform-api/src/repositories/projects/stageAdvanceRepository.js` 阶段推进仍检查 `currentStage.approval_status = approved`，属于泛化阶段审批门禁。
- 在线平台已有阶段资料附件表和附件上传/查看/下载能力，可作为本阶段默认文件保存方式。

## 项目编号口径

- 创建项目时允许 `projectCode` 为空。
- `projectCode` 不再是创建项目必填项。
- 项目立项审批通过、项目立项通知发布后，再填写或生成 `projectCode`。第一版建议触发点为 `1.2 项目立项审批表` 审核通过，且 `1.3 项目立项通知` 已提交或上传完成。
- `projectCode` 仍需唯一，但唯一性只约束非空编号。
- 多个空 `projectCode` 必须允许共存；空字符串应在后端归一化为 `NULL` 或等价空值。
- 后置填写或生成 `projectCode` 只更新项目编号和追溯字段，不得重建阶段、资料清单或附件。
- 项目编号后置填写权限沿用项目维护权限，或由项目经理、管理员等现有权限边界控制，不新增复杂权限模型。
- 如果 MySQL 当前普通 `UNIQUE` 索引允许多个 `NULL`，可保留唯一索引并将字段改为 nullable；如果实际数据库或兼容层不允许多个空值，则需要调整为非空唯一索引策略。
- 前端项目列表、项目详情、工作台和任务列表遇到空编号时显示“待生成”或等价文案，不应展示空白、`undefined` 或报错。
- 排序、筛选、搜索不能因为 `projectCode` 为空出错；搜索应继续支持项目名称、客户、项目经理等字段。

## 审批/提交口径

资料完成规则以 `docs/9.11_20260625项目流程资料审批口径规划.md` 最终 64 项为准：

| completionMode | 数量 | 完成口径 |
| --- | ---: | --- |
| submit_only | 33 | 提交或上传后即完成 |
| approval_required | 24 | 提交后进入待审核，审核通过才完成 |
| conditional_submit | 7 | 条件未触发时不算缺失、不阻塞；条件触发后提交或上传即完成 |
| conditional_approval | 0 | 当前 64 项没有资料使用，仅保留未来扩展口径 |

具体规则：

- `submit_only`：不需要审核按钮，不进入审核待办；提交或上传后计为完成。
- `approval_required`：提交后为待审核；确认/审批通过后计为完成；退回后需要重新提交。
- `conditional_submit`：第一版不新增复杂触发流程引擎，复用现有 `isApplicable` 适用性机制表达触发状态；`isApplicable=false` 表示未触发或不适用，不计缺失、不阻塞阶段推进；`isApplicable=true` 表示已触发，按 `submit_only` 处理。
- `conditional_approval`：当前数量为 0，不在本轮实现复杂流程；后续如业务新增再单独变更。
- 不做合同审批流、采购审批流、付款流、发票审批流、设计变更流程引擎。
- 不做泛化阶段关口审批；资料级 `approval_required` 审核与阶段推进分开表达。

接口完成状态口径：

- 后端返回资料项时必须包含 `completionMode`。
- 后端必须返回派生完成状态字段，例如 `isComplete`、`completionStatus` 或等价字段，避免前端仅凭基础 `status` 误判。
- `submit_only` 且 `status=submitted` 时，派生完成状态必须表示 `completed` / 已完成，不得显示为待审核。
- `approval_required` 且 `status=submitted` 时，派生完成状态必须表示 `pending_review` / 待审核。
- `returned` 状态仍表示未完成，必须重新提交或重新审核后才可完成。
- 项目总览、我的资料任务、我的工作台、项目详情资料操作和附件区域都必须使用派生完成状态，不得把 `confirmed` 作为唯一完成口径。
- 我的工作台只展示资料责任待办、资料审核待办和阶段推进待办；不返回、不展示 `stage_gate_approval`、待我阶段关口审批或待阶段关口审批入口。

关键资料口径：

- `4.14 产品平面图 = submit_only`
- `4.15 产品零部件清单 = submit_only`
- `4.16 图纸审查记录 = approval_required`
- `3.4 发票（预付款） = submit_only`
- `6.2 发票（发货款） = submit_only`
- `8.1 发票（尾款） = submit_only`

## 阶段推进口径

- 阶段推进只检查当前阶段“适用资料”是否按各自 `completionMode` 完成。
- `submit_only` 达到 `submitted` 即完成。
- `approval_required` 达到 `confirmed` 才完成。
- `conditional_submit` 使用现有 `isApplicable` 表达触发状态：`isApplicable=false` 时不阻塞；`isApplicable=true` 后达到 `submitted` 即完成。
- 当前 64 项没有 `conditional_approval`；未来若使用，触发后达到 `confirmed` 才完成。
- 不再额外叠加一个泛化阶段审批门禁。
- 不再要求“全部资料 `confirmed`”才推进；阶段推进只看当前阶段适用资料按各自 `completionMode` 派生出的完成状态。
- 资料全部按 `completionMode` 完成后即可由有权限用户推进阶段，不再要求额外提交或通过阶段关口审批。
- 仍保留权限控制：谁能提交、谁能审核、谁能推进阶段，按已有角色、项目经理、中心归属和资料归属中心规则规划。
- 阶段推进继续只推进当前阶段，不新增跳阶段、回退、自动流转或复杂流程编排。

## 文件保存口径

- 本阶段文件只保存在在线平台现有附件系统。
- 不调用文件管理平台。
- 不写 `filePlatformClient`。
- 不保存 file-platform project/stage/document folder mapping。
- 不产生 `not_archived`、`archived`、`archive_failed` 归档状态。
- 不展示文件平台文件列表、下载入口或归档失败重试入口。
- 附件上传、查看、下载和删除均表示在线平台附件状态，不表示文件平台归档完成。
- 后续若恢复文件平台联动，再单独 change 处理，并需重新审查归档触发规则是否按 `completionMode` 执行。

## 实施顺序建议

1. 数据库和模板字段规划落地：`project_code` nullable、`completion_mode` 字段、64 项模板重新初始化。
2. 后端项目创建和项目编号后置填写/更新，后置节点为 `1.2 项目立项审批表` 通过且 `1.3 项目立项通知` 已提交。
3. 后端资料状态和审核待办按 `completionMode` 分流。
4. 后端资料接口返回 `completionMode` 和派生完成状态，确保 `submit_only + submitted` 展示为已完成。
5. 阶段齐套和阶段推进改为按 `completionMode` 计算，并移除泛化阶段审批门禁。
6. 前端项目编号非必填、空编号展示、后置填写入口。
7. 前端资料列表按钮/状态/文案按 `completionMode` 和派生完成状态分流。
8. 补齐 projectCode、submit_only、approval_required、conditional_submit、阶段推进和附件本地保存测试。
