> 生成日期：2026-07-09
> 项目名称：凯尔夫智能测控技术有限责任公司 - 数字化管理平台（Digital Transformation Platform）

## 1. 项目定位与背景

### 1.1 平台定位

数字化管理平台定位为**智能制造项目全过程管理主入口**，覆盖从市场调研到项目结题的8阶段全生命周期。与现有文件管理平台形成上下层关系：

- **数字化管理平台**：管项目、流程、阶段、资料状态、业务权限、管理看板
- **文件管理平台**：管文件、目录、下载、权限、日志（底层文件底座，不推翻）

### 1.2 第一版核心口径

> 项目从这里创建，阶段资料从这里提交；文件管理平台不推翻，继续作为文件存储、下载和权限控制系统。第一版先不做复杂权限同步，也不先接钉钉金蝶，重点把**项目创建 → 阶段流程 → 资料提交 → 自动归档 → 进度看板**跑通。

### 1.3 第一版做 / 不做

| 做 | 不做 |
|---|---|
| 项目创建、8阶段流程 | 钉钉深度集成、金蝶深度集成 |
| 阶段资料清单、责任人 | 复杂审批流引擎 |
| 在线表单、文件上传 | 复杂跨系统权限同步 |
| 文件归档到文件管理平台 | 文件在线预览 |
| 资料状态、阶段推进 | 多套部门模板、移动端 |
| 项目总览看板、业务日志 | 替换文件管理平台现有功能 |

## 2. 技术架构总览

### 2.1 系统架构

```
┌─────────────────────────────────────────────────────────┐
│                    数字化管理平台（前端）                      │
│  Vue 3 + Element Plus + Vite + 自研Hash路由                │
│  http://localhost:5174  →  API代理到 localhost:3001        │
├─────────────────────────────────────────────────────────┤
│                    数字化管理平台（后端）                      │
│  Node.js 20 + Express 4 + ESM                            │
│  domain → repositories → services → storage              │
│  MySQL 17张核心表 + 17个migration                           │
├──────────────────────┬──────────────────────────────────┤
│                      │        文件管理平台                   │
│                      │  （现有系统，弱绑定联动）               │
│                      │  项目目录创建 / 文件归档 / 下载权限     │
└──────────────────────┴──────────────────────────────────┤
│                    DeepSeek API（可选）                     │
│  周报AI整理 + 评分辅助                                      │
└─────────────────────────────────────────────────────────┘
```

### 2.2 后端分层架构（digital-platform-api）

| 层 | 目录 | 职责 |
|---|---|---|
| **路由层** | `src/routes/` | Express路由定义与HTTP适配（25+项目端点 + 日报/周报/中心日报端点） |
| **中间件层** | `src/middleware/` | 认证(requireAuth/requirePlatformAdmin)、错误处理(20+种领域错误映射)、文件上传解析 |
| **领域层** | `src/domain/` | **纯业务逻辑，无IO** — 输入校验、权限判断、状态流转规则、常量与枚举 |
| **数据访问层** | `src/repositories/` | SQL查询 + 业务编排 + 权限校验 + 操作日志写入（按项目域/资料域/报表域拆分） |
| **服务层** | `src/services/` | 跨模块编排 — 导航/导出/AI/定时调度 |
| **存储层** | `src/storage/` | 文件本地存储 — 附件/生成文件/表单图片/日报附件 |
| **工具层** | `src/utils/` | Office XML渲染引擎（xlsx/docx模板填充） |
| **配置层** | `src/config/` | 环境变量 + 前端导航菜单配置 |

**关键设计特点**：
- domain层纯业务，可独立测试
- repository层按域拆为6-8个子模块
- 统一错误处理映射20+种领域错误到HTTP状态码
- PBKDF2密码 + sha256 token + MySQL auth_sessions（12小时TTL）
- 资料清单版本管理（v20260625 64项 / v20260629 71项）

### 2.3 前端架构（digital-platform-web）

| 特性 | 说明 |

## 3. 数据模型与核心业务流程

### 3.1 数据库核心表（17张）

**项目管理域**：
- `projects` — 项目主数据（编号/名称/客户/经理/模式/状态）
- `project_stages` — 8阶段状态（stage_key/order/status/is_current）
- `stage_document_templates` — 资料模板（71项，版本化）
- `project_stage_documents` — 项目级资料清单（状态/责任人/追溯/返工）
- `project_stage_document_attachments` — 资料附件
- `project_initiation_review_nodes` — 立项审批节点（商务/技术/总经理三节点）
- `project_stage_document_forms` — 在线表单（草稿/提交）
- `project_stage_document_generated_files` — 生成文件（xlsx/docx）
- `project_stage_document_form_images` — 表单图片
- `project_stage_approvals` — 阶段审批流转
- `business_operation_logs` — 业务操作日志

**用户认证域**：
- `users` — 用户（账号/姓名/部门/组织角色/平台管理员）
- `auth_sessions` — 认证会话

**日报周报域**：
- `daily_reports` / `daily_report_items` / `daily_report_plans` / `daily_report_attachments`
- `weekly_reports` / `weekly_report_summaries` / `weekly_report_plans` / `weekly_report_approval_history`
- `center_daily_report_schedules` / `report_weekly_rest_mode_anchors`

### 3.2 标准8阶段流程

| 序号 | 阶段 | 阶段key | 关键节点 |
|---|---|---|---|
| 1 | 项目立项 | `project_initiation` | 市场调研 → 立项审批 → 立项通知 |
| 2 | 方案设计 | `scheme_design` | 方案评审 |
| 3 | 合同签订 | `contract_signing` | 合同审核 |
| 4 | 详细设计 | `detailed_design` | 设计评审 |
| 5 | 生产制作 | `production_manufacturing` | 生产过程 |
| 6 | 预验收 | `pre_acceptance` | 预验收记录 |
| 7 | 终验收 | `final_acceptance` | 终验收记录 |
| 8 | 项目结题 | `project_completion` | 结题报告 |

### 3.3 立项阶段三节点审批（1.2 项目立项审批表）

```
商务评价(business_review) → 研发评价(technical_review) → 总经理审批(general_review)
      ↓ 打回                    ↓ 打回                    ↓ 打回
  阻断后续节点              阻断后续节点              触发1.1返工
```

### 3.4 资料状态流转模型

```
not_submitted → submitted → confirmed
                  ↓              ↑
               returned ←───────┘

completion_mode:
  submit_only:         提交/上传 → 自动完成
  approval_required:   提交 → 待审核 → 审核通过完成
  conditional_submit:  复用isApplicable表达触发
```

### 3.5 组织角色与权限

| 角色 | 代码 | 核心权限 |
|---|---|---|
| 总经理 | `general_manager` | 全局可见、审批立项/合同/结题、审阅中心负责人周报 |
| 系统管理员 | `system_admin` | 仅用户管理，不自动获得业务权限 |
| 总经理助理 | `general_manager_assistant` | 只读/汇总，不显示审批/推进入口 |
| 中心负责人 | `center_manager` | 本中心范围审批、确认、推进、责任人分配 |
| 员工 | `employee` | 资料责任人可填写，项目经理可推进 |
