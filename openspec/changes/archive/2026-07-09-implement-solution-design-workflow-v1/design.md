## Context

本设计基于领导更新的《智能制造项目（自研模式）管理流程图20260704.pdf》和《AI数字化管理系统思维导图20260706.pdf》，以及本轮确认的业务口径。当前系统已有 8 大阶段、阶段资料清单、立项在线表单、模板文件生成、附件上传、审批/返工、项目业务日志和项目工作区基础能力，但“方案设计阶段”仍缺少内部节点状态机、项目内角色分配、三段成本估算、报价/投标分支和财务文件保密规则。

本 change 已完成 OpenSpec 规划，当前按任务清单分轮实现。已完成方案设计内部节点、项目内角色、工作计划/产品功能框图/方案设计 8 个产出上传槽、项目方案分析在线表单、内部/客户方案评审在线表单、成本估算三段后端基础闭环、财务文件后端脱敏、本地下载权限和报价/投标分支后端闭环；当前第七轮实现项目方案分析表、C15 内部方案评审记录表、C16 客户方案评审记录表的 Excel 模板文件生成、状态记录和下载后端闭环。不包含前端、合同签订阶段业务、文件平台、PDF、Word、额外格式成品生成、旧数据迁移或 SQL dump 导入。后续实现必须仍保留现有 8 大阶段：立项、方案设计、合同签订、详细设计、生产制作、预验收、终验收、结题；“方案设计准备”到“报价/投标”均是“方案设计阶段”内的子节点，不是新的大阶段。

关键边界：

- 不改变 v20260629 / 71 项资料数量。
- 不接文件平台，不生成 PDF，不生成 Word 或额外格式成品文件；当前第七轮仅生成已确认在线表单对应的 Excel 模板文件。
- 不迁移旧项目，不导入 SQL dump，不覆盖历史数据。
- 报价单当前模板目录中没有对应模板，第一版按文件上传处理。
- `项目方案分析表-模板.xlsx` 和 `方案评审记录表-模板.xlsx` 存在，规划为在线表单并生成模板文件；C15 内部方案评审记录表和 C16 客户方案评审记录表必须保持两个独立产出/资料项，只复用同一个方案评审记录表模板。
- `方案设计工作计划表-模板.xlsx` 虽然存在，但第一版仍按文件上传处理。
- `成本估算表-模板.xlsx` 虽然存在，但第一版不拆在线表单，按三段文件上传处理。

## Goals / Non-Goals

**Goals:**

- 定义方案设计阶段内部 9 个子节点及其顺序：方案设计准备 -> 项目方案分析 -> 方案设计 -> 内部方案评审 -> 客户方案评审 -> 研发成本估算 -> 制造成本估算 -> 财务成本估算 -> 报价/投标 -> 合同签订阶段。
- 定义方案设计准备节点的项目内角色分配规则：研发中心负责人分配项目经理、技术负责人、商务负责人、采购负责人、财务会计、财务负责人；其中项目经理必须复用并更新项目现有项目经理用户关联。
- 定义每个子节点的产出、提交人、提交方式、审批人、通过/退回路径和重提口径。
- 定义项目方案分析表、C15 内部方案评审记录表和 C16 客户方案评审记录表的在线表单/模板文件生成方向。
- 定义 C17 成本估算表作为一个主资料项下的研发、制造、财务/运营三段文件协作，不新增资料项。
- 定义报价/投标作为一个节点的分支状态、上传槽、投标审批路径和报价未被客户接受后的退回或项目结束路径。
- 定义财务/运营成本估算文件的后端权限和前端脱敏展示要求。
- 定义工作台待办、业务日志、前端节点展示、验收规则和后续实现任务。

**Non-Goals:**

- 不把 9 个方案设计子节点升级为大阶段。
- 不新增主管机械工程师、主管电气工程师、主管软件工程师分配流程；第一版方案设计 8 个产出均由技术负责人提交。
- 不新增商务负责人、技术负责人、采购负责人、财务会计、财务负责人为全局组织角色；它们是项目内流程角色。
- 不把方案设计工作计划、报价单、成本估算表第一版做成在线表单。
- 不拆分或重算成本估算 Excel 的公式和多 sheet。
- 不把商务标、技术标新增为独立资料项；它们是投标书下的两个必填上传槽。
- 不做精确打回到单个产出；项目方案分析和评审退回均按节点整体重提口径处理。
- 不实现外部文件平台、PDF、旧项目迁移、SQL dump 导入、通知推送或本轮之外的报表能力。

## Decisions

### 1. 使用阶段内状态机建模，而不是新增大阶段

方案设计阶段继续归属于既有 8 大阶段导航。内部状态机记录 9 个子节点状态，节点状态至少包含：未开始、待提交、待审批、已退回、已通过、已跳过/不适用、已结束。项目满足进入合同签订阶段的后端门禁必须以商务负责人确认报价被客户接受或投标总经理审批通过为准；第六轮仅返回 `permissions.canAdvanceToContract=true` 并记录 `solution_design.ready_for_contract` 门禁日志，当前第七轮不改变该门禁口径，不直接实现合同签订阶段业务或真实阶段推进。

理由：保持现有阶段模型和 71 项资料口径稳定，避免把流程图中的内部节点误建成新的大阶段。

替代方案：新增更多阶段或复用泛化阶段审批流。该方案会改变 8 大阶段口径，并且无法表达报价/投标单节点分支和三段成本协作，故不采用。

### 2. 项目内角色独立于全局组织角色

研发中心负责人在“方案设计准备”节点统一分配项目内角色：项目经理、技术负责人、商务负责人、采购负责人、财务会计、财务负责人。项目经理必须复用项目现有项目经理用户关联，例如现有 `projectManagerUserId` / `projectManagerUser` 口径；如在该节点重新指定项目经理，系统应更新项目项目经理关联，并同步影响项目经理工作台、项目详情展示、方案设计工作计划待办和后续项目经理相关权限。系统不得长期维护一套与项目现有项目经理字段冲突的方案设计专用项目经理。历史项目缺少方案设计准备分配记录时，应以项目现有项目经理关联作为默认项目经理来源。后端以项目内角色和当前用户组织属性共同计算可操作动作，前端只消费后端返回的权限和待办。

理由：这些角色是项目流程责任人，不应被建模成新的全局组织体系；项目经理已有项目级字段，必须保持单一来源；同时可保留现有 `department`、`organization_role`、负责人身份和平台管理员模型。

替代方案：把商务负责人、技术负责人等加入用户全局角色。该方案会扩大组织模型变更范围，且与本轮“项目内流程角色”口径冲突，故不采用。

### 3. 在线表单只用于两个已确认模板产出

项目方案分析表、C15 方案评审记录表（内部方案评审）和 C16 方案评审记录表（客户方案评审）规划为在线表单，并复用立项阶段在线表单提交后生成模板文件的能力。内部方案评审和客户方案评审必须继续作为 C15、C16 两个独立产出/资料项存在；本 change 仅要求二者复用同一个 `方案评审记录表-模板.xlsx`，并通过节点上下文或评审类型区分生成文件、在线表单记录、审批历史和最新有效版本。系统不得将 C15/C16 合并为一个资料项，也不得因此减少 71 项资料数量。

方案设计工作计划表虽然有模板，但第一版按文件上传；成本估算表虽然有模板，但第一版不拆在线表单；报价单无模板，第一版按上传。

理由：优先覆盖领导确认的在线填写口径，避免在成本估算公式、多 sheet 和报价模板未确认时扩大表单建模复杂度。

替代方案：将所有 Excel 模板都做成在线表单。该方案会增加公式还原、版本兼容和模板字段确认成本，且与本轮确认的文件上传口径冲突。

### 4. 成本估算用一个主资料项下的三段文件协作

C17 成本估算表仍保持一个主产出，不新增研发、制造、财务三个资料项。系统在 C17 下建三段上传槽/版本：研发成本估算文件、制造成本估算文件、财务/运营成本估算文件。后一部门可以下载前一部门文件作为基础，但不得覆盖删除前一段文件。

审批路径为：研发段由研发中心负责人审批，制造段由制造中心负责人审批，财务段先由财务负责人审批，再由总经理审批。总经理审批不通过时返回研发成本估算节点，并重新走研发、制造、财务三段流程。

理由：符合“C17 仍为一个主产出”的资料清单口径，同时能表达三部门协作、历史版本和审批轨迹。

替代方案：新增三个成本估算资料项。该方案会改变 71 项资料数量，故不采用。

### 5. 财务/运营成本估算文件必须后端保密

财务/运营成本估算文件只允许总经理和运营中心授权处理人查看或下载。运营中心授权处理人至少包括本项目财务会计、财务负责人，以及后续规格明确授权的运营中心人员。研发中心负责人、技术负责人、项目经理、商务负责人、采购负责人、制造中心负责人、非授权运营中心人员、总经理助理、系统管理员和其他无关用户只能查看节点状态和审批结果，不能查看文件名、附件明细、预览地址或下载入口。

理由：财务文件属于敏感信息，前端隐藏不足以满足保密要求，后端节点详情、资料清单、附件列表、业务日志、预览和下载接口都必须统一执行保密过滤。

替代方案：前端隐藏文件入口但后端仍返回附件元数据。该方案会泄露文件名和附件明细，故不采用。

### 6. 报价/投标是一个节点内的分支

财务成本估算经总经理审批通过后进入“报价/投标”节点，初始状态为未选择。总经理选择报价流程或投标流程，并记录业务日志。报价流程由商务负责人上传报价单，并由商务负责人执行报价结果处理；商务负责人确认报价被客户接受后，系统将报价/投标节点置为已通过，返回 `permissions.canAdvanceToContract=true` 并记录 `solution_design.ready_for_contract` 门禁日志。若客户不同意报价，商务负责人应先线下与总经理讨论，再在系统中选择退回研发成本估算或项目结束。投标流程使用投标书主资料项下的商务标和技术标两个必填上传槽，均完成后提交总经理审批；审批通过后同样只开放进入合同签订阶段门禁，不直接实现合同签订阶段业务或真实阶段推进；不通过返回投标节点重提，不默认退回成本估算。

理由：报价/投标在领导口径中是同一决策节点的两条路径，不能在导航上拆成两个固定大节点。

替代方案：把报价和投标拆成两个阶段或两个资料项集合。该方案会增加阶段/资料项数量并影响现有流程导航，故不采用。

### 7. 待办由后端权限和状态派生

个人工作台待办必须由后端根据项目内角色、组织角色、节点状态和资料/审批状态生成或返回，前端不得硬猜可处理人。待办范围覆盖研发中心负责人、项目经理、技术负责人、采购负责人、制造中心负责人、财务会计、财务负责人、总经理、商务负责人。

理由：方案设计阶段权限横跨项目内角色、部门负责人、总经理和财务保密规则，仅靠前端判断容易越权或漏待办。

替代方案：前端根据用户名称/部门硬编码入口。该方案不可审计且无法保证安全，故不采用。

### 8. 专用节点状态机是方案设计资料完成的唯一业务真相

纳入方案设计内部状态机管理的资料项和上传槽，其完成状态必须由专用节点状态机派生；普通阶段资料提交、确认或退回接口不得绕过或替代项目方案分析节点审批、内部方案评审审批、客户方案评审审批、研发成本估算审批、制造成本估算审批、财务负责人审批、总经理财务成本估算审批、商务负责人报价结果处理或投标审批。前端不得为这些专用流程对象展示普通资料审核入口；后端对直接调用普通审核接口的请求必须拒绝或返回专用流程错误。

理由：避免出现“普通资料确认已通过，但节点仍未通过”或“节点已通过，但普通资料未完成导致阶段阻塞”的双状态冲突。阶段推进门禁和工作台待办必须以专用节点状态为准，不得同时生成普通资料审核待办和专用节点审批待办。

替代方案：保留普通资料审核和节点审批两套状态并行。该方案会形成双重真相，无法稳定解释齐套、推进和待办结果，故不采用。

## Risks / Trade-offs

- [Risk] 内部节点、资料项、上传槽和审批历史之间关系复杂，可能与现有阶段资料接口耦合过重 -> [Mitigation] 后续实现先建立阶段内状态机和上传槽模型，再逐步接入项目工作区展示。
- [Risk] 方案评审记录支持多版本后，最新有效版本和历史版本展示口径可能混淆 -> [Mitigation] 后端明确 latest/current version 与历史列表，前端第一版展示最新有效版本并提供历史入口。
- [Risk] 财务文件保密若只在附件列表过滤，下载接口仍可能越权 -> [Mitigation] 所有附件元数据、预览、下载、生成文件入口均复用同一后端权限检查。
- [Risk] 报价未被客户接受后可项目结束会阻止后续阶段，可能影响历史待办 -> [Mitigation] 项目 `ended` 状态统一阻止合同签订及后续阶段操作，历史记录仅审计展示。
- [Risk] 成本估算三段文件都归属 C17，普通资料齐套统计可能只看主资料状态 -> [Mitigation] C17 完成状态必须由三段审批最终结果派生，总经理最终通过前不得视为完成。
- [Risk] 普通阶段资料审核入口可能绕过方案设计专用节点审批 -> [Mitigation] 纳入专用状态机的资料和上传槽必须屏蔽或拒绝普通审核入口，完成状态、阶段推进和待办均以专用节点状态为准。
- [Risk] 报价单模板缺失导致后续实现无法生成模板文件 -> [Mitigation] 第一版明确报价单为文件上传，不承诺在线表单或生成文件。
