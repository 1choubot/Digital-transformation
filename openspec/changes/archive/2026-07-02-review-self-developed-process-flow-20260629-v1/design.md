## Context

20260629 版自研模式流程图是当前 20260625 64 项资料模板之后的新输入。当前系统已经完成项目总览入口、项目工作区框架、立项阶段 `1.1 / 1.2 / 1.3` 在线表单、`1.2` 专用评价/审批、精准返工和 completionMode 阶段推进。新版流程图不能直接当作代码修改清单，否则会把流程备注、蓝色节点、资料模板、审批规则和项目模式变化混在一起。

## Goals / Non-Goals

**Goals:**

- 固化 20260629 流程图影响分析结果。
- 固化流程图变化分类机制。
- 明确当前系统仍以 20260625 64 项资料模板作为运行基线。
- 明确后续阶段蓝色节点和产出映射必须逐阶段实施。
- 为后续方案设计、合同签订、详细设计、生产制作、验收和结题阶段迁移提供依据。

**Non-Goals:**

- 不实现后 7 阶段蓝色节点。
- 不新增或修改 64 项资料模板。
- 不改变 completionMode、审批/返工规则或阶段推进状态机。
- 不新增在线表单。
- 不改 API、前端代码、数据库或 migration。
- 不处理文件平台联动、日报/周报、通知推送、账号管理或项目模式补录。

## Decisions

### Decision 1: 20260629 只作为迁移输入，不作为当前运行基线

当前运行基线仍是 20260625 64 项资料模板。20260629 中出现的疑似新增文件、阶段移动或 completionMode 变化，必须先进入独立 change 做业务确认、规格调整、测试设计和迁移评估。

Alternative considered: 直接用 20260629 替换当前模板。该方案会影响项目初始化、阶段推进、工作台、责任人、返工和归档触发边界，风险过高。

### Decision 2: 流程图变化必须先分类

后续任何流程图更新必须先归类为：

- 文案/备注变化：只影响说明、责任人描述或 checklist 备注。
- 资料模板变化：新增、删除、改名、阶段移动、必填性变化或 completionMode 变化。
- 蓝色节点变化：只影响项目工作区阶段内业务语境和产出分组。
- 审批/返工规则变化：影响 YES/NO、责任人、审核人、返工目标或状态机。
- 在线表单变化：影响表单字段、保存/提交入口、表单权限或生成文件。
- 项目模式变化：影响自研/外采/其他项目模式分支、适用模板或阶段路径。

分类完成前，不允许直接改代码。

### Decision 3: 后 7 阶段逐阶段迁移

第一批优先做方案设计阶段，因为该阶段与当前 20260625 模板覆盖度高，适合验证后续蓝色节点映射方法。后续依次处理合同签订、详细设计、生产制作、预验收/终验收/结题。

Alternative considered: 一次性补齐所有阶段节点。该方案会同时引入模板差异、审批差异和前端工作区差异，难以验证和回滚。

### Decision 4: 保留旧资料清单辅助区

即使后续逐阶段补齐蓝色节点，旧资料清单仍必须作为辅助兼容区保留，直到对应阶段的节点、产出、权限、阻塞原因和验收路径全部稳定。

### Decision 5: 准备/签订节点按草稿和成品拆分

20260629 PDF 图面红色产出数为 67；按业务修正后的逻辑产出候选数为 71。差异来自 4 处流程图省略：准备节点和签订节点共用一个成品红色产出，实际业务应拆为草稿产出和成品产出。

需要拆分的 4 处为：

- 合同签订阶段：准备技术协议 -> 技术协议草稿；签订技术协议 -> 技术协议。
- 合同签订阶段：准备销售合同 -> 销售合同草稿；签订销售合同 -> 销售合同。
- 生产制作阶段：准备技术协议 -> 技术协议草稿；签订技术协议 -> 技术协议。
- 生产制作阶段：准备采购合同 -> 采购合同草稿；签订采购合同 -> 采购合同。

后续阶段迁移时不应长期支持这些伪多对一。建模顺序应为“蓝色节点 -> 独立产出/草稿产出 -> 成品产出”。`成本估算表` 是当前确认的例外，可按研发、制造、运营和总经理多人/多节点协作同一表处理。

后续模板变更时必须按阶段区分同名 `技术协议草稿` / `技术协议`，不能仅按中文名去重；生产制作阶段供应商侧 `技术协议` 成品也必须作为独立候选显式评审。

## Risks / Trade-offs

- [Risk] 图上 `项目立项审批` NO 语义可能与当前 `1.1` 返工和 `1.2` 重填实现不一致。 → Mitigation: 标记为 P0 业务确认项，本 change 不修改立项状态机。
- [Risk] 疑似新增资料直接进入模板会破坏 64 项统计和阶段推进。 → Mitigation: 投标书、4 个草稿产出候选、合格供应商评价表、生产记录表、资料移交清单必须由后续独立 change 确认。
- [Risk] 把准备/签订共用成品产出误判为真实多对一，会让草稿审核和成品签订无法区分。 → Mitigation: 除 `成本估算表` 外，不长期支持伪多对一；后续按草稿和成品拆分建模。
- [Risk] 多角色合同/采购审核容易被误做成通用流程引擎。 → Mitigation: 后续 change 先判断是否仍由单一 approval_required 资料承接，不默认新增 BPM。
- [Risk] 弱化旧清单可能影响其他 7 阶段临时操作。 → Mitigation: 后续迁移必须保留旧资料清单辅助入口。
- [Risk] 付款、发票、未收款启动/发货批准容易扩展出财务流程。 → Mitigation: 当前只记录为 P1 风险，不在本 change 实现付款流或发票审批流。

## Migration Plan

本 change 不执行系统迁移。后续迁移建议按阶段拆分：

1. 方案设计阶段蓝色节点和产出映射。
2. 合同签订阶段蓝色节点和产出映射。
3. 详细设计阶段蓝色节点和产出映射。
4. 生产制作阶段蓝色节点和产出映射。
5. 预验收、终验收、结题阶段蓝色节点和产出映射。

每个实施 change 都必须先确认是否仅为 P2 前端节点映射；如涉及资料模板、completionMode、审批/返工或在线表单，必须同步补齐后端规格、smoke 和迁移策略。

## Open Questions

- `项目立项审批` NO 是否表示项目结束，还是仍按当前实现进入 `1.1` 返工和 `1.2` 重填？
- `投标书` 是否作为新增资料项进入方案设计阶段，还是并入报价或销售材料？
- 4 个草稿产出候选是否正式进入资料模板，具体 documentCode、阶段、必填性和 completionMode 如何定义？
- `技术协议（生产制作阶段/供应商侧成品）` 是否正式进入资料模板？若进入，编码、阶段、责任人、必填性、completionMode 如何定义？
- `合格供应商评价表`、`生产记录表`、`资料移交清单` 是否进入正式资料模板？
- 销售合同和采购合同的多角色审核是否仍由现有合同审核资料承接？
- `项目启动通知`、`发货通知` 是否形成资料项，还是只作为过程节点？
