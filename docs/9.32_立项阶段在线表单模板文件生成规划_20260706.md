# 9.32 立项阶段在线表单模板文件生成规划

## 背景

立项阶段已经完成 `1.1 / 1.2 / 1.3` 在线表单模板对齐、商务/技术协同填写、项目编号前置和现有评价审批主流程调整。当前填写入口已经是在线表单，但有权限人员查看已提交产出时仍主要查看在线表单内容，不是领导确认的“按真实模板产出文件”形态。

领导确认后的方向是：负责人在线填写，提交或审批完成后生成填好内容的模板文件；其他有权限人员默认查看生成后的产出文件。该能力需要先在立项阶段做通一个竖切，同时底层设计必须能复用到后续阶段资料模板生成，不能只为 `1.1 / 1.2 / 1.3` 写一次性硬编码。

## 目标

- 保留在线表单作为填写入口。
- 在线表单提交或审批完成后，生成对应真实模板文件。
- 有项目和资料查看权限的用户默认查看生成文件。
- 文件生成状态、版本、失败原因、下载元数据可追踪。
- 以立项阶段作为全局文件生成能力的首个竖切，沉淀可复用的模板渲染、文件记录、权限检查和产出卡片展示模型。

## 生成时机

- `1.1 项目需求表`：在线表单提交后生成 `.xlsx`。
- `1.2 项目立项审批表`：第一版建议在总经理最终通过后生成正式 `.xlsx`。如果后续需要草稿版或中间版，应作为扩展能力独立设计，不在本规划强制实现。
- `1.3 项目立项通知`：在线表单提交后生成 `.docx`。

退回、返工或重新填写后再次达到生成条件时，应生成新版本或使旧版本失效，不能静默覆盖旧文件导致审计断点。

## 架构分层

- 在线表单负责填写和结构化数据保存。
- 审批流负责资料状态、评价审批、退回和项目结束。
- 文件渲染器负责读取模板并填充结构化数据。
- 文件记录负责版本、状态、路径、文件名、下载元数据、失败原因和生成操作者。
- 产出卡片负责展示填写入口、生成状态和文件查看/下载入口。

该分层是为了让后续阶段复用同一套生成模型。立项阶段只作为首个竖切，不应把模板填充逻辑散落在前端或硬编码在某个页面组件里。

## 模板字段映射清单

后续实现前必须为 `1.1 / 1.2 / 1.3` 建立 mapping manifest，不能只靠字段名猜测填充模板，也不能再次扩展模板外字段来凑生成内容。mapping manifest 至少包含：

- `templateKey`
- `templatePath`
- `fileType`
- `outputDocumentCode`
- 目标位置：Excel 单元格、表格区域、Word 占位符或段落位置
- 来源字段：在线表单字段、项目主数据字段、审批意见字段、静态文本
- 必填字段
- 格式保留要求：合并单元格、边框、字体、行高列宽、图片、段落格式等
- 生成触发事件
- 失败时应记录的映射错误

mapping manifest 必须与真实模板字段一致。真实模板没有的字段不得被新增为生成字段或在线表单字段。

## 文件版本和状态

第一版建议保留版本，不覆盖旧文件；默认展示最新有效版本。文件状态至少需要区分：

- `generating`：生成中。
- `generated`：已生成，可按权限查看或下载。
- `failed`：生成失败，必须记录失败原因。
- `superseded`：已被后续版本替代。

生成失败不得假装成功，也不得把资料状态误判为文件已产出。后续实现需要明确重试入口、重试权限和业务日志。

## 源数据快照与哈希

历史生成文件必须能追溯到生成时的表单内容、审批意见和模板版本，不能只依赖会变化的当前 `formId` 来解释历史文件。后续文件记录应包含：

- `sourceFormSubmittedAt` 或 `sourceFormVersion`
- `sourceFormDataHash`
- `sourceSnapshotJson` 或等价不可变快照引用
- `triggerEvent`
- `reviewSnapshot`
- `templateVersion` 或 `templateHash`

`1.2 项目立项审批表` 的 `reviewSnapshot` 需要记录商务评价、技术评价、总经理审批意见、人员和时间。重新填写或退回后新生成文件必须关联新的源快照或源 hash。

## 生成失败与主流程关系

- 文件生成失败不得回滚在线表单提交或审批通过结果。
- 文件生成失败不得把资料状态假装为已生成。
- 第一版中，文件生成失败只影响“生成文件查看入口”，不自动撤销资料状态。
- 是否将“生成文件成功”作为阶段推进硬门禁，必须通过后续实现 change 明确；本规划暂不承诺生成失败会阻塞阶段推进。
- 产出卡片必须展示失败状态、失败摘要和后续重试入口。
- 重试必须受权限控制并写业务日志。

## 权限与查看入口

- 填写人仍通过在线表单入口填写和提交。
- 有资料查看权限的人默认在产出卡片看到生成文件入口。
- 无权限用户不得查看或下载生成文件。
- 产出卡片需要展示未生成、生成中、已生成、生成失败等状态。
- 不新增普通附件上传入口来替代在线表单或模板文件生成。

## 存储与数据库规划

规划阶段不写数据库 migration。后续实现如果需要文件元数据表，应先明确字段设计，至少包含：

- `projectId`
- `documentId`
- `formId`
- `templateKey`
- `version`
- `status`
- `fileName`
- `storagePath`
- `generatedBy`
- `generatedAt`
- `failureReason`
- `sourceFormSubmittedAt` 或 `sourceFormVersion`
- `sourceFormDataHash`
- `sourceSnapshotJson` 或等价不可变快照引用
- `triggerEvent`
- `reviewSnapshot`
- `templateVersion` 或 `templateHash`

第一版可规划为本系统内部存储和记录。文件平台联动、归档、同步目录、平台权限映射后续通过独立 change 处理。

## 模板路径治理

- 模板路径必须由后端模板注册表、配置或白名单统一管理。
- 业务接口不得接收任意模板路径参数。
- 代码中不得把绝对路径散落在业务逻辑中。
- 模板缺失、路径不可读、格式不支持时，文件记录必须进入 `failed` 或等价状态并记录原因。
- 不得因模板路径错误导致服务崩溃或泄露本地路径给无权限用户。

## 边界

- 本规划只做立项阶段模板文件生成能力规划，不实现业务代码。
- 本规划不生成 Excel / Word / PDF。
- 本规划不接文件平台，不处理 `file-platform-integration-v1`。
- 不新增资料项，不改变 v20260629 / 71 项资料数量。
- 不迁移旧项目，不补旧项目数据。
- 不删除在线表单填写入口。
- 不删除或隐藏兼容资料区。

## 风险

- 模板填充库选择会影响 Excel/Word 格式保真和部署复杂度。
- Excel 合并单元格、公式、样式、图片和 Word 段落格式需要专项验证。
- 文件版本保留会带来存储增长，需要清理或归档策略。
- 与 `file-platform-integration-v1` 必须保持边界，避免本 change 直接耦合文件平台目录和权限。
- 旧项目和旧表单数据不在本规划中迁移，旧数据是否能生成文件需要后续独立评估。
