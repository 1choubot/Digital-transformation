## Why

当前项目级阶段资料清单已经支持手工提交、确认、退回和阶段必填资料齐套摘要，但齐套摘要默认把所有必填资料都纳入口径。真实项目会因为合同范围、项目类型或交付内容差异导致部分默认必填资料不需要提交，因此需要增加项目级资料项“适用 / 不适用”的手工标记能力，避免把业务上不需要的资料强制计入齐套缺口。

## What Changes

- 为项目级阶段资料项增加独立适用性字段，默认适用，可手工标记为不适用，也可恢复适用。
- 标记不适用时必须填写非空原因；恢复适用时清空当前不适用原因。
- 保存最小追溯字段，用于追踪谁在什么时候标记不适用、谁在什么时候恢复适用。
- 后端新增资料项适用性操作接口：标记不适用、恢复适用。
- 适用性操作必须要求登录态，只做 `requireAuth`，不做复杂权限、角色权限或轻角色校验。
- 后端统一校验资料项属于当前项目、标记不适用原因非空，并阻止已不适用资料项继续提交、确认或退回。
- 阶段资料清单查询返回资料项适用性字段、不适用原因和适用性追溯字段。
- 阶段资料齐套摘要调整为只统计“适用且必填”的资料项；不适用资料项不计入 `requiredTotal`、`confirmedRequiredCount`、`incompleteRequiredCount`，也不出现在 `incompleteRequiredDocuments`。
- 前端项目详情页展示资料项适用性、不适用原因和追溯信息，并提供“标记不适用”和“恢复适用”操作。
- 页面文案必须明确：不适用是人工业务判断，用于说明该项目不需要该资料，不代表资料已提交、已确认或已归档。
- 本变更不实现项目类型模板、自动批量标记不适用、阶段推进、文件上传/下载、文件管理平台联动、在线表单、业务日志、责任人分配、个人待办、管理层看板或复杂权限。

## Capabilities

### New Capabilities

- 无。

### Modified Capabilities

- `stage-document-checklist`: 扩展项目级阶段资料项适用性模型、适用性操作接口、查询返回字段、状态操作校验和阶段齐套摘要统计口径。
- `project-core-frontend`: 扩展项目详情页阶段资料清单，展示资料项适用性和追溯信息，并提供手工标记不适用、恢复适用操作。

## Impact

- 后端影响 `digital-platform-api` 的项目阶段资料清单查询、资料项状态操作校验、资料项适用性操作路由、资料项仓储和数据库迁移。
- 前端影响 `digital-platform-web` 的项目详情页、项目 API 客户端、资料清单刷新和操作错误提示。
- 需要为 `project_stage_documents` 增加适用性、原因和最小追溯字段，但不引入新状态枚举，不改变现有 `not_submitted`、`submitted`、`confirmed`、`returned` 状态流转。
- 不引入文件平台 API、在线表单、阶段推进、业务日志、项目类型模板、自动批量规则或权限体系变更。
