## Context

数字化管理平台已经具备项目主数据、8 阶段初始化、项目详情基础状态和基础登录能力。`docs/9.1_8阶段流程与阶段定义表.md` 定义了第一版 8 阶段主线，`docs/9.2_阶段资料清单与责任角色表.md` 定义了每个阶段的资料项、必需口径、提交方式、默认责任角色、确认角色和文件平台目标目录。

后续在线表单、文件上传、资料齐套率和阶段推进都需要依赖项目级资料项清单。本变更先实现“模板 + 项目级清单 + 只读查询展示”，并严格避免进入提交、上传、确认、退回、齐套率和阶段推进范围。

## Goals / Non-Goals

**Goals:**

- 设计阶段资料项模板，字段覆盖阶段、资料项名称、是否必填、默认责任部门或责任角色、提交方式、文件平台目标文件夹路径，以及第一版为空的 `targetFolderId`。
- 资料项模板以 `docs/9.1` 的阶段标识和 `docs/9.2` 的资料清单为第一版口径。
- 项目创建成功后初始化该项目的阶段资料清单。
- 已有历史项目可以通过后端脚本或命令补初始化资料清单，且补初始化应避免重复生成。
- 新增查询某项目阶段资料清单的接口，并按阶段分组返回资料项。
- 前端项目详情页增加“阶段资料清单”区域，只展示资料项基础信息和状态。
- 资料项状态第一版使用系统枚举 `not_submitted`、`submitted`、`confirmed`、`returned`，页面分别展示“待提交”、“已提交”、“已确认”、“已退回”。
- 为后续文件上传、在线表单、资料齐套率和阶段推进预留必要字段。

**Non-Goals:**

- 不实现文件上传。
- 不实现文件下载。
- 不实现文件管理平台真实联动。
- 不实现在线表单填写。
- 不实现表单生成归档文件。
- 不实现资料齐套率计算。
- 不实现阶段推进。
- 不实现退回或确认操作。
- 不实现复杂权限。
- 不实现业务日志。
- 不做前端补初始化按钮。
- 不开放给普通用户的补初始化接口。

## Decisions

### 决策一：模板与项目实例分表

后端使用阶段资料项模板表保存第一版标准清单，再使用项目级阶段资料项表保存每个项目的资料清单实例。模板表用于初始化和后续模板版本追溯，项目级表用于保存状态、责任字段、目标目录映射和后续预留字段。

不选择只在代码中写死清单，因为历史项目补初始化、模板版本追溯和后续字段扩展都需要数据库中的结构化数据。

### 决策二：模板来源固定为 docs/9.1 和 docs/9.2 第一版口径

阶段标识必须与现有 8 阶段保持一致：`initiation`、`solution`、`contract`、`detailedDesign`、`manufacturing`、`preAcceptance`、`finalAcceptance`、`closeout`。资料项编号、名称、必填口径、提交方式、默认责任角色、确认角色和目标目录必须完全来自 `docs/9.2_阶段资料清单与责任角色表.md`。

不在本变更中重新解释或补充资料项，避免凭空扩展业务清单。如果实现时无法可靠解析 `docs/9.2`，必须暂停并说明原因，不得自行编造资料项。

### 决策三：提交方式使用稳定枚举

提交方式在系统中使用稳定枚举保存：

- `online_form`：在线表单
- `file_upload`：文件上传
- `mixed`：混合
- `tbd`：暂未确定

`docs/9.2` 中“在线表单+文件上传”映射为 `mixed`。后续若模板确认表进一步细化，只修改模板数据，不改变项目级清单结构。

### 决策四：资料项状态使用系统枚举，页面展示中文口径

项目级资料项状态第一版使用稳定枚举保存：

- `not_submitted`：页面显示“待提交”
- `submitted`：已提交
- `confirmed`：已确认
- `returned`：已退回

初始化状态必须为 `not_submitted`，前端展示为“待提交”，与 `docs/9.2` 的资料状态建议保持一致。本变更不提供状态流转接口，不实现确认、退回、提交、上传或归档动作。

### 决策五：项目创建后同步初始化资料清单

项目创建事务或等价流程必须在项目主数据、8 阶段记录创建成功后，为该项目初始化阶段资料清单。资料清单初始化失败时，项目创建不能假装完整成功。

不选择由前端创建后再单独初始化，因为资料清单是项目基础结构，应该由后端统一保证一致性。

### 决策六：历史项目补初始化通过后端脚本或命令执行

已有历史项目允许缺少资料清单。第一版补初始化通过后端脚本或命令执行，按项目检查缺失资料项后补齐，已经存在的资料项不得重复生成。第一版不做前端补初始化按钮，也不开放给普通用户调用的补初始化接口。

不选择启动时自动给所有历史项目补齐，也不选择开放普通用户接口，因为批量变更需要可控执行和明确验证。

### 决策七：查询接口默认按阶段分组返回

新增接口建议为 `GET /api/projects/:projectId/stage-document-checklist`，返回项目资料清单并按 8 阶段顺序分组。每个分组包含阶段基础信息和该阶段资料项列表。

不在本变更中返回齐套率、缺失统计或阶段推进判断，因为这些属于后续能力。

### 决策八：文件平台字段只保存目标映射，不做真实联动

第一版只保存 `targetFolderPath`。`targetFolderId` 字段可以预留但必须为空，待文件平台真实联动时再回填目录 ID。本变更不调用文件平台 API，不创建目录，不上传/下载文件，不判断文件权限。

### 决策九：项目详情页只读展示

项目详情页“阶段资料清单”区域只读展示资料项基础信息和状态，不提供上传、填写、提交、确认、退回、补初始化、阶段推进按钮。

不选择在详情页放置占位操作按钮，因为这会提前暴露未实现流程，容易让用户误解第一版范围。

## Risks / Trade-offs

- [模板口径后续变化导致项目实例不一致] -> 保存模板版本或模板来源字段，项目级实例保留初始化时的模板字段快照。
- [历史项目补初始化重复生成资料项] -> 使用项目、阶段、资料编号或资料 key 的唯一约束保障幂等。
- [前端只读清单被误解为可提交] -> 页面只展示资料项基础信息和状态，不提供上传、填写、确认、退回或推进按钮。
- [文件平台路径被误用为真实联动] -> 仅保存目标路径或目录标识，不调用文件管理平台 API。
- [实现时无法可靠解析资料清单文档] -> 暂停实现并说明解析问题，不自行编造模板资料项。

## Migration Plan

- 新增阶段资料项模板表和项目级阶段资料项表。
- 按 `docs/9.2` 初始化第一版模板数据。
- 修改项目创建流程，创建项目后初始化该项目资料清单。
- 增加历史项目补初始化脚本或命令。
- 增加项目阶段资料清单查询接口。
- 前端项目详情页新增“阶段资料清单”只读区域，不提供上传、填写、提交、确认、退回、补初始化或阶段推进按钮。
- 验证项目创建自动初始化、历史项目补初始化幂等、按阶段分组查询、详情页展示、以及未引入排除能力。

## Resolved Questions

- 第一版 `targetFolderId` 为空，只保存 `targetFolderPath`；等文件平台真实联动时再回填目录 ID。
