## 背景

当前数字化管理平台已经具备项目创建、项目列表、项目详情、标准 8 阶段初始化、项目级阶段资料清单、资料项手工状态流转、资料项适用性和阶段适用必填资料齐套摘要。项目详情页可以看到每个阶段资料齐套情况，但阶段本身仍只能停留在初始化后的只读状态。

本变更需要实现阶段推进的最小闭环：已登录用户只能手工推进当前阶段，后端根据当前阶段顺序自动推进到下一阶段，并在推进前用当前阶段适用必填资料齐套摘要作为门禁。该门禁仍基于当前手工状态和人工适用性判断，不代表文件已上传、文件已归档或在线表单已提交。

## 目标 / 非目标

**目标：**

- 新增 `POST /api/projects/:projectId/stages/advance` 接口，只推进当前阶段。
- 阶段推进接口要求登录态，第一版只做 `requireAuth`。
- 服务端校验项目存在、项目 `status` 未完成、当前阶段唯一、当前阶段状态为 `current`，并按 1 -> 2 -> 3 -> ... -> 8 顺序推进。
- 推进前只检查当前阶段适用必填资料齐套摘要；当前阶段项目级阶段资料清单必须已初始化，`incompleteRequiredCount = 0` 时允许推进，已有资料项记录但 `requiredTotal = 0` 时视为齐套。
- 缺失适用必填资料时拒绝推进，并返回缺失资料列表，最少包含 `id`、`documentCode`、`documentName`、`status`。
- 推进成功时在事务中更新当前阶段、下一阶段和第 8 阶段完成项目状态。
- 前端项目详情页展示“手工推进阶段”入口、缺失适用必填资料提示和推进成功后的刷新行为。
- 更新 `digital-platform-api/README.md` 和 `digital-platform-web/README.md`，说明阶段推进接口、齐套门禁口径、项目完成口径和能力边界。

**非目标：**

- 不支持阶段回退、跳阶段、批量推进或自由指定目标阶段。
- 不实现阶段推进审批流、通知、个人待办、责任人分配或管理层看板。
- 不记录阶段推进操作人完整日志，不新增业务日志。
- 不实现项目经理角色校验、复杂权限、角色权限或轻角色校验。
- 不实现文件上传/下载、文件管理平台联动、在线表单、表单草稿或表单归档文件生成。
- 不修改、完成或归档 `define-digital-platform-v1`。

## 设计决策

### 决策一：使用显式动作接口推进当前阶段

新增接口建议为：

- `POST /api/projects/:projectId/stages/advance`

请求体不接收目标阶段、目标阶段顺序或目标阶段标识。后端根据项目当前阶段的 `stage_order` 自动定位下一阶段，并只允许当前阶段推进到下一阶段。

备选方案是提供 `PATCH /api/projects/:projectId/stages/:stageId` 或接收 `targetStageId`。第一版不采用该方案，因为自由指定目标会引入跳阶段、回退和前端篡改目标的风险，也会扩大校验面。

### 决策二：阶段推进状态机由后端统一校验

推进前后端必须校验：

- 项目存在。
- 只要项目 `status = completed`，必须拒绝推进；即使异常数据中仍存在当前阶段，也不得继续推进。
- 项目必须有且只有一个 `is_current = 1` 的阶段。
- 当前阶段 `stage_status` 必须是 `current`。
- 当前阶段必须属于项目标准 8 阶段之一。
- 如果当前阶段不是第 8 阶段，则下一阶段必须存在、顺序为 `current.stage_order + 1`、`stage_status` 必须是 `not_started`，并且 `is_current` 必须是 `0`。
- 不允许跳阶段、回退、批量推进或由前端指定目标阶段。

如果项目已完成、没有当前阶段、存在多个当前阶段、当前阶段状态异常、下一阶段缺失、下一阶段顺序异常、下一阶段不是 `not_started` 或下一阶段已被标记为当前阶段，接口必须拒绝推进，并不得修改任何阶段或项目状态。

### 决策三：齐套门禁复用当前阶段适用必填资料口径

阶段推进只检查当前阶段，不检查历史阶段或未来阶段。门禁口径与阶段资料清单 `completenessSummary` 保持一致：

- 当前阶段必须已经存在项目级阶段资料项记录；如果当前阶段没有任何 `project_stage_documents` 资料行，视为阶段资料清单尚未初始化，必须拒绝推进，并且不得修改项目状态或任何阶段状态。
- 只统计 `is_required = true` 且适用的项目级资料项。
- `confirmed` 视为完成。
- `not_submitted`、`submitted`、`returned` 视为未完成。
- 建议资料项和不适用资料项不计入门禁。
- 只有在当前阶段资料项记录存在、但适用必填资料数为 0 时，`requiredTotal = 0` 才视为当前阶段齐套并允许推进。
- 只有 `incompleteRequiredCount = 0` 时允许推进。

实现上可以复用已有齐套摘要计算函数，也可以在阶段推进事务中执行等价查询。无论采用哪种方式，返回给前端的拒绝结果必须包含缺失适用必填资料列表，且每项至少包含 `id`、`documentCode`、`documentName` 和 `status`。

### 决策四：阶段和项目状态更新必须在事务中完成

推进成功时必须在同一数据库事务中完成：

- 当前阶段 `stage_status` 更新为 `completed`。
- 当前阶段 `is_current` 更新为 `0`。
- 当前阶段记录 `completed_at`。
- 如果存在下一阶段，下一阶段 `stage_status` 更新为 `current`。
- 如果存在下一阶段，下一阶段 `is_current` 更新为 `1`。
- 如果存在下一阶段，下一阶段记录 `started_at`。

如果当前阶段是第 8 阶段 `closeout`，则只完成当前阶段，并将项目 `status` 更新为 `completed`，项目不再有当前阶段。

事务中应尽量锁定项目和项目阶段记录，避免并发推进导致两个阶段同时为当前阶段或重复完成当前阶段。第一版不新增阶段推进日志表。

### 决策五：项目完成后允许当前阶段为空

项目第 8 阶段推进成功后，项目 `status = completed` 且没有 `is_current = 1` 的阶段。项目列表和项目详情需要允许当前阶段为空，并以“已完成”或等价状态展示。

阶段推进接口必须把 `status = completed` 作为终态拦截条件。只要项目已经完成，无论数据中是否异常残留当前阶段，都必须拒绝推进，并不得修改项目状态或任何阶段状态。非完成项目仍必须保持有且只有一个当前阶段；如果非完成项目没有当前阶段或存在多个当前阶段，阶段推进接口应拒绝并返回可读错误。

### 决策六：前端只提供当前阶段手工推进入口

项目详情页只在当前阶段上下文中展示“手工推进阶段”入口，不提供选择目标阶段、跳阶段、回退或批量推进入口。页面应展示：

- 当前阶段名称和状态。
- 当前阶段适用必填资料齐套摘要。
- 当前阶段缺失适用必填资料列表。
- “手工推进阶段”说明，明确不代表文件已上传或已归档。

当当前阶段未齐套时，页面应提示不能推进。推进失败时展示后端返回的缺失适用必填资料。推进成功后必须刷新项目详情、8 阶段状态和阶段资料清单，使齐套摘要与当前阶段显示同步更新。

## 风险 / 取舍

- [并发重复推进] -> 在事务中读取并更新项目阶段，必要时锁定项目和阶段行，失败时整体回滚。
- [用户误解齐套为文件已归档] -> 前端和 README 明确齐套门禁基于当前手工状态与人工适用性判断，不代表文件上传或归档。
- [没有角色权限导致任意已登录用户可推进] -> 第一版明确只做 `requireAuth`，角色权限留给后续独立变更。
- [项目完成后当前阶段为空影响既有展示] -> 项目列表和详情明确允许 `completed` 项目当前阶段为空，并展示完成状态。
- [齐套计算逻辑重复] -> 优先复用已有 `completenessSummary` 计算逻辑；如无法直接复用，推进接口的等价查询必须覆盖相同测试口径。
- [不记录完整操作日志] -> 本变更只记录阶段 `started_at`、`completed_at` 和项目状态，不实现业务日志或推进操作人完整日志。

## 迁移计划

1. 确认 `project_stages` 已包含 `started_at` 和 `completed_at` 字段；如目标环境缺失，则新增迁移补齐。
2. 实现阶段推进接口和事务更新逻辑，不改变既有项目创建和阶段初始化顺序。
3. 更新前端项目详情页，在当前阶段区域加入手工推进入口和缺失适用必填资料提示。
4. 更新 API/Web README，说明阶段推进、齐套门禁、完成项目口径和明确不做范围。
5. 回滚时移除阶段推进路由和前端入口即可；已有阶段表字段可保留，已完成的阶段状态需要按业务数据恢复方案另行处理。

## 待确认问题

当前第一版没有待确认问题。项目第 8 阶段推进成功后的项目状态采用现有 `completed`，当前阶段为空。
