# 14. 业务操作后的数据刷新与失效规则

## 规则目的

节点页面动态化后，各节点会拥有自己的局部业务状态。

例如：

-   在线表单；
-   附件列表；
-   审批节点；
-   责任人选择；
-   生成文件状态；
-   当前资料权限。

因此，禁止节点页面在业务操作完成后继续依赖旧的本地状态。

所有提交、保存、审批、退回、完成整改、责任人变更、附件变更等操作完成后，必须重新同步后端权威数据。

------------------------------------------------------------------------

## 基础刷新规则

任何业务操作成功后，必须刷新项目级权威数据：

    workspace
    navigation
    checklist

说明：

-   `workspace` 用于刷新节点、产出、阻塞原因；
-   `navigation` 用于刷新左侧流程树状态；
-   `checklist` 用于刷新资料权限、责任人、状态、附件绑定等工作区数据源。

禁止：

-   只修改前端本地状态来模拟后端结果；
-   只刷新当前节点，忽略流程树；
-   只刷新流程树，忽略资料权限；
-   操作后继续使用旧的 `permissions`、`blockingReasons`、`status`。

------------------------------------------------------------------------

## 当前业务面板处理规则

业务操作成功后，当前业务面板必须二选一处理：

### 1. 当前面板已经失效：清空

适用场景：

-   审批退回；
-   流程跳转到其他节点；
-   当前资料不再是下一步处理对象；
-   当前表单状态已经不适合继续编辑。

要求：

    clear current business panel state

例如：

-   清空当前在线表单；
-   清空当前审批详情；
-   清空当前附件临时状态；
-   让用户重新进入目标节点后重新拉取数据。

------------------------------------------------------------------------

### 2. 当前面板仍然有效：重新拉取

适用场景：

-   保存草稿后继续停留当前表单；
-   提交后仍展示当前表单结果；
-   上传/删除附件后仍停留当前资料；
-   审批操作后仍展示当前审批详情。

要求：

    reload current business detail from API

例如：

-   当前在线表单仍展示时，必须重新请求在线表单接口；
-   当前附件面板仍展示时，必须重新请求附件列表；
-   当前审批面板仍展示时，必须重新请求审批节点详情。

禁止：

-   使用操作前的 `activeOnlineForm`；
-   使用操作前的附件列表；
-   使用操作前的审批节点权限；
-   只保留面板而不刷新其内部权限。

------------------------------------------------------------------------

## 跨节点影响规则

如果一个节点的操作会影响其他节点或其他资料，必须刷新受影响对象。

典型示例：

    1.2 项目立项审批表
        ↓ 总经理退回整改
    1.1 项目需求表进入返工

此时不能只刷新 `1.2`。

必须同步刷新：

-   `1.2 项目立项审批表`；
-   `1.1 项目需求表`；
-   `market_research` 节点；
-   `initiation_approval` 节点；
-   左侧流程树状态；
-   当前资料权限和阻塞原因。

------------------------------------------------------------------------

## 节点页面事件约定

节点页面完成业务操作后，不直接假设影响范围。

推荐向 `ProjectDetailLayout` 抛出统一事件：

``` javascript
emit('business-state-changed', {
  changedDocumentIds: [documentId],
  affectedNodeCodes: ['market_research', 'initiation_approval'],
  refreshCurrentDetail: true,
  clearCurrentDetail: false,
  refreshWorkspace: true,
  refreshNavigation: true,
  refreshChecklist: true
});
```

`ProjectDetailLayout` 负责：

-   刷新项目级状态；
-   决定是否保留当前节点页面；
-   通知当前节点页重新拉取业务详情；
-   必要时清空失效业务面板。

------------------------------------------------------------------------

## 默认判断顺序

业务操作成功后，按以下顺序执行：

    1. 刷新 workspace / navigation / checklist
    2. 判断当前业务面板是否仍有效
    3. 若无效，则清空当前业务面板
    4. 若有效，则重新拉取当前业务详情
    5. 若影响其他节点或资料，则刷新受影响节点和资料

该规则不是三选一。

要求：

    基础刷新必须执行
    清空或重新拉取二选一
    跨节点影响时额外刷新关联数据

------------------------------------------------------------------------

## 验收要求

新增或迁移任意节点页面时，必须验证：

-   保存后权限状态正确；
-   提交后状态正确；
-   审批通过后流程树和节点状态正确；
-   审批退回后返工目标节点可重新编辑；
-   退回影响其他节点时，相关节点不显示旧状态；
-   页面刷新后节点和业务状态可恢复；
-   不存在后端已允许、前端仍禁用的旧状态问题。
