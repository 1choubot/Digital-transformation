## 为什么

当前平台已有基础用户表和登录能力，但只有初始化账号脚本，没有可维护用户的后台入口。后续责任人分配、个人待办、项目成员和文件权限联动都依赖稳定的用户主数据，因此需要先补齐第一版最小用户管理能力。

## 变更内容

- 为 `users` 增加 `is_platform_admin` 字段，作为第一版最小平台管理员开关。
- 初始化账号默认必须保存为 `isPlatformAdmin = true`；可支持显式配置覆盖，但不得导致系统初始化后无人可管理。
- 平台管理员开关只保护用户管理 API 和用户管理页面入口，不扩展为项目权限、资料权限、阶段权限、文件权限或复杂 RBAC。
- 后端新增用户管理接口：用户列表、新增用户、编辑用户基础信息、启用/禁用用户、重置用户密码，并可按设计提供查询单个用户接口。
- 用户管理接口必须要求登录态，并要求当前用户 `is_platform_admin = true`；不得只使用 `requireAuth`。
- 用户管理接口不得信任前端传入当前操作人，不得返回 `password_hash` 或其他密码内部字段。
- 用户管理必须防止系统进入 `0` 个启用平台管理员状态；禁用用户、取消 `isPlatformAdmin` 或同时编辑 `isEnabled` / `isPlatformAdmin` 时必须校验该不变量。
- 新增用户时校验 `account` 必填且唯一，`displayName`、`department`、`role`、`password` 必填；`isEnabled` 默认 `true`，`isPlatformAdmin` 默认 `false`，`filePlatformUserId` 可为空。
- 编辑用户时允许修改 `displayName`、`department`、`role`、`isEnabled`、`isPlatformAdmin`、`filePlatformUserId`，不得通过普通编辑接口修改 `account` 或 `password`。
- 重置密码必须走单独接口。
- 禁用用户不能继续新登录；当前已登录用户被禁用后的会话处理沿用现有认证逻辑，不新增强制踢下线能力。
- 前端新增用户管理页面，仅平台管理员可进入或可操作；非管理员不展示入口，或访问时展示无权限提示。
- 用户管理页面展示用户列表、新增用户表单、编辑基础信息、启用/禁用、重置密码，并展示 `filePlatformUserId` 字段但不做文件平台联动校验。
- README 更新 API/Web 的用户管理接口、管理员边界、字段说明和不做范围。
- 本变更不实现项目成员管理、项目经理账号绑定、责任人分配、个人待办、复杂权限/角色权限矩阵、部门权限继承、审批流、文件平台用户同步、文件平台权限判断、登录审计、用户管理操作日志、单点登录或批量导入用户。

## 能力范围

### 新增能力

- `user-management`: 定义第一版基础用户管理能力，包括平台管理员边界、用户管理 API、字段规则、安全边界和不做范围。

### 修改能力

- `basic-users-auth`: 用户主数据增加 `isPlatformAdmin`，登录和当前用户识别返回该字段，初始化账号默认保存为平台管理员并支持升级兜底恢复。
- `project-core-frontend`: 前端增加用户管理页面和平台管理员入口边界，并调整现有前端登录态展示能力以支持 `isPlatformAdmin`。

## 影响

- 后端影响 `digital-platform-api` 的用户表迁移、初始化账号脚本、认证返回模型、用户仓储、用户管理路由、管理员校验中间件和 README。
- 前端影响 `digital-platform-web` 的当前用户模型、导航入口、用户管理页面、用户管理 API 客户端、错误提示和 README。
- 不引入文件平台 API、SSO、审计日志、项目权限矩阵、按钮级权限矩阵、责任人分配、个人待办或批量导入依赖。
