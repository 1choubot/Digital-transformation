# 9.33 立项审批新版模板与通知编号流程修正规划

## 背景

领导确认当前立项流程整体方向无问题，但用户已将 `智能制造项目管理文件模板` 文件夹内容替换为 `智能制造项目管理文件模板v1` 内容。项目需求表和项目立项通知模板名称不变，预计不影响 `1.1` 和 `1.3` 基础读取路径；新版项目立项审批表仍位于：

`D:\Digital transformation\智能制造项目管理文件模板\项目立项审批表-模板.xlsx`

本次规划需要修正两个口径：

- 新版 `1.2 项目立项审批表` 已取消项目号/项目编号字段，项目编号不应再由 `1.2` 填写或校验。
- 项目编号应由 `1.3 项目立项通知` 正式确定，`1.3` 提交时校验必填、唯一，并写入项目主数据。
- `1.3` 通知文件不应每次只写当前项目，而应累计展示已确定编号项目和当前项目。
- 当前 `1.3` 通知生成存在未使用空白表格行挤压落款日期的问题，需要在生成时删除空白行；项目数量超过模板第一页容量时克隆数据行继续追加，允许自然分页。

## 修改前流程

1. `1.2 项目立项审批表` 基础模块填写项目编号。
2. `1.2` 提交或进入评价审批前校验项目编号必填、唯一。
3. `1.2` 提交时将项目编号写入 `projects.project_code`。
4. `1.3 项目立项通知` 只读带出项目编号。
5. `1.3` 通知生成文件只写当前项目一行。

## 修改后流程

1. `1.2 项目立项审批表` 不填写项目编号。
2. `1.2` 只负责商务/技术评分协同填写和后续审批。
3. `1.3 项目立项通知` 填写项目编号。
4. `1.3` 提交事务中校验项目编号必填、唯一，并通过命名锁保护并发唯一性后写入 `projects.project_code`。
5. 立项流程项目不得通过独立 `project-code` 更新接口绕过 `1.3` 写入项目编号；项目存在 `1.3` 资料项时，该接口统一返回“项目编号由 1.3 管理”的业务错误。
6. `1.3` 生成项目编号通知时输出累计项目清单，而不是只有当前项目。

## 新版 1.2 模板字段分析

新版 `项目立项审批表-模板.xlsx` 的规划口径如下：

- 表头不再有项目号/项目编号。
- 表头包括项目名称、客户名称、客户项目联系人、客户项目联系电话、本公司商务负责人、本公司商务负责人联系电话。
- 商务模块评分项为 6 项：
  - 客户企业属性
  - 项目来源
  - 项目定位
  - 商务竞争条件
  - 项目预算
  - 付款条件
- 技术模块评分项为 5 项：
  - 项目需求
  - 特殊环境要求
  - 行业门槛
  - 技术成熟度
  - 研发模式
- 意见区包括营销中心意见、研发中心意见、总经理意见。
- 项目开展模式包括自研模式/供应链模式。领导已明确该字段必须填写；第一版作为 `1.2` 商务负责人填写的必填单选字段，并只用于生成新版 `1.2 项目立项审批表` 产出文件。该字段不关联系统项目模式，不写入 `projects.project_mode`，不影响项目阶段、项目状态、项目筛选或其他业务逻辑。

第一版实现按新版 workbook 坐标显式 mapping：

- 表头：`A2` 项目名称、`A3` 客户名称、`A4` 客户项目联系人、`G4` 客户联系电话、`J4` 本公司商务负责人、`M4` 本公司商务负责人联系电话。
- 商务评分：6-11 行，分值写 `K` 列，信息收集说明写 `L` 列，责任人写 `O` 列，不覆盖 `C`/`H` 固定条款和评价标准。
- 技术评分：12-16 行，分值写 `K` 列，信息收集说明写 `L` 列，责任人写 `O` 列。
- 意见区：营销/研发/总经理意见分别写 `A17`/`A18`/`A19`，日期写 `M17`/`M18`/`M19`，签字栏保留模板原文。
- 项目开展模式：根据 `1.2` 在线表单必填单选字段勾选 `D20` 自研模式或 `G20` 供应链模式；生成时必须保留模板富文本结构，只替换复选框符号 run，不得整格替换为 Unicode 复选框文本导致中文字体错乱。

## 1.3 通知文件生成规划

`1.3 项目立项通知` 生成时需要输出累计项目编号清单。实现阶段需要明确查询口径和排序规则，建议第一版规则如下：

- 数据来源：已提交 `1.3` 且项目编号非空的项目。
- 当前项目：本次 `1.3` 提交事务写入项目编号后，当前项目必须包含在清单中。
- 截断口径：累计清单只包含 `1.3.submitted_at <= 当前项目 1.3.submitted_at` 的项目。当前项目通知生成失败后重试，也必须继续使用当前项目提交时间作为 cutoff，不得把后续才提交的“未来项目”补进旧通知。
- 排序规则：按 `1.3` 提交时间升序；提交时间缺失时按项目 id 升序。
- 每行字段：序号、项目编号、项目名称、客户单位、立项日期。
- Word 表格：删除未使用空白行，避免落款日期被挤到第二页。
- 超出模板预置行数：克隆数据行追加，保留表格样式，允许 Word 自然分页。
- source snapshot/hash：包含累计清单、每行字段和 cutoff，用于解释历史通知生成时的输入。

## 边界

- 不接文件平台。
- 不生成 PDF。
- 不新增资料项。
- 不改变 v20260629 / 71 项资料数量。
- 不迁移旧项目历史数据。
- 不重做 `1.1 项目需求表`。
- 未提交 `1.3` 或项目编号为空的旧项目不进入累计清单。
- 已存在 `project_code` 的旧项目打开 `1.3` 时可以默认带出；提交时仍需要按唯一性规则校验。
- 项目编号唯一性实现必须具备并发防护，不能只依赖普通查询校验；实现阶段需采用数据库唯一索引、命名锁、串行化事务或等价机制。
- 所有仍可能写入 `projects.project_code` 的后端路径必须共用同一并发保护策略；第一版对存在 `1.3` 资料项的立项流程项目禁用独立更新接口，遗留无 `1.3` 资料项路径也必须复用项目编号命名锁。
