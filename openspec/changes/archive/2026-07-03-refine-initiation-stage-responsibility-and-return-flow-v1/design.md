## Context

v20260629 / 71 项资料模板已经作为新建项目默认模板运行。领导补充的立项阶段流程图不是要求新增模板项或接入文件平台，而是要求把立项阶段内部责任人、项目输入节点、评价拒绝退回和总经理拒绝去向说清楚。

## Goals / Non-Goals

**Goals:**

- 项目先创建，创建后进入项目详情和立项阶段。
- 创建项目时增加商务负责人、技术负责人。
- 商务负责人来自营销中心启用用户。
- 技术负责人来自研发中心启用用户。
- 项目输入节点展示项目基础信息和两个负责人。
- 1.1 / 1.2 默认由商务负责人填写。
- 商务评价拒绝、技术评价拒绝退回项目市场调研。
- 总经理拒绝支持退回项目市场调研或项目结束。
- 总经理拒绝选择项目结束时记录结束原因、操作者、操作时间和业务日志。
- 项目结束后抑制相关可处理待办，并在项目总览和项目详情展示已结束状态与原因摘要。
- 在线表单内容对有权限用户可浏览。

**Non-Goals:**

- 不推翻 v20260629 / 71 项模板。
- 不新增资料项，不改变 71 项模板数量。
- 不做真实自动生成 Word/PDF/文件平台文件。
- 不接文件平台，不生成附件。
- 不引入 BPM 或通用流程引擎。
- 不新增第二套资料状态机。
- 本规划阶段不改业务代码、不改数据库 schema、不写 migration。

## Decisions

### Decision 1: 项目先创建

系统 MUST 保持“先创建项目，再进入项目详情 / 立项阶段”的结构。创建项目时填写项目名称、客户名称、客户联系人和电话，并选择商务负责人、技术负责人。

这个设计保留当前项目创建入口，不把立项阶段前置成独立的无项目流程。

### Decision 2: 两个项目级责任人

创建项目时新增两个项目级责任人：

- 商务负责人：必须是营销中心启用用户。
- 技术负责人：必须是研发中心启用用户。

后续实现 MAY 通过新增项目字段、项目责任人绑定表或等价稳定结构保存，但必须能被项目详情、立项在线表单、审批权限和日志追踪复用。

### Decision 3: 自动生成文件按在线表单可浏览理解

流程图里的“自动生成项目需求表 / 项目立项审批表 / 立项通知”第一版 MUST NOT 理解为生成真实 Word、PDF、文件平台文件或普通附件。

第一版口径是：在线表单填写完成后，有权限浏览项目的人能查看在线表单内容。文件平台联动、导出和归档必须后续独立 change 处理。

### Decision 4: 1.1 / 1.2 / 1.3 主流程尽量保持

1.1 项目需求表、1.2 项目立项审批表、1.3 立项通知继续走现有在线表单主流程。1.2 专用评价审批继续作为立项审批的专用能力，不退回为普通上传、普通提交或普通审核。

### Decision 5: 评价拒绝退回市场调研

商务评价审批由营销中心负责人处理。通过时节点完成；拒绝时 MUST 退回项目市场调研。

技术评价审批由研发中心负责人处理。通过时节点完成；拒绝时 MUST 退回项目市场调研。

退回项目市场调研时 SHOULD 复用现有在线表单、精准返工、状态、权限和业务日志能力，不创建第二套评价状态机。

### Decision 6: 总经理拒绝有两个去向

总经理审批 MUST 在商务评价和技术评价都完成后才可处理。同意时进入立项通知；拒绝时必须支持：

- 退回项目市场调研。
- 项目结束。

项目结束后 MUST 阻止继续推进立项通知、方案设计和后续资料操作。项目结束的状态字段、权限和旧项目兼容策略在后续实现阶段明确。

总经理拒绝选择“项目结束”时，系统 MUST 记录结束原因、操作者、操作时间和业务日志。项目结束后，相关待办 MUST NOT 继续作为可处理待办展示；项目总览和项目详情 MUST 展示项目已结束状态，并至少展示原因摘要。

## Target Flow

```text
创建项目
  -> 填项目名称、客户名称、客户联系人和电话
  -> 选择商务负责人、技术负责人
  -> 进入项目详情 / 立项阶段

项目输入
  -> 展示项目基础信息和商务/技术负责人
  -> 完成后进入项目市场调研

项目市场调研
  -> 商务负责人填写 1.1 项目需求表在线表单
  -> 完成后进入项目立项审批

项目立项审批
  -> 商务负责人填写 1.2 项目立项审批表在线表单
  -> 完成后启动商务评价、技术评价

商务评价审批
  -> 营销中心负责人处理
  -> 通过：节点完成
  -> 拒绝：退回项目市场调研

技术评价审批
  -> 研发中心负责人处理
  -> 通过：节点完成
  -> 拒绝：退回项目市场调研

总经理审批
  -> 商务评价和技术评价都完成后才可处理
  -> 同意：进入立项通知
  -> 拒绝：可退回项目市场调研，或项目结束
  -> 选择项目结束：记录结束原因、操作者、操作时间和业务日志，抑制后续可处理待办

立项通知
  -> 营销中心负责人填写 1.3 立项通知在线表单
  -> 完成后进入方案设计阶段
```

## Project End State

项目结束是总经理拒绝后的明确终态，不是普通返工状态。后续实现 MUST 保证：

- 结束操作记录结束原因、操作者、操作时间和业务日志。
- 立项通知、方案设计、阶段推进和后续资料操作不可继续处理。
- 相关待办不再作为可处理待办出现在我的工作台或其他待办入口。
- 项目总览和项目详情展示项目已结束状态，并至少展示原因摘要。
- 历史在线表单、审批记录和业务日志仍可按权限浏览，用于审计。

## Open Questions For Implementation

- 商务负责人、技术负责人使用项目主表字段还是项目责任人绑定表保存。
- 旧项目缺少商务/技术负责人时的兼容显示、补录或非阻塞策略。
- 项目结束使用现有项目状态扩展还是新增稳定终态枚举。
- 项目结束原因摘要在项目总览和项目详情中的字段来源与展示长度。
- 总经理拒绝退回市场调研时，1.1 / 1.2 哪些字段需要精准返工。

这些问题必须在后续 implementation 阶段明确，不能在规划阶段改业务代码临时决定。
