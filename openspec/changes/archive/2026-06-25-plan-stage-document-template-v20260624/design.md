## Context

当前运行系统已经支持 `v20260610` 的 54 项阶段资料模板，并为模板和项目级资料快照维护 `ownerDepartment` / `reviewDepartment`。20260624 版流程图在 8 个阶段不变的前提下，补充了更细的阶段产出文件和业务节点。本 change 将当前 active 模板切换到 `v20260624` 的 64 项。

已有审查文档 `docs/9.9_阶段资料清单与责任归属表_20260624.md` 记录了 66 行规划项，其中人工数出的流程图直接产出文件为 64 个，另外 2 行是非红色文件节点：`7.P1 随机资料移交` 和 `8.P1 资料服务器核查`。本 change 只把 64 个直接产出文件写入当前普通阶段资料模板。

## Goals / Non-Goals

**Goals:**

- 从 `v20260610` 54 项升级到 `v20260624` 64 项阶段资料模板。
- 每个资料项只表达“阶段产出文件”，字段包括稳定 `documentCode`、阶段、文件名称、`ownerDepartment`、`reviewDepartment`、审核/确认对象备注、模板默认必填、适用条件、提交方式和备注。
- 保持现有阶段推进规则：当前阶段适用且必填资料全部 `confirmed`，且阶段审批通过后，才允许推进。
- 明确当前规划以 64 个直接产出文件为准，不采用 66 行规划表作为普通资料模板数量。
- 基于模拟数据性质重置旧项目资料，并重新初始化新版 64 项模板。

**Non-Goals:**

- 不执行 migration。
- 不新增独立实现 change。
- 不要求兼容旧模拟项目中的 `v20260610` 项目资料。
- 不把 `7.P1 随机资料移交` 和 `8.P1 资料服务器核查` 纳入普通阶段资料模板，除非后续业务确认它们形成独立文件。
- 不建模合同审核流程、付款/发票流程、采购流程、设计变更流程、资料服务器核查流程或多中心协同任务流。
- 不修改、完成或归档 `define-digital-platform-v1`。

## Decisions

### 64 项直接产出文件作为模板规划口径

`64` 表示 20260624 流程图中人工识别出的直接产出文件。`66` 表示资料归属文档额外列入了 `7.P1 随机资料移交` 和 `8.P1 资料服务器核查` 两个关键流程节点。为了避免把业务节点误导入普通阶段资料模板，本规划以 64 项为准。

review 复核后的 64 项阶段分布固定为 3/15/4/17/17/2/4/2：第 3 阶段只保留技术协议、销售合同、销售合同审核记录表和预付款发票；项目启动书归入第 4 阶段；采购申请表归入第 5 阶段；发货单和现场安装调试记录归入第 7 阶段；尾款发票和项目结题报告归入第 8 阶段。

备选方案是保留 66 行作为模板输入，但这会把没有明确独立文件的节点也纳入资料齐套门禁，后续容易造成阶段推进被非文件节点阻塞。

### 资料项只表达阶段产出文件

每个资料项只记录阶段、文件名称、必填口径、`ownerDepartment`、`reviewDepartment`、提交方式和备注。发票、合同审核记录表、采购申请表、采购合同审核记录表、设计变更资料等可以作为产出文件被列入 64 项，但本 change 不定义它们背后的付款、审批、采购或变更流程。

备选方案是同步设计复杂业务流，但这会让模板规划与流程实现混在一起，超出当前审查目标。

### 使用阶段内稳定 documentCode

`v20260624` 规划表使用阶段内稳定编号作为 `documentCode`：立项阶段为 `1.1` 起，方案设计阶段为 `2.1` 起，合同签订阶段为 `3.1` 起，详细设计阶段为 `4.1` 起，生产制作阶段为 `5.1` 起，预验收阶段为 `6.1` 起，终验收阶段为 `7.1` 起，结题阶段为 `8.1`。同名资料必须通过编号区分，例如方案设计阶段 `2.4 3D模型`、详细设计阶段 `4.3 3D模型`、生产制作阶段 `5.13 3D模型（设计变更）`。

备选方案是继续使用全局序号 1-64，但全局序号不适合作为后续模板代码，也不利于阶段内增删时保持同阶段语义稳定。

### 拆分中心字段和非中心确认对象

`ownerDepartment` / `reviewDepartment` 只能保存可落地业务中心枚举或空，包括 `operations_center`、`marketing_center`、`manufacturing_center`、`rd_center`。总经理、客户、项目经理、供应商、相关负责人等不是中心枚举，必须放入“审核/确认对象备注”或来源备注。

备选方案是在 `ownerDepartment` / `reviewDepartment` 中混写中文角色和业务对象，但这会破坏后续权限、工作台和模板初始化逻辑的结构化口径。

### 拆分模板默认必填和适用条件

`模板默认必填` 只表达布尔口径，后续可直接映射当前系统 `isRequired`。客户要求、付款/发票节点、设计变更等条件性资料统一写入 `适用条件`，例如“客户要求时适用”“对应付款/发票节点发生时适用，待业务确认”“发生设计变更时适用”。

备选方案是在必填字段写自由文本，但这会让后续实现无法稳定映射到现有 `isRequired` 布尔字段，也容易把条件性资料误设为所有项目默认阻塞项。

### 阶段推进沿用现有门禁

后续实现 `v20260624` 模板时，阶段推进仍沿用现有机制：当前阶段适用且必填资料必须全部审核通过，阶段审批也必须通过。此规划不新增状态机、不改变阶段审批流程、不引入自动触发审批。

前端阶段资料汇总必须区分“全部阶段产出”和“推进门禁”。非必填/条件性资料需要在阶段汇总区可见，并保留责任人分配、提交、审核和适用性操作能力；但这些资料不计入适用必填齐套门禁，不应影响阶段推进判断。

### 模拟数据可重置

当前项目资料属于模拟数据。切换 `v20260624` 模板时，清理旧项目资料并按新版 64 项重新初始化，不需要为模拟数据设计 `v20260610` 到 `v20260624` 的兼容迁移、双模板共存或历史映射。reset 脚本必须确认当前连接库是 `digital_platform`，然后清理旧项目阶段资料快照、阶段资料附件记录、附件物理文件、阶段资料相关业务日志、阶段审批历史和旧模板，并将项目流程状态重置为第 1 阶段 current、其他阶段 not_started、所有阶段审批 not_submitted、项目状态 normal。

## Risks / Trade-offs

- 64 项表仍基于人工阅读流程图和既有审查文档，个别文件的最终命名、责任中心或必填口径可能需要业务复核 -> 在规划表备注中标记待审查项，后续实现前再确认。
- 发票、合同审核、采购申请和设计变更文件被列入 64 项，但复杂流程暂不建模 -> 本 change 只作为普通产出文件处理，复杂流程应另开 change。
- 不兼容旧模拟资料可降低实现复杂度，但会清理现有模拟项目资料 -> 仅适用于当前模拟数据环境，真实生产数据上线前必须重新评估迁移策略。
- `7.P1` 和 `8.P1` 被排除后，流程节点仍需要业务管理 -> 后续若确认存在移交记录或核查记录，应另行补充模板或 checklist 能力。
