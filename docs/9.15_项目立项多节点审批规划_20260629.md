# 9.15 项目立项多节点审批规划 20260629

## 当前基线

本规划基于已归档的 20260625 在线平台内部资料闭环、精准返工能力和项目全量可见/业务日志查看口径：

- 项目流程仍为 8 阶段、64 项普通阶段资料。
- `completionMode` 总体统计保持 `submit_only 33`、`approval_required 24`、`conditional_submit 7`、`conditional_approval 0`。
- 阶段资料附件仍保存在在线平台，不恢复文件管理平台联动。
- 精准返工已明确 `1.2 项目立项审批表` 审批 NO 时固定返工候选为 `1.1 项目需求表`。
- 中心负责人、总经理助理等角色的全量查看权限不等于跨中心业务操作权限。

## 当前问题

当前 `1.2 项目立项审批表` 仍按普通 `approval_required` 单资料审核处理。实际风险是：营销中心负责人或某个中心负责人对该资料审核通过后，资料基础状态就可能进入 `confirmed`，从而被阶段齐套、阶段推进和项目编号后置门禁视为 `1.2` 已完成。

这与流程图中的“商务评价审批、技术评价审批、总经理审批”多节点口径不一致。`1.2` 不能只由单个中心负责人或单个资料审核动作通过后就放行第 1 阶段或项目编号填写。

## 目标口径

`1.2 项目立项审批表` 必须成为特殊多节点审批资料。它仍属于当前 64 项普通阶段资料模板，仍占用既有 `approval_required` 数量口径，但最终完成状态不能只依赖 `project_stage_documents.status = confirmed`。

`1.2` 最终完成必须同时满足：

- 商务评价审批通过。
- 技术评价审批通过。
- 总经理审批通过。
- `1.2` 自身没有未处理退回状态。
- `1.1 项目需求表` 不存在由 `1.2` NO 触发且未清除的 `revision_required`。
- `1.2` 自身通过节点状态、基础状态或专用多节点状态表达待审、退回或未通过，不作为返工目标写 `revision_required`。

任意单个节点通过不得使 `1.2` 被阶段齐套或项目编号门禁视为完成。

## 审批节点规划

第一版只规划 `1.2 项目立项审批表` 的专用多节点审批，不做通用审批流引擎。

已确认业务规则：

- `1.2` 普通资料提交/上传入口继续保留，并作为多节点审批启动条件。
- 只有 `1.2` 基础状态达到 `submitted`，或兼容旧数据的 `confirmed`，商务评价和技术评价节点才进入待审批/可审批。
- `1.2 status = not_submitted` 时，三类节点可以预创建，但商务评价和技术评价节点必须等待资料提交，不得进入审批人工作台。
- `1.2 status = returned` 时，必须等待普通 `1.2` 资料重新提交后，商务评价和技术评价节点才重新进入审批流程。
- 旧 `confirmed` 只作为“资料已提交过”的兼容输入，不得视为多节点最终通过。
- 商务评价审批与技术评价审批并行。
- 商务评价审批人固定为营销中心负责人。
- 技术评价审批人固定为研发中心负责人。
- 总经理审批：由总经理负责。
- 总经理审批待办只在商务评价审批和技术评价审批都通过后生成。
- 总经理审批通过后，`1.2 项目立项审批表` 才可最终通过。

退回重跑范围已确认：

- 商务评价退回：商务评价节点进入退回阻塞状态；已通过的技术评价节点 MUST 保留；总经理节点 MUST 失效、清空或回到未开始。
- 技术评价退回：技术评价节点进入退回阻塞状态；已通过的商务评价节点 MUST 保留；总经理节点 MUST 失效、清空或回到未开始。
- 总经理审批退回：总经理节点进入退回阻塞状态；已通过的商务评价和技术评价节点 MUST 保留。
- 并行节点中未被退回的一侧已通过结果 MUST 保留，不强制重跑。

## 精准返工联动

本规划复用已归档的精准返工口径：

- `1.2 项目立项审批表` 任一节点审批 NO 时，必须回到 `1.1 项目需求表`。
- 固定返工候选为 `1.1`，不得自由勾选第 1 阶段全部资料。
- 不整阶段退回。
- 不自动退回全部前置资料。
- 审批人必须填写退回原因/返工原因。
- 被选中的 `1.1` 设置 `revision_required = true`，并记录来源审批资料、请求人和请求时间。
- `1.2` 自身不作为返工目标写 `revision_required`，其阻塞由节点状态、基础状态或专用多节点状态表达。
- `1.1 revision_required` 未清除前，`1.2` 不得最终完成。
- `1.1 revision_required` 未清除前，被退回的 `1.2` 节点不得被审批通过。
- `1.1` 返工清除后，后端必须自动将被退回的 `1.2` 节点回到待审批/可审批状态，并回到对应节点审批人的待办。
- 第一版不新增 `1.2` 重提按钮，不新增单独的 `1.2` 重提待办。
- 被退回节点仍必须由对应节点审批人重新审批通过，不得绕过节点审批人。
- 只有重跑节点及其下游节点重新满足后，`1.2` 才能最终完成。

`1.2` 的多节点审批 NO 与精准返工的关系是：节点退回动作产生 `1.2` 节点退回状态，同时按 A 类固定候选返工规则标记 `1.1` 返工。不得把 `1.1` 基础状态改成 `returned`，也不得把第 1 阶段整体退回。

## 阶段推进门禁

第 1 阶段推进仍要求当前阶段所有适用资料按各自 `completionMode` 完成，且不存在未清除的 `revision_required`。

针对 `1.2` 的特殊口径：

- 不能因为商务评价、技术评价或某个中心负责人单独通过就放行。
- `1.2` 必须多节点全部通过后，才可被阶段齐套判断视为完成。
- `1.2` 如果存在任一节点待审、退回、未通过或相关精准返工未清除，必须计入未完成资料。
- `1.3 项目立项通知` 仍按 `submit_only` 完成规则处理。

## 项目编号后置门禁

项目创建时项目编号仍允许为空。项目编号填写门禁从“`1.2 confirmed + 1.3 submitted`”升级为：

- `1.2 项目立项审批表` 多节点审批最终通过。
- `1.3 项目立项通知` 已提交/上传完成。
- `1.1` 不存在由 `1.2` NO 触发且未清除的精准返工。
- `1.2` 自身不存在待审、退回、未通过或其他专用多节点阻塞状态。
- 当前用户仍需具备既有项目编号填写/维护业务权限；仅系统管理员身份不等于业务项目编号维护权限。

不允许商务评价或技术评价单独通过后提前填写项目编号。

## 工作台待办

工作台仅新增 `1.2` 专用多节点审批待办，不恢复泛化阶段关口审批待办。

- 商务评价审批待办只给商务评价审批人。
- 技术评价审批待办只给技术评价审批人。
- `1.2 status = not_submitted` 时，不生成商务评价或技术评价审批待办。
- `1.2 status = returned` 时，必须等待普通 `1.2` 资料重新提交后，才重新生成商务评价或技术评价审批待办。
- `1.2 status = submitted` 或旧 `confirmed` 兼容初始化后，商务评价和技术评价审批待办并行出现。
- 总经理审批待办只在商务评价和技术评价都通过后生成，并只给总经理。
- 不给所有中心负责人生成 `1.2` 审批待办。
- 不生成 `stage_gate_approval` 待办。
- 不做推送通知，仅通过工作台查询接口拉取。
- 资料责任、资料审核、阶段推进待办继续保持既有边界。

## 前端规划

项目详情中 `1.2 项目立项审批表` 需要展示多节点审批状态：

- 商务评价、技术评价、总经理审批节点状态。
- 节点审批人或审批角色。
- 审批意见/退回原因。
- 审批时间。
- 当前整体完成状态。
- 因精准返工导致未完成的原因。

商务评价和技术评价可并行展示和处理。总经理节点在商务评价和技术评价都通过前应显示为未开始或等待前置，不显示可处理待办。退回后，被退回节点显示退回、返工阻塞或后端返回的待审批状态；并行另一侧如已通过则 MUST 保持通过显示；总经理节点在任一前置节点退回后显示失效或等待前置重新完成。

当前用户仅在自己有权处理的节点看到审批通过/退回入口。页面不能把 `1.2` 多节点审批展示为泛化阶段关口审批，也不能因为中心负责人全量查看项目而显示跨中心审批入口。

第一版前端只保留现有资料提交/上传入口来启动 `1.2` 多节点审批，不新增 `1.2` 专用节点提交按钮。节点审批入口只用于有权审批当前节点的审批通过/退回动作。

阶段推进按钮、项目编号填写入口仍必须按后端权限字段和门禁结果展示。

## 后端架构规划

后续实现建议新增 `project_initiation_review_nodes` 或等价专用结构保存 `1.2` 多节点审批状态。建议字段包括：

- `project_id`
- `stage_document_id`
- `node_key`：`business_review`、`technical_review`、`general_review`
- `node_status`：至少覆盖 `waiting_document_submission`、`pending`、`approved`、`returned_blocked_by_rework`、`not_started` / `waiting_prerequisite`，可选支持 `invalidated`
- `reviewer_user_id` 或 `reviewer_role`
- `comment`
- `return_reason`
- `submitted_by_user_id`
- `submitted_at`
- `reviewed_by_user_id`
- `reviewed_at`

`submitted_by_user_id`、`submitted_at` 如保留，只用于追溯普通 `1.2` 资料提交/上传触发多节点激活的事实，不表示节点自身需要前端提交按钮才能从 `submitted` 状态流转。节点状态枚举中不应设计需要前端“节点提交”按钮才能继续审批的 `submitted` 状态。

`project_stage_documents.status` 仍保留资料基础状态，但 `1.2` 的业务完成状态必须结合多节点状态派生。不能复用 legacy 阶段审批流作为当前主路径，也不能引入通用审批流引擎。

后续 migration 或幂等初始化逻辑必须为既有项目中适用的 `1.2 项目立项审批表` 创建 `business_review`、`technical_review`、`general_review` 三类节点。既有项目初始化必须按 `1.2` 基础状态分流：

- `not_submitted`：节点存在但等待资料提交，不生成商务/技术审批待办。
- `submitted`：商务评价和技术评价节点进入待处理/可处理状态，总经理节点进入未开始或等待前置状态。
- 旧 `confirmed`：作为“资料已提交过”的兼容输入，商务评价和技术评价节点进入待处理/可处理状态，总经理节点等待前置；但不得直接视为多节点最终通过。
- `returned`：节点存在但等待普通 `1.2` 资料重新提交，不生成商务/技术审批待办。

节点状态机必须按已确认规则实现：普通 `1.2` 资料提交/上传后，`business_review` 与 `technical_review` 并行可处理；两者均 `approved` 后，`general_review` 进入待处理/待办；`general_review approved` 后，`1.2` 才最终通过。商务或技术任一退回时，总经理节点必须失效、清空或回到未开始；总经理退回时不强制商务/技术重跑。

## 业务日志规划

`1.2` 多节点审批必须写结构化业务日志。建议动作类型：

- `initiation_review.submitted`
- `initiation_review.business_approved`
- `initiation_review.business_returned`
- `initiation_review.technical_approved`
- `initiation_review.technical_returned`
- `initiation_review.general_approved`
- `initiation_review.general_returned`
- `initiation_review.completed`

其中 `initiation_review.submitted` 如保留，表示普通 `1.2` 资料提交后多节点审批启动/节点激活，不代表新增前端节点提交按钮或独立节点提交流程。

日志至少包含：

- 项目 ID。
- `1.2` 资料 ID、资料编号和资料名称。
- 节点 key、节点名称和节点状态变化。
- 审批人/操作人。
- 审批意见或退回原因。
- 操作时间。
- 若触发精准返工，关联 `document.revision_requested` 的来源资料和目标资料上下文。

失败操作不得写成功日志。节点状态变更、精准返工字段变更和业务日志必须在同一事务中提交。

## 非目标

本 change 只做规划，不实现代码，不改数据库，不写 migration，不运行 migration。

后续实现也不得因本规划：

- 恢复文件管理平台联动。
- 处理日报/周报。
- 处理用户管理。
- 实现推送通知。
- 实现通用审批流引擎。
- 改变 64 项资料模板数量。
- 改变 `completionMode` 总体口径。
- 改变中心负责人全量查看项目/资料/日志的查看权限。
- 放宽任何跨中心资料操作权限。
- 新增泛化阶段关口审批待办。
