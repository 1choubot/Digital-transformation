## Context

当前方案设计 workflow 已有覆盖较完整的 `test:solution-design` 回归套件，包含角色、节点、上传、在线表单、模板生成、派生齐套和自动推进。立项阶段已经具备专用在线表单、`1.2` 多节点评价/审批、精准返工、`1.3` 项目编号写入和阶段推进规则，但关键路径缺少同等级的后端自动化回归保护。

这会放大后续工作的风险：权限 resolver、共享阶段机制和合同签订阶段都可能复用或触碰阶段资料、工作台待办、派生完成和自动推进逻辑。该 change 只规划补测试，不改变现有业务行为。

## Goals / Non-Goals

**Goals:**

- 为立项阶段 `1.1 / 1.2 / 1.3` 建立后端 repository 级关键路径回归测试规划。
- 覆盖 `1.1 项目需求表` 提交、`1.2 项目立项审批表` 多节点审批、总经理不通过触发精准返工、返工重提、`1.3 项目立项通知` 前置门禁、项目编号唯一性和立项完成自动推进。
- 覆盖工作台待办在立项关键路径中的稳定性，避免关键待办漏出或无关待办乱增。
- 增加方案设计完成后 C04-C19 派生完成并自动推进到第 3 阶段的集成回归规划，保护立项到方案设计再到合同阶段的阶段链路。
- 复用现有后端 test harness、mock db 和 repository 级测试风格，避免真实数据库和外部服务依赖。

**Non-Goals:**

- 不实现测试代码。
- 不改业务代码、前端代码、migration 或数据库表。
- 不改变现有 API、DTO、权限、状态机、operation log、自动推进或模板生成行为。
- 不改变 8 大阶段和 71 项资料数量。
- 不实现合同签订阶段业务。
- 不清理旧模板，不下线旧阶段关口审批。
- 不用前端 E2E 替代后端关键路径测试。

## Decisions

### 1. 使用 repository 级后端测试作为主保护层

后续实现 SHOULD 优先放在 `digital-platform-api/test/projects/`，沿用现有 fake db / repository 测试模式。这样可以直接断言资料状态、精准返工状态、项目阶段、工作台待办、operation log 和自动推进结果，且不需要真实 MySQL、文件平台或浏览器环境。

Alternative considered: 只补前端 smoke 或 Playwright E2E。该方案能发现部分 UI 问题，但无法稳定覆盖状态机、派生完成和待办生成细节，也更慢、更脆弱，因此不作为主保护层。

### 2. 将立项主链路拆成少量高信号场景

后续实现不需要为每个字段写宽泛测试，而是围绕业务断点建场景：

- `1.1 项目需求表` 提交成功后资料完成状态、生成文件或表单状态符合现有规则。
- `1.2 项目立项审批表` 在 `1.1` 完成后进入可协同填写/审批状态。
- 营销评价、研发评价、总经理最终审批通过形成 `1.2` 派生完成。
- 总经理最终审批不通过触发 `1.1` 精准返工，且阻断 `1.3` 和阶段推进。
- `1.1` 返工重提后 `1.2` 可重新进入审批。
- `1.3 项目立项通知` 提交前必须满足 `1.2` 完成且 `1.1` 无未清返工。
- `1.3` 提交时项目编号写入 `projects.project_code`，且非空项目编号唯一。
- 立项齐套后自动推进到方案设计阶段。

### 3. 单独保留第 2 阶段派生自动推进集成回归

虽然本 change 聚焦立项测试，但后续实现 MUST 覆盖方案设计完成后 C04-C19 派生完成并自动推进到第 3 阶段。该测试用于防止立项测试只保护第 1 阶段，而后续共享阶段机制改动破坏第 2 阶段派生齐套和自动推进组合链路。

### 4. 工作台待办测试只覆盖关键口径

后续实现 SHOULD 验证关键立项处理项在对应阶段出现，例如 `1.1` 责任处理、`1.2` 营销/研发评价、总经理审批、`1.1` 精准返工、`1.3` 处理。测试也应断言不生成与当前主流程冲突的普通阶段推进待办或无关资料待办。测试不需要把工作台 UI 表现作为本 change 的验收对象。

## Risks / Trade-offs

- 测试桩与生产 SQL 行为不一致 -> 后续实现 MUST 尽量复用现有 fake db 模式，并补齐与真实 repository 查询/更新路径对应的表行和状态。
- 测试过宽导致维护成本高 -> 场景 SHOULD 聚焦关键状态断点、权限断点和自动推进结果，不对无关字段做脆弱断言。
- 新增测试脚本与现有脚本重复 -> 如果新增 `test:initiation-workflow`，它 SHOULD 与现有 Node test runner 风格一致，并保留 `npm run check` 作为语法和 smoke 基线。
- 误把测试规划当成业务变更 -> specs 和 tasks MUST 明确这些测试只作为回归保护，不改变现有业务口径。
