## Context

`plan-stage-document-template-v20260629-71-v1` 已归档，口径为：下一版目标资料模板以 20260629 PDF 图面产出为目标基线，在 67 项图面产出基础上补 4 个流程图漏画草稿产出形成 71 项候选；20260625 64 项只作为当前运行和迁移对照；第一版默认文件上传；上方蓝色模块产出卡片是未来主操作入口。

本 change 实现“大框架 shell 第一版”，但不把 `v20260629` 接入新项目默认初始化。

## Goals / Non-Goals

**Goals:**

- 定义 `v20260629` 目标模板配置和 71 项产出映射，但该配置不作为运行默认模板。
- 定义受控开关设计边界，但不在本 change 默认切换新项目模板。
- 明确旧项目第一版不自动补初始化、不迁移、不改 64 项资料状态。
- 实现项目工作区返回/渲染阶段、蓝色模块和产出卡片 shell。
- 实现产出卡片入口和状态展示：责任人、状态、阻塞原因、附件/提交/审核处理入口。
- 非立项产出卡片处理入口定位到旧资料清单对应资料，不直接执行上传、提交、审核或退回。
- 规划旧资料清单降级为辅助兼容区。
- 规划权限、验收和风险边界。

**Non-Goals:**

- 不在本 change 做复杂在线表单。
- 不新增通用流程引擎。
- 不做文件平台联动。
- 不做付款流、发票流或项目模式分支。
- 不一次性精细实现所有阶段审批/返工。
- 不删除旧资料清单。
- 不处理 `file-platform-integration-v1` 或 `define-digital-platform-v1`。
- 不改数据库或 migration。

## Decisions

### Decision 1: 第一版只落 shell，不完成所有业务流

本 change 的第一版目标是让 8 阶段、蓝色模块、产出卡片和旧清单兼容区的承载结构成立。产出卡片只做入口和状态展示，不默认承接通用文件上传、提交、审核/退回的执行迁移；复杂在线表单、细粒度审批/返工、付款流、发票流和项目模式分支必须后续分 change 处理。

### Decision 2: 建议新增目标模板版本 v20260629

后续实现可定义 `v20260629` 目标模板配置，表达 71 项产出、阶段、蓝色模块、产出卡片、草稿/成品和第一版文件上传规则。该配置在本 change 中不得成为新项目默认模板；当前 20260625 64 项继续作为运行安全边界，未切换的项目不得被隐式改写。

### Decision 3: 本 change 不默认切换新项目模板

本 change 只允许定义 `v20260629` 配置或受控开关设计。真正让新项目使用 `v20260629` 默认初始化，必须另开 change 明确开关、模板版本记录、回滚策略、验证和迁移影响。

### Decision 4: 旧项目第一版不自动补初始化或迁移

旧项目第一版不自动补初始化、不迁移、不改 64 项资料状态。如需要旧项目迁移，必须另开 change 单独定义 64 到 71 的映射、保留项、废弃项、草稿/成品拆分、附件和责任人追溯规则。

### Decision 5: 上方产出卡片 shell 承载入口和状态展示

目标工作区结构为：项目总览 -> 项目工作区 -> 8 阶段 -> 蓝色模块 -> 产出卡片 -> 责任人/状态/入口展示。非立项产出卡片 shell 第一版展示资料状态、责任人、阻塞原因和处理入口；用户点击处理入口后，页面定位到旧资料清单对应资料。产出卡片不直接执行上传、提交、审核或退回；真正迁移执行能力必须后续逐阶段实现。

### Decision 6: 立项在线表单继续沿用

立项 `1.1 / 1.2 / 1.3` 继续使用现有在线表单入口和 `1.2` 专用评价审批能力。shell 实现不得把这三项回退为旧资料清单普通文件上传主入口。

### Decision 7: 旧资料清单保留为兼容区

下方旧资料清单在对应阶段迁移稳定前不得物理删除。旧清单和上方产出卡片必须共享后端资料状态，避免出现两个互相矛盾的资料完成状态。

### Decision 8: 权限和状态以后端返回为准

前端不得本地推断责任人分配、附件上传、提交、审核或退回权限。产出卡片必须使用后端返回的 `permissions`、`status`、`completionMode`、`blockingReasons` 和责任人字段驱动展示。

## Data / API Shape Direction

后续 shell 实现可在现有项目工作区接口或独立 workspace 查询能力中返回以下概念，具体字段由 implementation change 决定：

- `templateVersion`: 当前项目使用的模板版本，例如 `v20260625` 或 `v20260629`。
- `stages`: 8 阶段列表。
- `modules`: 阶段内蓝色模块列表。
- `outputs`: 蓝色模块下的产出卡片列表。
- `document`: 产出绑定的项目级资料项。
- `permissions`: 责任人、附件、提交、审核、退回等入口展示权限。
- `blockingReasons`: 不可提交、不可审核、不可推进的原因。

本 shell implementation 可新增或调整只读 workspace shell 字段，但不得通过这些字段切换运行模板、迁移旧项目或创建第二套上传/提交/审核执行逻辑。

## Risks / Trade-offs

- [Risk] 一次性做完整业务流会扩大到流程引擎。 -> Mitigation: 第一版只落大框架，复杂流程后续分 change。
- [Risk] shell change 顺手切换新项目默认模板会扩大影响。 -> Mitigation: 本 change 只允许配置和受控开关设计，真正切换另开 change。
- [Risk] 旧项目自动补初始化会改写 64 项历史状态。 -> Mitigation: 第一版旧项目不自动补初始化、不迁移、不改资料状态。
- [Risk] 产出卡片直接承接通用提交/审核会引入跨阶段行为回归。 -> Mitigation: 第一版只定位旧资料清单对应资料，通用执行迁移后续逐阶段做。
- [Risk] 前端为了展示产出卡片本地硬编码权限。 -> Mitigation: 权限和状态必须以后端返回为准。
- [Risk] 旧清单过早删除导致非立项阶段无可用操作入口。 -> Mitigation: 旧资料清单作为兼容区保留。
- [Risk] 文件上传误解为文件平台联动。 -> Mitigation: 第一版只使用数字化平台现有附件/文件上传能力，不做文件平台联动。

## Migration Plan

本 change 不执行迁移。后续 implementation change 应至少拆成：

1. A 层：定义非运行默认的 `v20260629` 71 项配置和受控开关设计。
2. B 层：返回并渲染项目工作区 8 阶段、蓝色模块、产出卡片 shell。
3. C 层：产出卡片展示入口、责任人、状态和阻塞原因，并定位旧资料清单对应资料。
4. 后续独立 change：新项目默认模板切换。
5. 后续独立 change：旧项目迁移或补初始化。
6. 后续逐阶段 change：将通用文件上传、提交、审核/退回执行能力迁移到产出卡片。
