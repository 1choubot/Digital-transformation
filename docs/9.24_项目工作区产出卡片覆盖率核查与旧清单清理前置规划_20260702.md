# 9.24 项目工作区产出卡片覆盖率核查与旧清单清理前置规划

日期：2026-07-02

## 1. 背景

当前系统已经将现有阶段资料的通用操作迁移到上方项目工作区产出卡片，下方旧资料清单降级为兼容区。下一步如果要默认折叠、隐藏或最终删除旧资料清单，必须先确认当前运行的 20260625 64 项资料是否都能在上方项目工作区找到对应产出卡片和主处理入口。

本规划用于建立覆盖率核查口径，不实施业务代码改造。

## 2. 当前目标

本 change `audit-workspace-card-coverage-before-legacy-checklist-cleanup-v1` 的目标是核查当前运行的 64 项资料与上方项目工作区产出卡片的覆盖关系，并输出旧资料清单清理前置判断依据。

本 change 明确不是：

- 不是切换 v20260629 71 项模板。
- 不是隐藏、折叠或删除旧资料清单。
- 不是迁移旧项目或补初始化旧项目。
- 不是新增上传、提交、审核、退回、返工或不适用逻辑。
- 不是数据库 schema 调整，不写 migration。

## 3. 核查对象

- 当前运行模板：20260625 64 项阶段资料。
- 对照目标：上方项目工作区阶段、蓝色模块和产出卡片。
- 辅助参照：v20260629 71 项目标模板配置，只用于判断映射和后续模板差异，不作为运行模板切换依据。
- 清理对象：下方旧资料清单的默认展示形态和未来清理可能性；本 change 不直接执行隐藏、折叠或删除。

## 4. 核查状态定义

覆盖核查 MUST 拆分两个维度：`workspaceCoverageStatus` 判断当前资料是否已有上方 workspace card 主入口；`targetTemplateStatus` 仅记录 v20260629 目标模板差异，作为后续切换 71 模板的参考。

旧资料清单是否可以进入默认折叠、隐藏或删除的后续 change，主要看 `workspaceCoverageStatus`；`targetTemplateStatus` 不能直接决定当前旧资料清单清理。

### workspaceCoverageStatus

- `covered_by_workspace_card`：当前 64 项运行资料存在上方产出卡片，且卡片绑定现有 documentId 或稳定 documentCode，可作为主操作入口。
- `legacy_only`：当前运行资料仍只能在下方旧资料清单中找到主入口，上方没有有效产出卡片。
- `shell_placeholder_only`：上方存在 shell 或候选产出，但未绑定当前运行资料，不能执行通用操作。
- `needs_mapping_fix`：资料应当有上方卡片，但 documentCode、阶段、蓝色模块或绑定关系存在明显映射问题。
- `needs_business_confirmation`：技术上可映射，但业务语义、草稿/成品拆分、责任中心或清理策略需要业务确认。

### targetTemplateStatus

- `kept_in_v20260629`：该当前运行资料在 v20260629 目标模板中继续保留。
- `removed_in_v20260629`：该当前运行资料在 v20260629 目标模板中不再保留。
- `split_in_v20260629`：该当前运行资料在 v20260629 目标模板中被拆分为多个目标产出。
- `renamed_or_mapped_in_v20260629`：该当前运行资料在 v20260629 目标模板中更名或映射为不同目标产出。
- `needs_business_confirmation`：该当前运行资料与 v20260629 目标模板的关系需要业务确认。

## 6. 覆盖率核查结果

核查来源：

- 当前运行 64 项资料主表：`digital-platform-api/src/domain/stageDocumentTemplateItemsV20260624.js` 导出的 `STAGE_DOCUMENT_TEMPLATE_ITEMS_V20260625`。
- v20260629 目标产出和蓝色模块：`digital-platform-api/src/domain/stageDocumentTemplateItemsV20260629.js` 的 `V20260629_TARGET_TEMPLATE_OUTPUTS` 和 `V20260629_WORKSPACE_BLUE_MODULES`。
- workspace 绑定逻辑：`digital-platform-api/src/repositories/projects/workspaceRepository.js` 按 `legacyDocumentCode` 从 checklist 资料中绑定 `documentId/documentCode`。
- 前端主入口逻辑：非立项已绑定资料由上方产出卡片承载通用操作；1.1 / 1.2 / 1.3 保持在线表单和 1.2 专用评价/审批；下方旧清单对已迁移资料降级为兼容只读区和定位入口。

阻塞项：

- `3.3 合同审核记录表（销售合同）`：当前 64 项运行资料存在，但 v20260629 workspace shell 没有对应 `legacyDocumentCode` 产出卡片，仍只能依赖下方旧资料清单。
- `5.4 采购合同审核记录表`：当前 64 项运行资料存在，但 v20260629 workspace shell 没有对应 `legacyDocumentCode` 产出卡片，仍只能依赖下方旧资料清单。

v20260629 目标模板中另有 9 个 target-only 产出（C19、C20、C22、C43、C45、C46、C47、C49、C68），当前 64 项运行资料没有对应旧资料记录；这些是 71 模板后续切换参考或 shell 占位，不用于判断当前旧资料清单是否可以清理。

结论：当前不建议直接进入“旧资料清单默认折叠/隐藏/删除”的后续独立 change。需要先补齐 `3.3`、`5.4` 的 workspace output/card 映射，或通过业务确认明确这两项在当前运行项目中的保留/替代策略；在此之前旧资料清单仍需保留兼容主入口。
