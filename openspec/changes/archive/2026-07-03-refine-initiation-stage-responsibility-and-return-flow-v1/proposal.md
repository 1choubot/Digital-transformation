## Why

领导补充了立项阶段细化流程，现有口径需要补齐项目创建时的商务/技术负责人、立项阶段责任归属和评价拒绝后的退回路径。这个 change 只修正立项阶段内部流程理解，不推翻 v20260629 / 71 项模板，也不引入文件平台联动。

## What Changes

- 创建项目时需要选择两个项目级责任人：
  - 商务负责人：营销中心启用用户。
  - 技术负责人：研发中心启用用户。
- 项目输入节点需要展示项目名称、客户名称、客户联系人和电话、商务负责人、技术负责人。
- 1.1 项目需求表和 1.2 项目立项审批表默认由商务负责人填写。
- 商务评价拒绝、技术评价拒绝需要退回项目市场调研。
- 总经理拒绝需要支持两个去向：退回项目市场调研或项目结束。
- 项目结束后不得继续推进立项通知、方案设计或后续资料操作。
- “自动生成项目需求表 / 项目立项审批表 / 立项通知”第一版按在线表单内容可浏览理解，不生成真实 Word/PDF/文件平台文件。
- 现有 1.1 / 1.2 / 1.3 在线表单和 1.2 专用评价审批主流程尽量保持不变。

不做：

- 不切换或重定义 v20260629 / 71 项模板。
- 不新增资料项，不改变 71 项模板数量。
- 不处理文件平台联动。
- 不导出 Word/PDF，不生成文件平台文件或普通附件。
- 不引入 BPM 或通用流程引擎。
- 不新增第二套资料状态机。
- 不在规划阶段修改业务代码、数据库 schema 或 migration。

## Capabilities

### New Capabilities

- 无。

### Modified Capabilities

- `project-core`: 修正项目创建责任人、立项阶段责任归属、评价拒绝退回路径和项目结束终态要求。
- `project-core-frontend`: 修正新建项目负责人选择、立项阶段项目输入展示、评价拒绝入口、总经理拒绝去向和在线表单浏览要求。
- `stage-document-checklist`: 明确 1.1 / 1.2 / 1.3 仍为在线表单资料，不新增资料项、不生成真实文件、不接文件平台。
- `technical-architecture`: 固化本 change 的架构边界，允许后续实现改 API/Web 和必要数据字段，但禁止引入 BPM、第二套状态机、文件平台或规划阶段业务代码改动。

## Impact

- 后续实现预计影响项目创建 API、项目基础信息存储或等价责任人绑定、新建项目页面、项目详情立项阶段展示、1.1 / 1.2 在线表单责任默认值、1.2 专用评价审批退回路径和项目终态门禁。
- 本轮只新增规划文档和 OpenSpec change artifacts，不修改业务代码，不修改数据库，不写 migration，不提交，不归档。
