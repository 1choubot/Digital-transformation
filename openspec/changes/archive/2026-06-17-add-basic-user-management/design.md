## 背景

当前数字化平台已经有 `users` 表、基础登录、退出登录、当前用户识别和初始化账号脚本。用户表已经保存 `account`、`display_name`、`department`、`role`、`is_enabled`、`file_platform_user_id` 和 `password_hash` 等字段，但缺少管理页面和管理 API。

用户管理不能只做 `requireAuth`，否则任意登录用户都能创建、禁用或重置其他账号。因此第一版需要一个最小平台管理员开关，只保护用户管理本身，不扩展为项目、资料、阶段或文件权限体系。

## 目标 / 非目标

**目标：**

- 为 `users` 增加 `is_platform_admin` 字段，映射为 API 的 `isPlatformAdmin`。
- 初始化账号默认保存为 `isPlatformAdmin = true`，并为已有环境升级提供可恢复平台管理员的兜底。
- 新增平台管理员校验，用户管理接口必须同时满足登录态和 `is_platform_admin = true`。
- 防止用户管理操作导致系统进入 `0` 个启用平台管理员状态。
- 提供用户列表、新增用户、编辑用户基础信息、启用/禁用用户、重置用户密码接口。
- 用户响应不返回 `password_hash` 或密码内部字段。
- 继续保证 `account` 唯一。
- 禁用用户不能建立新登录态。
- 前端新增用户管理页面，并只对平台管理员展示入口或允许操作。
- 更新 API/Web README。

**非目标：**

- 不实现项目成员管理、项目经理账号绑定、责任人分配或个人待办。
- 不实现复杂 RBAC、部门权限继承、项目级权限矩阵、资料项权限或按钮级权限矩阵。
- 不实现审批流、单点登录或批量导入用户。
- 不同步文件平台用户，不读取文件管理平台用户表，不判断文件平台权限。
- 不实现登录审计或用户管理操作日志；第一版不写 `business_operation_logs`，后续如需系统/用户管理审计另起 change 设计。
- 不修改、完成或归档 `define-digital-platform-v1`。

## 设计决策

### 决策一：使用 `is_platform_admin` 作为最小管理员开关

在 `users` 表新增：

- `is_platform_admin TINYINT(1) NOT NULL DEFAULT 0`

API 映射为 `isPlatformAdmin`。该字段只用于用户管理 API 和用户管理页面入口，不参与项目权限、资料权限、阶段权限、文件权限或业务按钮矩阵。

备选方案是复用现有 `role` 字段判断管理员。第一版不采用，因为 `role` 当前是展示性基础字段，可能保存岗位、角色名称或组织角色，直接复用会造成权限语义不稳定。

### 决策二：初始化账号成为平台管理员

初始化账号脚本运行后，初始化账号默认必须保存为 `is_platform_admin = 1`，API 安全用户模型中表现为 `isPlatformAdmin = true`。可以支持显式配置覆盖默认值，但覆盖策略不得导致系统初始化后无人可管理；如果显式配置会让初始化完成后没有任何启用平台管理员，脚本必须拒绝或恢复初始化账号为启用平台管理员。

已有用户迁移时，字段可先默认 `0`，但升级后的初始化账号脚本必须能够把初始化账号设置/恢复为平台管理员，用于现有环境升级兜底。历史环境中如果没有启用平台管理员，重新运行初始化账号脚本必须能恢复至少一个启用平台管理员。

### 决策三：新增管理员校验中间件

用户管理 API 使用两层校验：

1. `requireAuth` 识别当前登录用户。
2. `requirePlatformAdmin` 校验 `req.auth.user.isPlatformAdmin === true`。

这样保持第一版边界清晰：用户管理不是复杂权限矩阵，但也不是任意登录用户可访问。

### 决策四：至少保留一个启用平台管理员

第一版必须维护用户管理自身的基础安全不变量：系统不得进入 `0` 个启用平台管理员状态。

以下操作都必须在保存前校验该不变量：

- 禁用用户。
- 取消用户的 `isPlatformAdmin`。
- 普通编辑中同时修改 `isEnabled` / `isPlatformAdmin`。

如果操作会导致没有任何 `is_enabled = 1` 且 `is_platform_admin = 1` 的用户，系统必须拒绝操作并返回稳定错误，例如 `LAST_ENABLED_PLATFORM_ADMIN_REQUIRED`。该规则只保护用户管理自身，不能扩展成项目权限、部门权限、资料权限、文件权限或按钮级权限矩阵。

### 决策五：统一用户字段命名口径

用户字段命名按层次统一：

- 请求字段使用 `displayName`。
- 响应字段沿用当前安全用户模型 `name`。
- 数据库存储字段仍为 `display_name`。
- 前端新增/编辑表单提交 `displayName`。
- 前端展示用户姓名时使用响应中的 `name`。

规格和实现不得继续使用 “`displayName` 或 `name`” 的二选一口径。这样既保持请求语义清晰，也避免破坏现有登录态和当前用户安全响应模型。

### 决策六：用户管理 API 使用安全响应模型

所有用户管理响应复用安全用户模型，返回：

- `id`
- `account`
- `name`
- `department`
- `role`
- `isEnabled`
- `isPlatformAdmin`
- `filePlatformUserId`
- 必要时返回 `createdAt`、`updatedAt`

不得返回 `password_hash`、密码盐、重置令牌或其他密码内部字段。

新增和编辑请求仍使用 `displayName`，由后端保存到数据库 `display_name`，再在响应中映射为 `name`。

### 决策七：普通编辑和密码重置分离

普通编辑接口只允许修改基础信息：

- `displayName`
- `department`
- `role`
- `isEnabled`
- `isPlatformAdmin`
- `filePlatformUserId`

普通编辑接口不得修改 `account` 或 `password`。密码重置使用单独接口，例如：

- `POST /api/users/:userId/reset-password`

这样可以让 UI 和后端校验更明确，减少误改账号和密码的风险。

### 决策八：第一版不做用户管理审计

虽然平台已有项目维度业务操作日志，但该日志当前只记录项目关键业务状态变化，并要求日志归属于项目。用户管理属于平台基础配置，不归属于项目。第一版不写用户管理操作日志，也不扩展 `business_operation_logs` 为全局审计日志。后续如需系统审计或用户管理审计，应独立设计。

## 风险 / 取舍

- [误把 `is_platform_admin` 扩展成复杂权限] -> 规格明确该字段只保护用户管理 API 和页面入口。
- [升级后无人是平台管理员] -> 初始化账号默认必须保存为平台管理员，迁移/初始化脚本必须提供恢复启用平台管理员的兜底。
- [禁用当前登录用户后的会话处理] -> 第一版沿用现有认证逻辑；如果现有 `requireAuth` 已校验 `is_enabled`，则禁用后继续请求会被拒绝；不新增主动踢下线。
- [管理员误禁用或取消全部管理员] -> 禁用、取消 `isPlatformAdmin` 或同时编辑 `isEnabled` / `isPlatformAdmin` 时必须校验至少保留一个启用平台管理员。
- [用户管理缺少审计] -> 第一版明确不做用户管理操作日志，避免把项目业务日志扩成全局审计。

## 迁移计划

1. 新增数据库迁移，为 `users` 增加 `is_platform_admin` 字段。
2. 更新初始化账号脚本，使初始化账号默认成为平台管理员，并能在已有环境升级时恢复至少一个启用平台管理员；在 README 中说明。
3. 更新用户安全映射、登录响应和当前用户响应，返回 `isPlatformAdmin`。
4. 新增用户字段映射：请求 `displayName`，数据库 `display_name`，响应 `name`。
5. 新增用户管理仓储、管理员校验中间件和用户管理路由。
6. 在禁用、取消平台管理员和编辑启用/管理员字段时校验至少保留一个启用平台管理员。
7. 新增前端用户管理 API 客户端、导航入口和用户管理页面。
8. 更新 API/Web README。
9. 回滚时先移除前端入口和用户管理 API，再保留或移除 `is_platform_admin` 字段；基础登录可继续使用已有字段。

## 待确认问题

当前第一版没有待确认问题。默认采用 `is_platform_admin` 最小管理员开关，不写用户管理操作日志，不引入复杂权限。
