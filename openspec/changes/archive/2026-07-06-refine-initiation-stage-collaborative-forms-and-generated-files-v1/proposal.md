## Why

立项阶段已和领导进一步核对，现有在线表单与真实模板字段、职责分工、项目编号时点和产出形态仍存在差异。需要固化并实现新口径，避免继续沿用“商务负责人填写整张 1.2、总经理通过后再填项目编号、只查看在线表单”的临时方案。

## What Changes

- 新建项目增加客户联系人字段，并在项目主数据、详情、列表、概览和在线表单自动带出中保持一致。
- `1.2 项目立项审批表` 基础模块补齐真实模板表头区域和对应已有字段，不扩展模板外客户/项目调查字段。
- 项目编号前置到 `1.2` 表单填写阶段，`1.2` 商务部分提交前必须已有项目编号，且继续保存到项目主数据并保持唯一。
- `1.2` 由商务负责人和技术负责人协同填写：商务负责人填写基础模块和商务模块，技术负责人填写技术模块。
- `1.2` 商务部分和技术部分都完成后，才进入现有商务评价、技术评价、总经理审批。
- `1.1 / 1.2 / 1.3` 在线表单提交后生成对应模板文件作为后续独立能力记录，本 change 不实现文件生成。

不做：

- 不新增资料项。
- 不改变 v20260629 / 71 项资料数量。
- 不删除兼容资料区。
- 不迁移旧项目。
- 不新增独立数据库 migration 文件；客户联系人通过项目主数据可空字段和 schema 初始化承载，`1.2` 协同状态复用现有在线表单数据存储。
- 不生成 Excel / Word / PDF，不接文件平台。
- 不改变商务评价、技术评价、总经理审批三节点本身。

## Capabilities

### New Capabilities

- 新建项目客户联系人。
- `1.2` 商务/技术协同填写与工作台分工待办。
- `1.2` 项目编号前置填写并带入 `1.3`。

### Modified Capabilities

- `project-core`: 项目主数据增加客户联系人；`1.2` 项目编号前置、协同填写门禁和评价审批启动条件调整。
- `project-core-frontend`: 新建项目、项目详情、`1.2` 在线表单和工作台需要呈现客户联系人、协同填写区域、双方完成状态和 `1.2` 表单内项目编号入口，不再展示顶部独立项目编号入口或 `1.2` 普通责任人分配入口。
- `stage-document-checklist`: `1.1 / 1.2 / 1.3` 仍为原资料项且数量不变，在线表单为填写入口，后续生成文件作为产出查看形态。
- `technical-architecture`: 固化不写独立 migration 文件、不接文件平台、不新增第二套审批状态机，以及文件生成/模板填充/文件存储作为后续独立设计点。

## Impact

- 实现影响项目创建 API、项目主数据返回、项目列表/概览、立项在线表单 schema、`1.2` 保存/提交门禁、工作台待办、`1.3` 自动带出和 smoke。
- 本 change 不提交、不归档；不处理文件平台相关 change，不生成 Excel / Word / PDF，不迁移旧项目。
