## Context

当前平台已经具备登录态、项目详情、阶段资料清单、资料状态流转、资料适用性、阶段资料责任人分配和项目业务操作日志。阶段资料项已经保存 `responsible_user_id`、`responsibility_updated_by_user_id` 和 `responsibility_updated_at`，但用户没有集中入口查看分配给自己的资料项。

“我的资料任务”第一版只把现有项目级阶段资料项按当前登录用户聚合展示，不创建新的业务状态，不生成待办实体表，不写业务日志，也不改变项目详情页已有资料操作能力。

## Goals / Non-Goals

**Goals:**

- 提供 `GET /api/me/stage-document-tasks`，查询分配给当前登录用户的阶段资料项。
- 默认展示未完成且适用的资料项：`not_submitted`、`submitted`、`returned`。
- 支持状态筛选和可选 `projectId` 筛选。
- 返回项目、阶段、资料、状态、适用性、退回原因和责任人更新时间等任务列表所需字段。
- 提供稳定排序，保证后端和前端对任务顺序理解一致。
- 前端新增“我的资料任务”页面和主导航入口，展示列表、状态筛选、项目关键字筛选和项目详情跳转。
- README 说明接口、筛选规则、页面能力和边界。

**Non-Goals:**

- 不实现文件上传/下载、文件管理平台联动或在线表单。
- 不实现消息提醒、超期提醒、截止日期、个人工作台大看板或跨项目统计。
- 不实现批量操作、复杂权限、项目成员权限或“只允许责任人提交”等权限规则。
- 不新增业务日志；本变更第一版只查询和展示，不产生业务状态变更。
- 不新增任务实体表，不复制阶段资料项数据。
- 不重构整个 `ProjectDetailPage.vue`；如需复用状态文案/helper，只做最小范围抽取。

## Decisions

### 使用 `/api/me/stage-document-tasks` 表达当前用户视角

第一版接口固定为 `GET /api/me/stage-document-tasks`。该路径表达“当前登录用户”的查询视角，服务端必须从登录态读取用户 ID，不接收也不信任前端传入的用户 ID。

替代方案是在项目接口下增加跨项目查询参数，但该能力不属于某个单一项目，使用 `/api/me` 更清晰，也能避免前端传入任意责任人 ID。

### 只做 `requireAuth`

接口必须要求登录态，但不要求平台管理员。任务列表只返回当前登录用户自己的责任资料项，不使用 `isPlatformAdmin`，不引入项目权限、资料权限、角色权限或轻角色校验。

平台管理员标识仍只保护用户管理，不扩展为我的资料任务权限。

### 默认待办口径为 `status=pending` 且适用

默认查询等价于 `status=pending`，包含 `not_submitted`、`submitted`、`returned`，并排除 `is_applicable = 0` 的资料项。`confirmed` 可通过 `status=confirmed` 查看，但默认不作为待办展示。

第一版不提供适用性筛选。无论 `status` 取值如何，结果都只返回适用资料项；返回 `isApplicable` 字段用于和阶段资料清单字段保持一致，并为后续扩展保留口径。

### 状态筛选枚举固定

`status` 支持 `not_submitted`、`submitted`、`returned`、`confirmed`、`pending`、`all`。`pending` 展开为 `not_submitted`、`submitted`、`returned`；`all` 展开为四种资料状态，但仍只返回适用资料项。

非法 `status` 必须返回稳定错误码 `INVALID_STAGE_DOCUMENT_TASK_STATUS`，通过统一错误处理返回明确 HTTP 状态，建议为 400；不得回退为默认查询，避免前端筛选状态和后端实际结果不一致。

### projectId 筛选只校验格式

`projectId` 为可选筛选条件。如果提供，必须是正整数。非数字、空字符串、0、负数、小数或其他非法格式必须返回稳定错误码 `INVALID_PROJECT_ID`，通过统一错误处理返回明确 HTTP 状态，建议为 400。

合法正整数 `projectId` 只作为过滤条件使用，第一版不额外校验项目是否存在；如果该项目不存在、该项目没有分配给当前用户的资料项，或筛选组合没有匹配结果，接口返回空列表。这样可以保持列表查询语义，也避免把我的任务查询扩展为项目存在性探测或项目权限规则。

### 项目和阶段状态不参与过滤

第一版不按项目状态、阶段状态或阶段是否当前过滤。只要资料项分配给当前登录用户、适用且状态符合筛选条件，就进入结果；页面通过返回的项目、阶段和资料信息帮助用户判断上下文。

选择该口径是为了让“我的资料任务”保持为责任资料项查询视图，不引入阶段锁定、项目权限、项目成员权限、复杂待办规则或“已完成项目自动隐藏责任资料”的额外业务语义。后续如需要隐藏已完成项目、只展示当前阶段或按阶段状态过滤，应单独设计 change。

### 排序规则固定在后端

后端按以下顺序返回任务：

1. 状态优先级：`returned`、`not_submitted`、`submitted`、`confirmed`。
2. 同状态下按 `responsibility_updated_at` 倒序，空值排在后面。
3. 再按 `project_code` 升序、`stage_order` 升序、`document_order` 升序、`document_id` 升序稳定排序。

选择后端排序是为了让页面、测试和后续接口调用获得一致顺序。前端不重新定义业务排序，只在项目关键字筛选后保留后端返回顺序。

### 前端采用项目关键字筛选

第一版前端支持状态筛选和项目关键字筛选。状态筛选调用后端 `status` 参数；项目关键字筛选在当前返回列表中按 `projectCode`、`projectName` 做前端过滤。

选择项目关键字筛选，是为了避免额外加载项目列表或引入复杂项目选择器。后端仍保留 `projectId` 参数，供后续从项目上下文或链接进入时复用。

### 点击任务只跳转项目详情

第一版点击任务跳转到对应项目详情页。若当前应用已有资料项锚点能力，可以复用；如果没有，不强制实现滚动定位或复杂锚点，因为这会牵涉项目详情页结构调整，超出最小闭环。

## Risks / Trade-offs

- [Risk] 用户可能误以为任务状态代表真实文件或表单状态。 → 页面和 README 必须明确状态是手工标记，不代表文件已上传或在线表单已填写。
- [Risk] 默认不显示已确认资料，用户可能需要回看历史责任项。 → 提供 `status=confirmed` 和 `status=all` 筛选。
- [Risk] 不适用资料被排除后用户无法在任务页查看。 → 第一版保持待办聚焦；如后续需要查看不适用责任资料，应单独增加适用性筛选。
- [Risk] 只跳转项目详情不能直接定位资料项。 → 作为后续增强，不阻塞第一版集中列表能力。

## Migration Plan

本变更不需要数据库迁移。部署顺序建议为先发布后端查询接口，再发布前端页面入口。回滚时可先隐藏前端入口，再回滚后端接口；不会影响既有项目详情、资料状态流转、适用性、责任人分配、阶段推进和业务日志。

## Open Questions

无。第一版固定为查询和展示当前登录用户的责任资料项，不设计通知、截止日期、文件联动或权限规则。
