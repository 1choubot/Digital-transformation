## Context

项目详情当前以阶段资料清单为主要组织方式。该方式适合保存和校验 64 项资料底座，但不适合业务人员按流程图理解项目进度、当前节点、责任人、产出和阻塞原因。

本 change 规划将项目详情重塑为 Project Workspace：项目列表和项目总览继续承担项目选择，项目详情内部建立 8 阶段导航骨架和节点工作区。它的架构层级是：全局建立项目工作区骨架，第一批完整落地立项阶段，后续阶段通过后续 change 逐步补齐。第一版同步调整轻量创建、立项阶段在线表单和 `1.2 项目立项审批表` 的评价/审批模型。

## Goals / Non-Goals

**Goals:**

- 规划项目详情从资料清单视角转为阶段节点工作区视角。
- 规划左侧 8 阶段导航骨架和右侧节点工作区。
- 固化立项阶段 4 个节点及 `1.1`、`1.2`、`1.3` 在线表单产出映射。
- 固化项目创建轻量字段：项目名称、客户、客户联系方式。
- 固化 `1.1`、`1.2`、`1.3` 的责任人和提交规则。
- 将 `1.2` 从旧三审批节点调整为营销评价、研发评价、总经理最终审批。
- 固化 `1.3` 必须在 `1.2` 总经理审批通过后才能填写/提交。
- 保留 64 项资料、`completionMode`、精准返工、阶段推进、项目编号门禁、工作台、业务日志和权限体系。

**Non-Goals:**

- 不实现业务代码。
- 不改数据库、不写 migration、不运行 migration。
- 不实现其他阶段完整节点表单。
- 不一次性补齐其他 7 个阶段的全部蓝色节点映射。
- 不要求其他阶段在本 change 在线表单化。
- 不恢复文件平台联动。
- 不做日报/周报、用户管理或通知推送。
- 不做通用审批流或 BPM。
- 不处理 `file-platform-integration-v1` 或 `define-digital-platform-v1`。
- 不修改 `D:\file-server-local`。

## Decisions

### Information Architecture

Project List / Overview 继续用于选择项目。进入单个项目后，Project Detail MUST 作为 Project Workspace 展示。

Project Workspace 左侧 MUST 展示 8 个阶段导航骨架。第一版 MUST 完整展示立项阶段的蓝色节点；其他阶段 MAY 先展示阶段标题、后续配置状态或旧资料清单入口。右侧 MUST 展示当前选中节点的节点工作区。

节点工作区 SHOULD 包含：

- 节点名称。
- 当前派生状态。
- 关联产出。
- 产出责任人。
- 表单、附件、评价、审批状态。
- 阻塞原因。
- 当前用户可执行操作。
- 最近日志和更新时间。

填写/编辑表单 MUST 通过节点工作区中的“填写/编辑”入口进入。节点第一屏 MUST NOT 直接变成编辑表单。

本 change 的范围必须保持清晰：全局是项目工作区骨架，第一批是立项阶段完整落地，其他阶段逐步补齐。8 阶段导航骨架不等同于一次性完成 8 个阶段的全部节点映射。

### Node State Derivation

蓝色节点 MUST NOT 单独拥有完成状态。节点状态 MUST 从关联产出、在线表单状态、评价/审批状态、精准返工状态和阶段资料完成规则派生。

这样做可以避免“节点状态”和“资料状态”出现两套真相。64 项阶段资料仍是产出数据底座，节点视图只是组织和导航方式。

第一版按一个蓝色节点对应一个或多个独立产出处理。多个蓝色节点共用同一产出的多入口场景暂不支持，等待流程图补齐后再规划。

其他阶段第一版可保留旧资料清单入口，不要求在线表单化，不要求完整节点映射，也不得因其他阶段未配置完整节点而阻塞立项阶段工作区实现。

### Initiation Stage Nodes and Outputs

立项阶段第一版节点为：

- 项目输入。
- 项目市场调研。
- 项目立项审批。
- 项目立项通知。

产出映射为：

- 项目市场调研 -> `1.1 项目需求表` -> 在线表单 -> `项目需求表-模板.xlsx`。
- 项目立项审批 -> `1.2 项目立项审批表` -> 在线表单 -> `项目立项审批表-模板.xlsx`。
- 项目立项通知 -> `1.3 项目立项通知` -> 在线表单 -> `关于确定项目名称及编号的通知-模板.docx`。

`项目输入` 节点承接项目创建时的轻量基础信息，不新增独立阶段资料产出。

其他 7 个阶段在第一版中只要求处在 8 阶段导航框架内；其节点和产出映射必须通过后续 change 逐阶段补齐。

### Online Form Abstraction

立项阶段 `1.1`、`1.2`、`1.3` MUST 规划为在线表单产出。模板文件作为字段设计来源，后续实现时需要将 xlsx/docx 模板转为系统在线表单 schema 和字段映射。

第一版可以先规划 form schema、form data、submission record 和字段映射，不要求直接实现模板解析。在线表单提交结果 MUST 回写或派生阶段资料完成状态，并继续受资料责任人、权限和精准返工规则约束。

### Project Creation

项目创建 MUST 轻量化。创建项目只要求项目名称、客户和客户联系方式。项目经理、项目模式、参与中心、项目编号、计划开始时间、计划结束时间和立项日期 MUST NOT 作为创建必填字段。

立项日期、项目经理、项目模式在业务上后置到第一阶段完成后、第二阶段开始时再考虑补录，但补录能力不属于本 change 的当前实现范围，必须由后续 change 另行规划。参与中心后续弱化或取消，不作为创建必填或核心流程判断依据。

已有项目中的 projectManager、projectMode、participatingDepartments、计划时间和立项日期继续展示兼容。

### Responsibility and Permissions

项目创建权限保持现有边界：总经理和中心负责人等具备创建项目权限的用户仍可创建项目，但创建字段变轻。

立项阶段操作权限 MUST 按下列规则规划：

- `1.1`、`1.2` 责任人分配：营销中心负责人。
- `1.1`、`1.2` 填写提交：对应资料责任人。
- `1.3` 填写提交：营销中心负责人。
- `1.2` 营销评价：营销中心负责人。
- `1.2` 研发评价：研发中心负责人。
- `1.2` 最终审批：总经理。
- 第二阶段补录项目经理、项目模式、立项日期：由后续 change 另行规划补录权限、入口和门禁。

中心负责人、项目经理、总经理、总经理助理或项目创建人不能仅因查看或管理身份代替责任人提交 `1.1`、`1.2`。

### 1.2 Evaluation and Approval State Model

`1.2 项目立项审批表` 不再使用旧的商务审批、技术审批、总经理审批三通过节点作为最终完成规则。新模型为两个评价加一个最终审批：

- 营销中心负责人填写营销评价文本。
- 研发中心负责人填写研发评价文本。
- 总经理执行最终审批。

营销评价和研发评价 MUST 独立保存评价文本、评价人和评价时间。评价没有通过/不通过选项，评价内容不决定最终通过。

总经理审批 MUST 独立保存审批结果、审批意见、审批人和审批时间。两项评价完成前，总经理审批操作 MUST 被后端拒绝，前端按钮 MUST 禁用。两项评价完成后，总经理可以审批。

`1.2` 可使用 `draft`、`submitted`、`evaluating`、`ready_for_general_approval`、`approved`、`returned_for_rework` 或等价设计状态表达整体状态。最终完成 MUST 要求普通 `1.2` 表单已提交、营销评价完成、研发评价完成、总经理审批通过且相关返工已清除。

总经理审批不通过后，系统 MUST 触发 `1.1 项目需求表` 精准返工，并要求 `1.2` 重新填写。`1.2` 原责任人继续处理，不因审批不通过重新分配责任人。

### 1.3 Gate

`1.3 项目立项通知` MUST 在 `1.2` 总经理审批通过后才能填写或提交。系统 MUST NOT 允许 `1.3` 早于 `1.2` 最终通过提交。

### Compatibility

旧资料清单接口可以保留，节点视图作为新的组织方式。正式 specs 中归档后的 `add-initiation-multi-review-flow-v1` 三审批节点口径将由本 change 的 delta 修改，不直接改归档目录。

旧字段继续兼容展示。后续实现前 MUST 设计旧三节点审批数据如何迁移或解释为新评价/审批模型，避免旧 `business_review` / `technical_review` / `general_review` 状态被误认为新模型已完成。

## Risks / Trade-offs

- [Risk] 节点视图与资料清单同时存在，容易出现状态口径不一致 -> Mitigation: 节点状态只派生，不单独保存完成状态。
- [Risk] 旧三节点审批数据与新评价/审批模型语义不同 -> Mitigation: 实现前明确迁移或兼容解释策略，旧通过状态不得自动绕过新模型。
- [Risk] 在线表单模板字段转换工作量较大 -> Mitigation: 第一版先规划 schema 和字段映射，不要求模板自动解析。
- [Risk] 将 8 阶段节点误解为一次性全量实现会扩大范围 -> Mitigation: 明确本 change 只要求全局工作区骨架和立项阶段完整落地，其他阶段允许占位、旧清单入口或后续配置。
- [Risk] 轻量创建会影响依赖项目经理、项目模式、参与中心的旧流程 -> Mitigation: 旧项目字段兼容展示，第二阶段补录由后续 change 单独规划，避免本 change 范围膨胀。
- [Risk] 管理角色查看范围较大，容易误导为操作权限 -> Mitigation: 所有提交、评价、审批、阶段推进和项目编号填写继续以后端权限字段为准。
