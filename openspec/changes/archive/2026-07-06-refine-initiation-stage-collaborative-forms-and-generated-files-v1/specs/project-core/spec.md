## ADDED Requirements

### Requirement: 立项项目主数据客户联系人

项目核心能力 MUST 支持新建项目客户联系人字段；客户联系人 MUST 作为项目主数据参与创建、查询、展示和在线表单自动带出。创建新项目输入 MUST 包含客户联系人，客户联系人为空时不得创建新项目；旧项目缺失客户联系人时允许为空返回。本能力 MUST NOT 新增资料项或改变 v20260629 / 71 项资料数量。

#### Scenario: 创建项目包含客户联系人
- **WHEN** 用户创建新项目
- **THEN** 项目创建输入 MUST 包含客户联系人
- **AND** 客户联系人字段 MUST 位于客户联系方式前
- **AND** 客户联系人为空时系统 MUST NOT 创建新项目

#### Scenario: 项目主数据返回客户联系人
- **WHEN** 系统返回项目详情、项目列表或项目总览数据
- **THEN** 响应 MUST 能包含客户联系人或等价字段
- **AND** 旧项目缺少客户联系人时 MUST 允许为空展示

#### Scenario: 在线表单自动带出客户联系人
- **WHEN** 系统生成或返回立项在线表单上下文
- **THEN** 系统 MUST 支持将客户联系人作为项目主数据自动带出字段
- **AND** 系统 MUST NOT 因客户联系人新增而迁移旧项目或补初始化旧资料

### Requirement: 1.2 项目编号前置到填写阶段

项目核心能力 MUST 将项目编号作为 `1.2 项目立项审批表` 填写阶段内容；项目编号 MUST 保存到项目主数据，`1.2` 提交前必须存在项目编号，且项目编号唯一性继续生效。

#### Scenario: 1.2 提交前必须填写项目编号
- **WHEN** 商务负责人提交 `1.2` 商务部分或整体进入待评价状态
- **THEN** 系统 MUST 校验项目编号已填写
- **AND** 若项目编号为空，系统 MUST 阻止 `1.2` 提交或阻止进入评价审批

#### Scenario: 项目编号唯一性继续生效
- **WHEN** 用户在 `1.2` 填写阶段保存项目编号
- **THEN** 系统 MUST 将项目编号保存到项目主数据
- **AND** 系统 MUST 校验非空项目编号唯一
- **AND** 系统 MUST 拒绝与已有项目重复的项目编号

#### Scenario: 退回和结束后保留项目编号
- **WHEN** `1.2` 被退回、返工或项目被总经理拒绝并结束
- **THEN** 系统 MUST 保留已填写的项目编号
- **AND** 项目编号 MUST 可用于后续审计和 `1.3` 自动带出

#### Scenario: 项目编号不绕过审批
- **WHEN** 项目已有项目编号但 `1.2` 商务/技术填写或评价审批尚未完成
- **THEN** 系统 MUST NOT 因项目编号存在而将 `1.2` 视为最终完成
- **AND** 系统 MUST NOT 因项目编号存在允许绕过商务评价、技术评价或总经理审批

### Requirement: 1.2 商务技术协同填写门禁

项目核心能力 MUST 支持 `1.2 项目立项审批表` 由商务负责人和技术负责人协同填写；商务负责人填写基础模块和商务模块，技术负责人填写技术模块，双方都完成后才进入现有评价审批节点。

#### Scenario: 商务负责人填写基础和商务模块
- **WHEN** 新项目进入 `1.2 项目立项审批表` 填写
- **THEN** 商务负责人 MUST 能填写基础模块和商务模块
- **AND** 基础模块 MUST 按模板表头层级包含顶层表头信息、客户基本信息和项目基本信息
- **AND** 顶层表头信息 MUST 只包含项目名称、项目号/项目编号
- **AND** 客户基本信息 MUST 只包含客户名称、项目联系人、联系方式
- **AND** 项目基本信息 MUST 只包含项目负责人和项目负责人联系方式
- **AND** 基础模块 MUST NOT 增加模板外客户调查或项目调查字段
- **AND** 技术负责人 MUST NOT 被强行加入基础模块表头字段
- **AND** 商务负责人 MUST 填写或确认项目编号

#### Scenario: 技术负责人填写技术模块
- **WHEN** 新项目进入 `1.2 项目立项审批表` 填写
- **THEN** 技术负责人 MUST 能填写技术模块
- **AND** 技术负责人 MUST NOT 修改基础模块和商务模块

#### Scenario: 禁止商务负责人代填技术模块
- **WHEN** 商务负责人处理 `1.2`
- **THEN** 系统 MUST NOT 默认允许商务负责人提交技术模块
- **AND** 若后续需要代填或授权，MUST 通过独立授权规则明确

#### Scenario: 双方完成后启动评价审批
- **WHEN** `1.2` 商务部分和技术部分均已完成
- **THEN** 系统 MUST 启动现有商务评价、技术评价、总经理审批节点
- **AND** 系统 MUST NOT 在双方完成前将 `1.2` 纳入评价审批待办

#### Scenario: 已提交部分锁定
- **WHEN** `1.2` 商务部分或技术部分已经提交
- **THEN** 系统 MUST 阻止对应部分继续保存或提交
- **AND** 系统 MUST NOT 允许通过再次保存清空或篡改已提交字段
- **AND** 只有评价退回并重置协同状态后，相关部分才可重新填写

#### Scenario: 1.2 不使用普通资料责任人
- **WHEN** 用户尝试为 `1.2 项目立项审批表` 手工分配资料责任人
- **THEN** 系统 MUST 拒绝该操作
- **AND** 我的资料待办 MUST NOT 因 `1.2` 历史 `responsible_user_id` 生成普通资料责任人待办

#### Scenario: 评价审批路径不退化
- **WHEN** `1.2` 进入评价审批
- **THEN** 系统 MUST 继续复用现有商务评价、技术评价、总经理审批节点
- **AND** 商务或技术评价拒绝 MUST 仍退回项目市场调研
- **AND** 总经理拒绝 MUST 仍支持退回项目市场调研或项目结束

### Requirement: 立项在线表单后续模板文件产出

项目核心能力 MUST 将 `1.1 / 1.2 / 1.3` 在线表单提交后生成模板文件作为后续能力方向；本 change MUST NOT 生成 Excel、Word、PDF 或文件平台文件。

#### Scenario: 在线表单仍是填写入口
- **WHEN** 用户填写 `1.1`、`1.2` 或 `1.3`
- **THEN** 系统 MUST 继续将在线表单作为填写入口
- **AND** 系统 MUST NOT 将三项资料退回普通附件上传流程

#### Scenario: 后续生成模板文件
- **WHEN** 后续实现文件产出能力
- **THEN** `1.1` 提交后 SHOULD 生成 `项目需求表-模板.xlsx` 对应文件
- **AND** `1.2` 提交或审批完成后 SHOULD 生成 `项目立项审批表-模板.xlsx` 对应文件
- **AND** `1.3` 提交后 SHOULD 生成 `关于确定项目名称及编号的通知-模板.docx` 对应文件

#### Scenario: 本 change 不生成文件
- **WHEN** 本 change 完成实现
- **THEN** 系统 MUST NOT 生成 Excel、Word、PDF、普通附件或文件平台文件
- **AND** 系统 MUST NOT 将文件生成写成已完成能力
