## 与 20260624 版的主要差异

1. 普通红色产出文件仍按 64 项核对，阶段和资料编号沿用 20260624 规划表。
2. `7.P1 随机资料移交` 和 `8.P1 资料服务器核查` 仍暂不计入普通 64 项产出文件，除非后续确认它们形成独立文件。
3. 项目最初输入和创建时没有正式项目编号。项目立项审批通过并发布 `项目立项通知` 后，才生成或填写项目编号。
4. 资料完成规则不再等同于“所有资料都要审核通过”。只有流程图中明确 YES/NO 或 YES-only 的资料/节点才需要确认或审批；没有确认/审批节点的产出资料提交或上传后即可放行。
5. 流程图中的 NO 回退不等于资料变成条件性资料。主线必产资料被退回修改，仍是主线必产，只是需要重新提交或重新确认。
6. 条件性资料只指特定分支触发才需要的资料，例如客户要求资料、厂内安装调试不通过后触发的设计变更 4 项资料。
7. `发票（预付款）`、`发票（发货款）`、`发票（尾款）` 按普通阶段产出文件处理；流程图未明确发票审批/确认节点，因此提交或上传后完成，不规划付款流或发票审批流。

## 程序完成规则枚举

| completionMode | 含义 |
| --- | --- |
| submit_only | 提交或上传后即视为完成 |
| approval_required | 必须确认或审批通过后才视为完成 |
| conditional_submit | 条件触发前不计缺失；触发后提交或上传即完成 |
| conditional_approval | 条件触发前不计缺失；触发后必须确认或审批通过；当前 64 项没有资料使用该模式 |

## 当前确认结论

当前无业务确认保留项。后续如领导明确某资料需要新增审批或确认节点，应通过独立变更调整该资料的 `completionMode`，不得在本规划中额外推导付款流、发票流、合同流、采购流或设计变更流程引擎。

## 规划结论

1. 后续不应继续把所有资料完成规则写死为 `status = confirmed`。
2. 阶段推进应改为只按当前阶段适用资料的 `completionMode` 判断完成，不再额外叠加一个泛化的阶段关口审批。
3. 项目编号应从创建必填改为立项后置生成或填写。
