# 9.13 审批不通过精准返工规划 20260626

## 当前基线

本规划基于已归档的 `online-platform-internal-document-flow-v1`：在线平台已经以 20260625 资料口径作为当前闭环基线，项目资料为 8 阶段、64 项，`completionMode` 统计保持 `submit_only 33`、`approval_required 24`、`conditional_submit 7`、`conditional_approval 0`。当前阶段附件仍保存在在线平台，不联动文件管理平台。

## 目标

审批资料 NO 时，审批人必须能精准指定需要返工的上游资料。系统不整阶段退回，不自动退回全部前置资料，不推送通知，不做通用审批流引擎，不改变现有 `completionMode` 数量口径。

本规划只解决“审批不通过后的精准返工”。不解决 `1.2 项目立项审批表` 内部商务评价、技术评价、总经理多节点在线审批。

## A 类：固定上游候选返工

A 类审批资料退回时必须从固定候选中至少选择 1 项返工目标。候选只包含当前项目中 `isApplicable !== false` 的资料；不得自由勾选本阶段全部资料。

| 审批资料 | 固定返工候选 |
| --- | --- |
| 1.2 | 1.1 |
| 2.12 | 2.4-2.11 |
| 2.13 | 2.4-2.11 |
| 3.3 | 3.2 |
| 4.12 | 4.3-4.11 |
| 4.13 | 4.3-4.11 |
| 4.16 | 4.14, 4.15 |
| 4.17 | 4.14, 4.15 |
| 5.4 | 5.3 |

特别约束：

- `2.12` / `2.13` 的候选范围不包含 `2.2` / `2.3`。
- A 类退回请求必须携带 `returnReason` 和 `revisionTargetDocumentIds`。
- `revisionTargetDocumentIds` 只用于 A 类固定上游候选返工。
- A 类未选择候选、选择非候选资料、选择条件未触发资料时，后端必须拒绝。

## B 类：单份退回

以下资料审批不通过时只退回自身，不展示上游候选：

- `2.2`
- `2.3`
- `2.14`
- `2.15`
- `3.1`
- `3.2`
- `5.1`
- `5.3`
- `7.4`
- 以及没有明确上游返工候选的普通 `approval_required` 资料

`3.2` 和 `5.3` 既是 B 类单份审批资料，又分别是 `3.3` / `5.4` 的 A 类返工候选，不矛盾。当它们自己审批不通过时，按 B 类退回自己；当它们被 `3.3` / `5.4` 指定返工时，按 A 类上游返工处理。

## C 类：厂内安装调试特殊处理

`5.12 安装调试记录（厂内）` NO 时触发设计变更分支。退回 `5.12` 时，审批人必须通过独立字段 `designChangeTargetDocumentIds` 勾选 `5.13`、`5.14`、`5.15`、`5.16` 中至少 1 项。

被勾选的 `5.13-5.16` 必须同时设置：

- `isApplicable = true`
- `revision_required = true`

未勾选的不触发。`5.9`、`5.17`、`6.1`、`7.2`、`7.3` 属于整改或外部处理，第一版仍按当前审批资料本身退回，不做上游候选选择。

C 类字段与 A 类字段分离：

- `5.12` 不使用 `revisionTargetDocumentIds`。
- `5.12` 可携带 `designChangeTargetDocumentIds`，这不算违反“非 A 类不得携带上游返工目标”。
- `designChangeTargetDocumentIds` 只能选择 `5.13`、`5.14`、`5.15`、`5.16`。
- 选择其他资料必须拒绝。
- A 类或 C 类每次设置 `revision_required = true` 时，必须清空 `revision_resubmitted_by_user_id` 和 `revision_resubmitted_at`。

## revision_required 机制

不复用 `returned` 标记上游资料。新增独立返工标记：

- `revision_required`
- `revision_reason`
- `revision_source_document_id`
- `revision_requested_by_user_id`
- `revision_requested_at`
- `revision_resubmitted_by_user_id`
- `revision_resubmitted_at`
- `revision_completed_by_user_id`
- `revision_completed_at`

`revision_required = true` 必须阻塞阶段推进，并进入责任人工作台。无责任人的返工资料不得丢失，项目详情必须显示“需返工但未分配责任人”，允许项目经理或有权限负责人先分配责任人。

## 清除返工规则

返工清除必须按上游资料的 `completionMode` 分流：

- 上游是 `submit_only` / `conditional_submit`：责任人重新上传或修改后，必须执行明确动作清除 `revision_required`；不得绕过资料权限。
- 上游是 `approval_required`：必须执行“返工重提”，并经审核确认后才能清除 `revision_required`，尤其覆盖 `3.2 销售合同`、`5.3 采购合同`。即使基础状态已经是 `confirmed`，只要 `revision_required = true`，责任人也必须允许重提；重提后基础状态进入 `submitted`，设置 `revision_resubmitted_by_user_id` 和 `revision_resubmitted_at`，进入待审核，但重提动作本身不得清除 `revision_required`。审核确认通过后才清除 `revision_required`；如重提后再次被退回，`revision_required` 保持，并清空或失效本轮 `revision_resubmitted_at`，直到后续重提并确认通过。

清除返工必须写业务日志。

待办规则：

- `approval_required + revision_required` 未重提前，只进入责任人待办。
- 未重提前不得进入审核待办。
- 重提后才进入审核待办，必须用 `revision_resubmitted_at` 或等价显式重提标记判断“已返工重提”，不得用 `submitted_at` 与 `revision_requested_at` 的时间比较替代。

## 前后端规划

后端：

- 查询审批资料时返回可选返工候选列表。
- A 类退回校验 `revisionTargetDocumentIds` 必须属于固定候选范围。
- 非 A 类退回不得携带 `revisionTargetDocumentIds`。
- C 类 `5.12` 校验 `designChangeTargetDocumentIds` 必须至少选择 1 项，且只能选择 `5.13-5.16`。
- 条件未触发资料不得进入候选。
- `revision_required` 参与齐套、推进、工作台和总览派生完成状态计算。
- A 类标记返工、C 类触发设计变更资料均写 `document.revision_requested`；`submit_only` / `conditional_submit` 明确完成返工后写 `document.revision_completed`；`approval_required` 审核确认并清除返工后写 `document.revision_completed`。

前端：

- A 类审批退回时展示固定候选多选，候选显示资料编号、资料名称、责任人、当前状态、完成规则、是否适用。
- A 类未选择候选时不能提交退回。
- B 类仍只展示退回原因；C 类 `5.12` 退回弹窗额外展示 `5.13-5.16` 设计变更触发项，至少选择 1 项后才能提交。
- 被打 `revision_required` 的资料卡片显示“需返工”。
- `approval_required + revision_required` 资料显示“返工重提”入口，不提供直接清除返工入口；重提后展示为待审核，审核确认后恢复完成。
- 工作台展示“需返工资料”待办。
- 未分配责任人的返工资料在项目详情可见，并提示先分配责任人。

## 非目标

- 不整阶段退回。
- 不自动退回全部前置资料。
- 不做推送通知。
- 不动文件平台。
- 不改 `completionMode` 数量口径。
- 不做通用审批流引擎。
- 不解决 `1.2 项目立项审批表` 内部多节点在线审批。
