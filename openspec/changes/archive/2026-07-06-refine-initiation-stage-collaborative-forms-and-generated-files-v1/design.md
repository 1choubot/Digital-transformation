## Context

当前系统已完成 v20260629 / 71 项新项目模板、立项阶段商务/技术负责人、`1.2` 专用评价审批、评价拒绝退回市场调研、总经理拒绝项目结束、`1.1 / 1.2 / 1.3` 在线表单真实模板字段第一轮对齐。领导进一步确认后，需要把客户联系人、`1.2` 协同填写、项目编号前置和模板文件产出口径纳入后续实现边界。

本 change 已进入实现阶段，允许修改 API/Web 业务代码和必要 schema 初始化逻辑；本轮不生成文件、不接文件平台、不新增独立 migration 文件、不迁移旧项目。

## Goals / Non-Goals

**Goals:**

- 新建项目支持客户联系人。
- `1.2` 基础模块补齐模板表头区域和对应已有字段。
- 项目编号前置到 `1.2` 表单填写阶段。
- `1.2` 商务负责人和技术负责人协同填写。
- 双方完成后才启动现有评价审批节点。
- 记录在线表单提交后生成模板文件的后续能力边界。

**Non-Goals:**

- 不新增资料项，不改变 v20260629 / 71 项数量。
- 不迁移旧项目，不补初始化旧项目。
- 不删除兼容资料区。
- 不新增独立数据库 migration 文件；客户联系人使用项目主数据可空字段，`1.2` 协同状态复用在线表单数据存储。
- 不生成 Excel / Word / PDF。
- 不接文件平台。
- 不新增第二套审批流、第二套资料状态机、BPM 或通用流程引擎。

## Decisions

### Decision 1: 客户联系人进入项目主数据

项目创建表单新增客户联系人字段，字段位置在客户联系方式前，即 `客户名称 -> 客户联系人 -> 客户联系方式`。新建项目客户联系人必须填写；客户联系人为空时不得创建新项目。旧项目或历史数据缺失客户联系人时允许为空展示，不迁移、不补数据。

后端项目模型、创建接口、项目详情、项目列表、项目总览、项目输入节点和在线表单自动带出支持客户联系人。实现使用 `projects.customer_contact_person` 可空字段，通过 schema 初始化逻辑补齐；旧项目允许为空，不批量补数据。客户联系人不作为新资料项，也不改变阶段资料模板数量。

### Decision 2: 1.2 基础模块补齐模板表头区域

`1.2 项目立项审批表` 的 `客户基本信息` 和 `项目基本信息` 都属于基础模块，不是独立模块、独立资料项或独立审批节点。实现按真实模板表头区域和对应已有字段补齐，不用一个笼统 textarea 表达，也不扩展模板外客户调查或项目调查字段。

基础模块由商务负责人填写，并按模板表头层级展示：顶层 `表头信息` 只包含项目名称、项目号/项目编号；`客户基本信息` 只包含客户名称、项目联系人、联系方式；`项目基本信息` 只包含项目负责人、项目负责人联系方式。项目联系人映射项目主数据 `customer_contact_person`，项目负责人优先沿用商务负责人。技术负责人是 `1.2` 技术模块协同填写角色，不作为基础模块表头字段自动加入。

### Decision 3: 项目编号前置到 1.2 填写阶段

项目编号在 `1.2` 表单填写阶段填写并保存到 `projects.project_code` 或等价稳定项目编号字段。`1.2` 提交前必须已有项目编号，项目编号唯一性继续生效。

`1.2` 退回后项目编号保留，不因返工自动清空。总经理选择项目结束后项目编号仍保留用于审计。`1.3 项目立项通知` 自动带出项目编号，不再要求在 `1.2` 总经理最终通过后另行补编号。

实现需要防止通过项目编号绕过 `1.2` 协同填写和评价审批门禁：编号只是 `1.2` 表单内容和项目主数据，不代表评价审批完成。

### Decision 4: 1.2 由商务和技术协同填写

商务负责人填写基础模块和商务模块。技术负责人填写技术模块。商务负责人不能代替技术负责人提交技术模块，除非后续独立授权规则明确允许；技术负责人不能修改基础模块和商务模块。

工作台需要分别生成商务填写待办和技术填写待办，并能展示双方完成状态。只有商务部分和技术部分都完成后，系统才允许启动现有商务评价、技术评价、总经理审批节点。

实现复用现有 `project_stage_document_forms.form_data_json`，通过 `_collaboration` 元数据保存商务部分和技术部分提交状态，不新增第二套审批流或资料状态机。商务部分提交后，基础模块和商务模块锁定；技术部分提交后，技术模块锁定。`1.2` 退回项目市场调研后，系统将协同提交状态重置为未完成，要求 1.1 返工清除后再由商务/技术重新提交对应部分。

### Decision 5: 评价审批继续复用现有节点

商务评价、技术评价、总经理审批继续复用现有 `1.2` 专用评价审批节点，不新增第二套审批流。商务/技术任一退回仍退回 `1.1 项目市场调研`；总经理拒绝仍支持退回项目市场调研或项目结束。

协同填写完成只是启动评价审批的前置条件，不改变评价审批三节点本身的授权、日志、返工和结束项目口径。

### Decision 6: 工作台合成 1.2 协同填写待办

工作台不新增资料项，也不强行把 `1.2` 拆成两个资料。系统按同一 `1.2 项目立项审批表` 的内部协同状态合成两类待办：

- 商务负责人：填写/提交 `1.2` 基础模块和商务模块。
- 技术负责人：填写/提交 `1.2` 技术模块。

商务部分提交后商务待办消失，技术部分提交前技术待办保留；双方完成后协同填写待办消失，并启动现有商务评价、技术评价和总经理审批待办。项目结束或 `1.1` 返工未清时不展示可处理的 `1.2` 协同填写待办。

`1.2` 不支持普通资料责任人手工分配。项目详情和工作区不展示 `1.2` 责任人下拉，`/api/me/stage-document-tasks` 和我的工作台不得因历史 `responsible_user_id` 将 `1.2` 当作普通资料责任人待办。

### Decision 7: 在线表单是填写入口，模板文件是后续产出形态

`1.1 / 1.2 / 1.3` 第一阶段仍使用在线表单作为填写入口。后续提交后应生成对应模板文件：

- `项目需求表-模板.xlsx`
- `项目立项审批表-模板.xlsx`
- `关于确定项目名称及编号的通知-模板.docx`

有权限查看者后续默认查看生成文件。文件生成失败时如何阻塞或重试、覆盖还是版本化、文件保存位置、下载/预览、模板填充库、文件平台是否接入，均标记为后续独立 change 的实现设计点。本 change 不生成 Excel / Word / PDF，不直接承诺文件平台归档。

## Risks / Trade-offs

- `1.2` 从单人填写改为协同填写，会影响在线表单权限、工作台待办、保存/提交门禁和 smoke。缓解：先在 specs 中明确模块边界、双方提交状态和评价审批启动条件。
- 项目编号从后置改为前置，会影响现有项目编号门禁和 `1.3` 待办文案。缓解：明确编号保存到项目主数据，但不代表审批完成；`1.3` 只自动带出。
- 模板文件生成涉及 Excel/Word 模板填充、存储、版本和下载预览，复杂度较高。缓解：本 change 只规划产出方向，文件生成后续独立 change 再做详细设计。
- 旧项目可能缺少客户联系人或已有旧表单数据。缓解：不迁移旧项目，旧数据兼容展示和迁移由后续独立 change 决定。
