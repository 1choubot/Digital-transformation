# 9.22 20260629 目标模板与项目工作区大框架 shell 第一版规划

## 1. 背景与目标

本规划承接 `docs/9.21_20260629资料模板71项确认规划_20260702.md` 和已归档的 `plan-stage-document-template-v20260629-71-v1`。

第一版目标不是完整业务流，而是“大框架 shell 落地”：

- 后续以 20260629 PDF 图面产出 + 4 个流程图漏画草稿修正 = 71 项作为目标资料模板。
- 项目工作区上方成为单项目主操作区。
- 左侧继续显示 8 阶段。
- 阶段内显示 20260629 对应蓝色模块。
- 蓝色模块下显示产出卡片。
- 产出卡片承载责任人、状态、阻塞原因和处理入口。
- 非立项阶段第一版默认文件上传或附件上传。
- 立项阶段 `1.1 / 1.2 / 1.3` 保持现有在线表单、责任人、评价审批和返工规则。
- 下方旧资料清单降级为兼容区，不物理删除。

本 change 实现项目工作区 shell 第一版，允许修改前后端 shell 相关代码；不改数据库、不写 migration。本 change 不默认把新项目切到 `v20260629`，只允许定义配置或受控开关设计；真正切换新项目模板必须另开 change。

## 2. 明确不做的事

- 不在本 change 实现复杂在线表单。
- 不新增通用流程引擎。
- 不做文件平台联动。
- 不做付款流、发票流或项目模式分支。
- 不一次性精细实现所有阶段审批/返工。
- 不删除旧资料清单。
- 不处理 `file-platform-integration-v1` 或 `define-digital-platform-v1`。
- 不切换新项目默认模板。
- 不迁移旧项目。
- 不迁移通用上传、提交、审核或退回执行能力。
- 不改数据库或 migration。

## 3. 后端大框架规划

### 3.1 目标模板版本

本 shell change 定义新的非运行默认目标模板版本，建议命名为 `v20260629`。

`v20260629` 第一版应表达：

- 8 阶段结构。
- 20260629 图面产出 + 4 个草稿修正形成的 71 项目标资料。
- 阶段、蓝色模块、产出卡片之间的映射。
- 每个产出的资料名称、阶段、来源蓝色模块、是否草稿/成品、责任中心、建议 completionMode、是否必填/条件触发。
- 非立项阶段默认文件上传/附件上传。
- 立项 `1.1 / 1.2 / 1.3` 继续绑定现有在线表单能力。

### 3.2 新项目模板切换边界

本 change 不默认把新项目切到 `v20260629`。后续如果要让新项目使用目标模板，必须另开 implementation change 明确：

- 模板版本常量、项目创建默认模板版本或受控开关如何启用。
- 项目创建时如何记录模板版本。
- 切换前，当前项目创建仍使用 20260625 64 项运行模板。
- 切换后，新项目如何按 `v20260629` 初始化项目级阶段资料和工作区产出映射。
- 切换行为如何验证和回滚，不能隐式混用 64 项和 71 项。

### 3.3 旧项目兼容与补初始化

旧项目第一版不自动补初始化、不迁移、不改 64 项资料状态。后续如需要支持旧项目迁移或补初始化，必须另开 change 单独规划：

- 仅对缺失的目标资料做补初始化，或保持旧项目使用旧模板。
- 明确 64 项到 71 项的映射、保留项、废弃项和草稿/成品拆分关系。
- 明确 `合同审核记录表（销售合同）`、`采购合同审核记录表` 等迁移对照项的处理。
- 补初始化必须可追溯，避免破坏旧项目当前资料状态、附件、责任人和阶段推进结果。

### 3.4 运行安全边界

后续实现必须保留当前 64 项运行安全边界：

- 未切换模板版本的项目继续按当前资料模板运行。
- 阶段推进、工作台、责任人、附件、提交、审核、返工仍由后端权限和状态驱动。
- 前端不得自行推断或硬编码权限。

## 4. 前端大框架规划

### 4.1 工作区结构

目标结构：

```text
项目总览
└─ 项目工作区
   ├─ 左侧 8 阶段
   └─ 右侧阶段工作区
      ├─ 阶段说明
      ├─ 蓝色模块列表
      ├─ 产出卡片列表
      └─ 产出操作区：责任人 / 状态 / 阻塞原因 / 处理入口
```

### 4.2 蓝色模块和产出卡片

项目工作区应从后端读取阶段、蓝色模块和产出卡片数据。

产出卡片第一版承载：

- 资料名称。
- 所属阶段和蓝色模块。
- 责任人或默认责任角色。
- 资料状态和完成状态。
- 阻塞原因或不可操作原因。
- 附件/提交/审核处理入口。
- 非立项阶段点击处理入口后定位到旧资料清单对应资料。
- 处理入口不得直接执行上传、提交、审核或退回。

产出卡片 shell 第一版只承载入口和状态展示，不迁移非立项阶段通用文件上传、提交、审核/退回执行能力。通用执行能力从旧资料清单迁移到产出卡片，必须后续按阶段独立实现。

### 4.3 旧资料清单兼容区

下方旧资料清单降级为辅助兼容入口：

- 对尚未迁移稳定的阶段继续可访问。
- 不再作为目标工作区长期主操作入口。
- 不在第一版物理删除。
- 旧资料清单和上方产出卡片必须共享后端资料状态，避免上下状态不一致。

## 5. 权限与状态原则

- 责任人分配、附件上传、提交、审核、退回必须以后端权限为准。
- 前端只展示后端返回的 `permissions`、`status`、`completionMode`、`blockingReasons` 和责任人字段。
- 非立项阶段第一版默认文件上传，不新增在线表单状态机。
- 立项 `1.1 / 1.2 / 1.3` 保持现有在线表单和专用审批规则，不回退到普通文件上传主入口。

## 6. 验收范围建议

本 shell change 验收至少覆盖：

- 项目工作区显示 8 阶段。
- 每个阶段可显示蓝色模块或占位模块。
- 蓝色模块下显示产出卡片。
- 产出卡片显示责任人、状态、阻塞原因和处理入口。
- 点击非立项产出卡片处理入口后定位到旧资料清单对应资料，不改变 route 到错误页面。
- 非立项阶段默认走文件上传/附件上传。
- 立项 `1.1 / 1.2 / 1.3` 仍走现有在线表单和专用审批。
- 下方旧资料清单仍可作为兼容区访问。
- 前端权限展示和操作入口由后端返回驱动。
- 桌面和移动 viewport 下 8 阶段、蓝色模块、产出卡片和旧清单不重叠、不溢出。

## 7. 本 change 实施拆分与后续独立 change

1. A 层：后端定义非运行默认的 `v20260629` 目标模板配置和 71 项产出映射。
2. A 层：设计受控开关或模板版本元数据，但不在本 change 默认切换新项目模板。
3. A 层：明确旧项目第一版不自动补初始化、不迁移、不改 64 项资料状态。
4. B 层：后端返回项目工作区阶段、蓝色模块和产出卡片 shell 数据。
5. B 层：前端项目工作区渲染蓝色模块和产出卡片 shell。
6. C 层：产出卡片展示责任人、状态、阻塞原因和附件/提交/审核入口。
7. 后续独立 change：切换新项目默认模板。
8. 后续独立 change：旧项目迁移或补初始化。
9. 后续逐阶段 change：将旧资料清单通用操作逐步迁移到产出卡片。
10. 做 API check、Web build、OpenSpec validate 和浏览器/人工验收。
