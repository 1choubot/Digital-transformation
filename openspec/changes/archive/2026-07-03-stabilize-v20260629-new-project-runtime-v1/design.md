## Context

`enable-v20260629-template-for-new-projects-v1` 已归档并提交，新建项目默认使用 v20260629 71 项资料模板。旧项目仍保持 20260625 64 项资料，`LC33 / LC54` 只作为旧项目 workspace 兼容输出。旧资料清单已经默认折叠为兼容资料区，但仍保留可展开查看。

本 change 是运行基线稳定验证 change。它不是新业务规则 change，也不是旧项目迁移 change。

## Goals / Non-Goals

**Goals:**

- 验证新项目默认生成 71 项资料且 `template_version = v20260629`。
- 验证新项目不包含 `3.3 / 5.4 / LC33 / LC54`。
- 验证旧项目保持 64 项，不迁移、不补初始化。
- 验证新项目项目工作区展示 8 阶段、蓝色模块和 71 项产出卡片。
- 验证旧项目仍能看到 `LC33 / LC54` 兼容卡片，新项目不显示 `LC33 / LC54`。
- 验证 `1.1 / 1.2 / 1.3` 在线表单和 `1.2` 专用评价审批不回退。
- 验证非立项资料在上方产出卡片可复用现有通用资料操作。
- 验证我的工作台能展示 v20260629 新项目待办。
- 验证阶段推进按项目实际资料集合判断，不混用 64/71。
- 验证兼容资料区在新项目中文案不误导，在旧项目中仍可展开核对。
- 允许修复验证发现的 v20260629 新项目运行阻塞 bug。

**Non-Goals:**

- 不新增业务规则。
- 不新增复杂在线表单、流程引擎、付款流、发票流或项目模式分支。
- 不处理文件平台联动。
- 不迁移旧项目，不补初始化旧项目。
- 不删除、不隐藏、不物理移除兼容资料区。
- 不改数据库 schema，不写 migration。
- 不处理 `file-platform-integration-v1` 或 `define-digital-platform-v1`。

## Decisions

### Decision 1: 先 API smoke，后浏览器验收

本 change 的 verification MUST 先用 API smoke 验证数据层和权限层，再用人工浏览器验收验证页面与交互。API smoke 应覆盖模板版本、资料数量、新旧项目并存、workspace 数据、workbench 待办和阶段推进门禁。

理由：v20260629 已成为新项目默认模板，数据层和权限层是项目工作区、工作台和阶段推进的共同基础。

### Decision 2: 所有验证使用新建测试项目

验证 v20260629 新项目运行基线时 MUST 使用新建测试项目或专门测试数据，不得通过迁移旧项目、补初始化旧项目或改写旧项目资料记录来制造测试条件。

### Decision 3: 修 bug 只能修现有入口

发现阻塞 bug 时，implementation MUST 优先修复现有项目创建、阶段资料、workspace 聚合、产出卡片、工作台、阶段推进、在线表单或兼容资料区入口。不得创建第二套资料状态机、第二套上传/提交/审核/退回/返工/不适用规则或临时业务分支。

### Decision 4: 浏览器自动化不是强制项

人工浏览器验收是本 change 的必要验收输出；浏览器自动化 MAY 使用，但不是强制项。如果 agent 自动化卡住，可以由人工测试会话执行并记录结果。

### Decision 5: 兼容资料区继续保留

本 change MAY 验证兼容资料区文案和展开能力，但 MUST NOT 删除、隐藏或物理移除兼容资料区。兼容资料区后续清理仍需独立 change。

### Decision 6: 文件平台和复杂流程保持后置

即使 v20260629 新模板包含草稿合同、供应商评价、生产记录或资料移交等资料，本 change 也只验证其第一版文件上传/附件上传运行能力。文件平台联动、在线表单化、专用审批和复杂流程必须后置。

## Runtime Audit Dimensions

- 新项目创建事务是否初始化 8 阶段和 71 项资料。
- `project_stage_documents.template_version` 或等价版本字段是否为 `v20260629`。
- 新项目是否排除 `3.3`、`5.4`、`LC33`、`LC54`。
- 旧项目是否保持 64 项资料且未补初始化。
- workspace 是否返回 8 阶段、蓝色模块和 71 项产出卡片。
- workspace 是否只在旧项目返回 `LC33 / LC54` 兼容卡片。
- 产出卡片是否绑定 documentId/documentCode、权限、责任人、附件和状态。
- stage-document-checklist 是否和 workspace 读取同一套后端状态。
- workbench 是否能统计 v20260629 新项目待办。
- 阶段推进是否按项目实际资料集合、completionMode、适用性和返工字段判断。

- 项目详情页能展示项目基础信息、8 阶段和项目工作区。
- 上方产出卡片可处理非立项资料责任人、附件、提交、审核、退回、返工和适用性操作。

- 兼容资料区默认折叠、可展开，且新项目文案不误导。

## Risks / Trade-offs

- [Risk] 新旧模板并存导致工作台或阶段推进混用 64/71。Mitigation：API smoke 必须覆盖项目实际资料集合判断。
- [Risk] 新项目误显示 `LC33 / LC54` 兼容输出。Mitigation：workspace 和前端验收必须分别覆盖新项目排除、旧项目保留。
- [Risk] 立项在线表单入口被普通资料操作覆盖。Mitigation：浏览器验收必须覆盖 `1.1 / 1.2 / 1.3` 和 `1.2` 专用评价审批。
- [Risk] 验证 change 被扩大为新业务规则实现。Mitigation：spec 和 tasks 明确禁止新增业务规则、流程引擎、文件平台联动和旧项目迁移。

## Migration Plan

本 change 不执行数据迁移。

如验证发现旧项目或新项目数据异常，必须先定位为现有逻辑 bug 或测试数据问题；不得通过迁移旧项目、补初始化旧项目或改写历史项目资料作为本 change 的修复方式。
