## Context

数字化管理平台第一版已经确定业务定位：数字化平台作为项目全过程业务主入口，文件管理平台作为文件存储、下载权限和文件日志底座。现有 `digital-demo` 只能作为前端页面形态参考，不能继续作为正式系统的数据架构或接口架构来源。

第一版需要在公司内网环境运行，并与现有文件管理平台部署在同一服务器。两套平台需要弱绑定联动，但必须保持服务端和数据库边界清晰，避免数字化平台直接读写文件管理平台数据库造成耦合和权限风险。

## Goals / Non-Goals

**Goals:**

- 确定正式数字化管理平台第一版技术栈：Vue + Vite、Node.js + Express、MySQL。
- 确定数字化平台是独立 Web 服务和独立服务端。
- 确定数字化平台独立维护业务数据库，不直接共用文件管理平台数据库。
- 确定文件仍由文件管理平台存储，数字化平台只保存业务数据、状态和必要映射。
- 确定两平台通过 HTTP API 联动。
- 确定部署形态为内网 Web，并与文件管理平台同服务器部署。
- 确定前端 Demo 只作为页面参考，不作为正式数据架构。

**Non-Goals:**

- 不实现业务功能、页面、接口或数据库迁移。
- 不定义在线表单字段、文件归档接口细节、权限矩阵或日志字段。
- 不替换文件管理平台的存储、下载、权限和日志能力。
- 不定义公网部署、移动端部署、钉钉/金蝶集成或微服务拆分方案。
- 不要求数字化平台和文件管理平台共用运行进程、数据库连接或数据表。

## Decisions

### 决策一：前端采用 Vue + Vite

正式系统前端采用 Vue + Vite，延续当前 Demo 的技术方向和页面参考价值，但正式前端必须改接后端 API 和 MySQL 持久化数据。

不选择继续使用 Demo 内置静态数据作为正式数据来源，因为静态数据无法支撑项目创建、阶段状态、资料状态和跨用户协作。

### 决策二：后端采用 Node.js + Express

数字化平台后端采用 Node.js + Express，提供项目、阶段、资料、权限和文件联动等业务 API 的服务端入口。第一版不把业务逻辑写在前端，也不直接调用数据库绕过后端。

不选择只做纯前端应用，因为正式系统需要持久化、接口校验、跨平台联动和后续权限控制。

### 决策三：数据库采用独立 MySQL

数字化平台使用 MySQL 保存自己的业务数据，包括项目、阶段、资料项、表单数据、业务状态、用户映射和必要的文件平台映射。数字化平台不得直接共用或直接读写文件管理平台数据库。

不选择共用文件管理平台数据库，因为两套系统的数据模型、权限边界和演进节奏不同，直接共库会增加耦合和误写风险。

### 决策四：文件仍由文件管理平台存储

归档文件、上传文件、下载权限和文件日志继续由文件管理平台负责。数字化平台只保存业务侧记录和必要的文件标识、目录映射、归档状态等关联信息。

不选择把文件二进制存入数字化平台数据库或本地目录，因为这会重复建设文件平台能力，并削弱既有文件权限和日志能力。

### 决策五：两平台通过 HTTP API 联动

数字化平台通过 HTTP API 调用文件管理平台能力，例如目录创建或绑定、文件归档、文件列表、下载链接和文件日志元数据传递。接口失败、超时和权限失败应在后续具体能力中定义处理规则。

不选择直接访问文件管理平台数据库或共享内部模块，因为 HTTP API 更适合作为两套独立系统之间的稳定边界。

### 决策六：同服务器内网部署但服务边界独立

第一版按内网 Web 部署，数字化平台与文件管理平台部署在同一服务器上，但作为独立 Web 服务运行。两套服务可通过本机或内网地址通信，端口、进程、配置和日志应保持独立。

不选择一开始拆成多服务器或复杂容器集群，因为第一版重点是业务闭环验证；同服务器部署能降低运维复杂度。

## Risks / Trade-offs

- [同服务器部署资源竞争] -> 后续实施时分别配置端口、进程管理、日志目录和基础资源监控。
- [HTTP API 联动失败影响业务闭环] -> 文件联动能力中再定义超时、重试、失败状态和人工处理入口。
- [独立数据库带来跨系统一致性问题] -> 通过文件平台返回 ID、状态和日志元数据建立弱绑定映射，不直接跨库事务。
- [Demo 参考被误解为正式架构] -> 在规格中明确 Demo 只用于页面参考，正式数据必须来自后端 API 和 MySQL。
- [Node.js + Express 需要规范化工程结构] -> 后续实施任务中补充目录结构、配置管理、错误处理和数据库访问约定。

## Migration Plan

- 新建数字化平台正式前端项目或整理现有前端，使其按 Vue + Vite 构建并接入后端 API。
- 新建数字化平台 Node.js + Express 服务端。
- 新建数字化平台 MySQL 数据库和独立连接配置。
- 在同一服务器上为数字化平台配置独立端口、进程管理和内网访问入口。
- 通过配置项维护文件管理平台 HTTP API 基础地址和调用凭据。

## Open Questions

- 同服务器部署时，数字化平台前端是否由 Express 静态托管，还是由独立 Web 服务器托管？
- MySQL 是使用现有 MySQL 实例新建库，还是安装独立 MySQL 实例？
- 两平台 HTTP API 调用的认证方式采用固定服务密钥、内网白名单，还是文件平台已有登录态/Token？
