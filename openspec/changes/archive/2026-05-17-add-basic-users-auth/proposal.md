## Why

第一版项目核心页面已经可以创建和查看项目，但后续资料提交、文件归档和日志记录需要有真实操作人。需要先补齐数字化管理平台自己的基础用户与登录能力，避免继续以匿名或静态用户口径推进后续业务。

本变更只建立数字化平台第一版最小用户登录能力，并严格保持与文件管理平台数据库隔离。

## What Changes

- 新增数字化平台用户主数据设计，保存账号、姓名、部门、角色、启用状态，以及可选 `filePlatformUserId` 映射字段。
- 新增基础登录能力，用数字化平台自己的用户表校验账号和密码，并返回当前登录用户信息。
- 新增 `POST /api/auth/logout` 退出登录接口，用于主动结束当前登录态。
- 新增前端登录页，调用数字化平台后端登录接口完成登录。
- 新增前端登录态保存能力，项目页面展示当前登录用户。
- 项目创建时必须根据当前登录态识别当前用户，并在 `projects` 表中记录 `createdByUserId`。
- 现有历史项目允许 `createdByUserId` 为空，避免影响已创建测试数据。
- 项目列表和项目详情必须返回创建人基础信息或创建人字段，至少能追溯是谁创建的项目。
- 前端创建项目请求必须携带登录态；未登录创建项目时后端必须拒绝，并返回需要登录的错误。
- 明确本变更只是认证和操作人记录，不实现复杂权限控制。
- 预留 `filePlatformUserId` 字段用于后续与文件管理平台用户建立映射，但本变更不实现用户同步、权限同步或下载权限判断。
- 明确本变更不实现复杂菜单权限、文件下载权限、与文件管理平台用户自动同步、部门权限继承、审批流、单点登录或细粒度按钮权限。

## Capabilities

### New Capabilities

- `basic-users-auth`: 覆盖数字化平台基础用户表、基础登录、前端登录页、前端登录态保存，以及项目页面当前登录用户展示。

### Modified Capabilities

- `project-core`: 项目创建需要登录态并记录创建人，项目列表和详情返回创建人追溯字段，同时兼容历史项目创建人为空。
- `project-core-frontend`: 新建项目请求携带登录态，项目列表和详情展示创建人追溯信息，未登录创建项目时提示需要登录。

## Impact

- 后续实现会影响 `digital-platform-api` 的 MySQL 表结构、用户认证接口、当前用户接口、退出登录接口，以及项目创建/列表/详情接口。
- 后续实现会影响 `digital-platform-web` 的登录页面、登录态存储、API client 请求处理，以及项目页面顶部当前用户展示。
- 后续实现会在项目主数据中增加可空 `createdByUserId`，并通过项目列表和详情提供创建人追溯信息。
- 数字化平台用户数据必须存储在数字化平台自己的 MySQL 数据库或 schema 中。
- 本变更不得直接读取、共用或迁移文件管理平台数据库；不得调用文件管理平台用户同步、权限同步或下载权限能力。
